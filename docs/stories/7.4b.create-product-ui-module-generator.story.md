# Story 7.4b: Create Product UI Module Generator

## Status
Approved

## Story
**As a** Developer,
**I want** the CLI to have a `scaffold ui-resource` command,
**so that** it generates React-Admin TypeScript resource files within my product's ui-module directory that integrate with the framework admin shell.

## Dependencies
**Story 7.4a** (prerequisite): Framework admin shell must be created first to provide reusable infrastructure (AdminShell component, dataProvider, authProvider, theme, shared components).

## Acceptance Criteria
1. Running `eaf scaffold ui-resource <Name> --module <productModule>` generates the necessary `.tsx` files (List, Create, Edit, Show, types.ts, EmptyState, LoadingSkeleton, index.ts, README.md - 9 files total) within `products/{productModule}/ui-module/src/resources/{resourceLowerCase}/` with HIGH priority UX enhancements (bulk actions, loading states, type-to-confirm delete, field grouping, validation messages).
2. The generated UI resource components correctly import and use the framework admin shell (from Story 7.4a), reference API endpoints and data structures from Story 7.3, include RFC 7807 error handling, and follow WCAG AA accessibility standards.
3. Generated ui-module exports a resource registration object that can be imported by apps/admin for dynamic resource registration.

### Previous Story Insights (Story 7.3)
**Key Learnings:**
- **Security-First TDD**: Implement security test (7.4-UNIT-001) FIRST before any file generation, validating 7 attack scenarios (path traversal, injection, PascalCase violations).
- **3-Layer Defense**: Input validation → path sanitization → canonical path verification for all file operations.

- **Mustache Templating**: Established templating engine with 16 templates in Story 7.3. Type-based conditionals not possible in Mustache; use list iteration for field-based generation.
- **Filename Convention**: Class name matches filename per ktlint class-filename rule.
- **Quality Gates**: Generated code must compile and pass quality gates immediately (ktlint: 0, detekt: 0).

### Kotlin DTO Parsing Strategy (Task 5.2 Guidance)

**Simple Regex Approach** (Recommended for MVP):
- Use regex pattern: `(val|var)\s+(\w+):\s*([A-Za-z0-9<>?,\s]+)` to extract field declarations
- Example parsing: `val productId: String` → name="productId", type="String"
- Handle nullable types: `String?` → type="string | null" in TypeScript
- Handle generic types: `List<String>` → type="string[]" in TypeScript
- Sufficient for flat DTOs with primitive/standard types (String, Int, BigDecimal, List<T>)
- Handles 90% of generated DTOs from Story 7.3
- Fast, no additional dependencies
- Limitation: Does not handle complex nested types or annotations

**AST Parser Approach** (Future Enhancement):
- Use kotlin-compiler-embeddable if complex types needed (nested DTOs, custom annotations)
- Add dependency: `implementation("org.jetbrains.kotlin:kotlin-compiler-embeddable:2.2.20")`
- Triggered by regex parse failure or complex type detection
- Add as enhancement in Story 7.5 if needed

**Fallback**:
- If DTO parsing fails (either approach), log clear error with fallback message:
  "Could not parse {FileName}Response.kt - use `--fields` option to manually specify types"
- Example: `eaf scaffold ra-resource Product --fields "name:string,price:number,active:boolean"`
- Ensures developer can always proceed even if automated parsing fails

[Source: Pragmatic technology selection principle, fail-fast with clear fallback, Task 5.2 implementation guidance]

### Security Patterns

**JWT Authentication (Layer 1 - Request Filter):**
```typescript
// apps/admin/src/dataProvider.ts
const authProvider = {
  login: async ({ username, password }) => {
    const response = await fetch('http://localhost:8180/realms/eaf/protocol/openid-connect/token', {
      method: 'POST',
      body: new URLSearchParams({ username, password, grant_type: 'password', client_id: 'eaf-admin' })
    });
    const { access_token, refresh_token } = await response.json();
    localStorage.setItem('token', access_token);
    localStorage.setItem('refresh_token', refresh_token);
  },
  checkAuth: async () => {
    return localStorage.getItem('token') ? Promise.resolve() : Promise.reject();
  },
  getPermissions: async () => {
    const token = localStorage.getItem('token');
    const decoded = jwtDecode(token);
    return decoded.realm_access?.roles || [];
  }
};

const dataProvider = simpleRestProvider('http://localhost:8080/api/v1', fetchJson);
const dataProviderWithAuth = (url, options) => {
  options.headers = new Headers({
    ...options.headers,
    'Authorization': `Bearer ${localStorage.getItem('token')}`,
    'X-Tenant-ID': extractTenantFromJWT(localStorage.getItem('token')) // Layer 1 tenant extraction
  });
  return dataProvider(url, options);
};
```

### API Endpoint Contract

**Assumptions (from Story 7.3 generated backend):**
- **List Endpoint**: `GET /api/v1/{resource}?page=0&size=20&sort=createdAt,desc&tenantId={currentTenant}`
- **Create Endpoint**: `POST /api/v1/{resource}` with JSON body (no tenantId - extracted from JWT)
- **Get Endpoint**: `GET /api/v1/{resource}/{id}?tenantId={currentTenant}`
- **Update Endpoint**: `PUT /api/v1/{resource}/{id}` with JSON body
- **Delete Endpoint**: `DELETE /api/v1/{resource}/{id}?tenantId={currentTenant}`

**Response Format (Success):**
```json
{
  "id": "uuid",
  "name": "string",
  "createdAt": "2025-01-15T10:30:00Z",
  ...fields
}
```

**Response Format (Error - RFC 7807):**
```json
{
  "type": "https://eaf.axians.com/errors/validation",
  "title": "Validation Failed",
  "status": 400,
  "detail": "Field 'sku' violates constraint 'pattern'",
  "instance": "/api/v1/products",
  "traceId": "uuid",
  "tenantId": "tenant-uuid"
}
```
