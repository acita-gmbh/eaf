# Story 4.5: Migrate Widget Domain Code from Framework to Product Module

## Status
Ready for Review

## Story
**As a** Framework Architect,
**I want** the Widget domain code (aggregate, handlers, tests) migrated from `framework/widget/` to `products/widget-demo/`,
**so that** framework modules contain only reusable infrastructure and can be published as libraries while domain logic lives in product modules.

## Business Context
This story addresses architectural debt discovered during Story 4.4 implementation. The Widget domain currently lives in `framework/widget/`, violating Clean Architecture principles and framework publishing requirements. This refactoring establishes the correct boundary between framework infrastructure (reusable) and product domain logic (business-specific).

## Architectural Rationale

### Why Widget Belongs in products/
1. **Framework Philosophy**: Frameworks provide INFRASTRUCTURE (like Spring Data's `Repository<T>`), not DOMAIN MODELS (like actual entities)
2. **Clean Architecture**: Domain entities (inner layer) should not live in infrastructure modules (outer layer)
3. **Maven Publication Litmus Test**: If framework is published to Maven Central, Widget aggregate should NOT ship with framework JARs
4. **Reusability Test**: licensing-server product does NOT reuse Widget - it's not generic infrastructure
5. **Spring Modulith Alignment**: "widget" is a domain bounded context, not a framework bounded context

### Historical Context
**Story 2.1** created `framework/widget/` as:
- Walking Skeleton demo (Epic 2: first end-to-end CQRS flow)
- Reference pattern for future aggregates
- Proof-of-concept for CQRS/ES integration

**Original Intent**: TEMPORARY demo/example, not permanent framework component

**Story 4.4** created `products/widget-demo/` as:
- Minimal Spring Boot reference implementation
- Framework validation and E2E testing
- Pattern: Like Spring Petclinic (spring-projects reference impl)
- Auto-configuration demo: Shows framework usage with ZERO config code

**Architectural Decision**: Framework provides auto-configured infrastructure, products provide domain models.

### Reference Patterns
- **Spring Petclinic** (spring-projects/spring-petclinic) - Reference impl separate from framework
- **Spring Boot Samples** (spring-boot/samples) - Demos not in framework modules
- **Axon Quick Start** (axoniq/axon-quick-start) - Example app separate from Axon Framework

## Dev Notes

### Critical Migration Considerations

**Package Rename Strategy**:
- From: `com.axians.eaf.framework.widget.*`
- To: `com.axians.eaf.products.widgetdemo.*`
- Shared API unchanged: `com.axians.eaf.api.widget.*` (correct location)

**Import Update Scope**:
- Widget domain files (Widget.kt, WidgetError.kt)
- Handler files (WidgetProjectionHandler.kt, WidgetQueryHandler.kt)
- All test files (unit + integration)
- Spring Modulith metadata (WidgetModule.kt)

**Shared API Boundary** (CRITICAL - DO NOT MOVE):
- `shared/shared-api/src/main/kotlin/com/axians/eaf/api/widget/` contains:
  - `CreateWidgetCommand`, `UpdateWidgetCommand` (commands)
  - `WidgetCreatedEvent`, `WidgetUpdatedEvent` (events)
  - `FindWidgetByIdQuery`, `FindAllWidgetsQuery` (queries)
- These represent the CONTRACT between command/query sides and MUST remain in shared-api
- Only the IMPLEMENTATION (aggregate, handlers) moves to products/widget-demo

**Test Application**:
- `WidgetFrameworkTestApplication.kt` → `WidgetDemoTestApplication.kt`
- Update Spring Boot test annotations if they reference old package
- Verify Testcontainers configuration (PostgreSQL, Keycloak) still works

**Spring Modulith**:
- WidgetModule.kt needs updated `allowedDependencies` to include framework modules
- Konsist rules should validate product → framework direction (never reverse)
- Run `./gradlew verifyAllModules` for architectural compliance

### Migration Risks and Mitigations

**Risk 1: Broken Imports Causing Build Failures**
- **Likelihood**: Medium (mechanical refactoring)
- **Impact**: High (blocks all builds)
- **Mitigation**: Use IDE refactoring tools, validate with `./gradlew clean build` frequently
- **Detection**: Compilation errors, ktlint violations

**Risk 2: Missing Framework Dependencies**
- **Likelihood**: Low (well-defined dependencies)
- **Impact**: Medium (runtime errors)
- **Mitigation**: Analyze framework/widget/build.gradle.kts before deletion, add needed deps to widget-demo
- **Detection**: Compilation errors about missing classes

**Risk 3: Integration Tests Failing After Migration**
- **Likelihood**: Low (tests should be portable)
- **Impact**: High (breaks quality gates)
- **Mitigation**: Run tests immediately after each migration step, compare to baseline
- **Detection**: Test failures, coverage drops

**Risk 4: Widget-Demo Application Won't Start**
- **Likelihood**: Low (application already exists and works)
- **Impact**: High (blocks validation)
- **Mitigation**: Test application startup immediately after Task 4 completion
- **Detection**: Spring Boot startup errors

### Rollback Strategy
If migration fails:
1. Restore framework/widget from git: `git checkout framework/widget/`
2. Restore settings.gradle.kts: `git checkout settings.gradle.kts`
3. Clean migrated code: Remove changes from products/widget-demo
4. Rebuild: `./gradlew clean build`
