# Story 6.6: Implement "Dockets Pattern" BPMN Template

## Status
Ready for Review

## Story
**As a** Core Developer,
**I want** a template BPMN 2.0 XML file that replicates the legacy "Dockets Hook" pattern,
**so that** we have a clear path for migrating DPCM/ZEWSSP automation.

## Acceptance Criteria
1. A BPMN 2.0 XML file is created that defines a workflow: (1) Start Event, (2) Service Task (Ansible PRESCRIPT), (3) Service Task (Dispatch Axon CORECOMMAND), (4) Receive Event Task (wait for Axon Event), (5) Service Task (Ansible POSTSCRIPT), (6) Error Boundary Event (Compensation), (7) End Event.
2. This BPMN process definition successfully deploys to the Flowable engine.

## Dev Notes

### Previous Story Insights
- Story 6.5 established the compensation pattern using `SetCommandTypeDelegate` + `DispatchAxonCommandTask` for BPMN error boundary handling with variable-driven command dispatch.[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#key-implementation-decisions]
- The `hello-world-ansible.bpmn20.xml` from Story 6.4 demonstrates basic Ansible task integration with error boundaries but lacks the full Dockets pattern (no wait-for-event or dual Ansible tasks).[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]
- BPMN element ordering is CRITICAL: `<documentation>` must appear before `<extensionElements>` per BPMN 2.0 XSD schema or Flowable deployment will fail with SAXParseException.[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#key-implementation-decisions]
- All existing BPMN processes use `flowable:delegateExpression="${beanName}"` pattern for service task configuration rather than Java class references.[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]
- Story 6.3 implemented `AxonEventSignalHandler` which performs two-step correlation (business key query → process instance ID query) to work around Flowable's business key limitation.[Source: architecture/component-specifications.md#flowable-axon-integration]

### Dockets Pattern Legacy Context
- The legacy "Dockets" system provided workflow orchestration for DPCM/ZEWSSP with a pattern of: PRESCRIPT (pre-automation) → CORECOMMAND (business logic) → POSTSCRIPT (post-automation) with error compensation.[Source: docs/prd/epic-6-core-framework-hooks-flowable-prep-revision-3.md#epic-goal]
- NFR6 (Extensibility) requires the framework to support PRESCRIPT/POSTSCRIPT "hooks" for future workflow orchestration without core rewrites, which this BPMN template provides via Flowable integration.[Source: docs/prd/requirements.md#nfr6-extensibility]
- This template serves as the canonical pattern for migrating legacy Dockets workflows to the Flowable BPMN engine, enabling DPCM product migration by Q3 2026.[Source: docs/prd/goals-and-background-context.md]

### BPMN File Structure and Standards
- **File Location**: `framework/workflow/src/axon-integration-test/resources/processes/dockets-pattern-template.bpmn20.xml` — follows convention established by Stories 6.3-6.5 for test BPMN resources.[Source: architecture/component-specifications.md#flowable-axon-integration]
- **Namespace Requirements**: Must include `xmlns="http://www.omg.org/spec/BPMN/20100524/MODEL"`, `xmlns:flowable="http://flowable.org/bpmn"`, and diagram namespaces (`bpmndi`, `omgdc`, `omgdi`) for full BPMN 2.0 compliance.[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]
- **Documentation Pattern**: Use `<documentation>` blocks within process element and individual tasks to explain purpose, required variables, and output variables.[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]

### Flowable-Axon Integration Components
- **DispatchAxonCommandTask**: Generic JavaDelegate that dispatches Axon commands using `commandClassName` (FQN) and `constructorParameters` (Map) process variables with package whitelist security (`com.axians.eaf.*`).[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#post-implementation-review]
- **AxonEventSignalHandler**: Axon event handler that signals waiting BPMN processes using two-step correlation query (business key → process instance ID) due to Flowable limitation.[Source: architecture/component-specifications.md#flowable-axon-integration]
- **RunAnsiblePlaybookTask**: Executes Ansible playbooks via SSH, requires `playbookPath`, `inventory` (optional), `extraVars` (optional) variables, sets `ansibleExitCode`, `ansibleStdout`, `ansibleStderr` output variables, throws `BpmnError("ANSIBLE_FAILED")` on failure.[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]
- **SetCommandTypeDelegate**: Updates `commandClassName` process variable during compensation flow (created in Story 6.5), enables generic DispatchAxonCommandTask to dispatch different commands in error paths.[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#key-implementation-decisions]

### Security and Multi-Tenancy Requirements
- **Tenant Context Propagation**: All service tasks must operate within tenant context established by Flowable process start; `tenantId` variable is MANDATORY for all BPMN processes.[Source: framework/workflow/src/axon-integration-test/resources/processes/hello-world-ansible.bpmn20.xml]
- **Command Dispatch Security**: DispatchAxonCommandTask enforces package whitelist (`com.axians.eaf.*`) and explicit command class whitelist (5 allowed commands) to prevent code injection via BPMN deployment.[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#security-hardening]
- **Error Message Sanitization**: BpmnError messages must not leak infrastructure details (CWE-209 protection), use generic messages like "Process step failed" rather than exposing SSH credentials or file paths.[Source: docs/stories/6.5.implement-workflow-error-handling-compensating-actions.story.md#qa-results]

### Testing Strategy and Framework Standards
- **Kotest 6.0.3 Framework**: All tests MUST use Kotest FunSpec with native runner; JUnit is explicitly forbidden.[Source: architecture/test-strategy-and-standards-revision-3.md#kotest-603-framework-mandatory---hybrid-runner-approach]
- **@SpringBootTest Pattern**: Use `@Autowired lateinit var` field injection in init block (NOT constructor injection which causes circular dependency during compilation).[Source: architecture/test-strategy-and-standards-revision-3.md#springboottest-pattern-mandatory]
- **axonIntegrationTest Source Set**: Complex Flowable + Axon workflows use dedicated source set with `@ActiveProfiles("test")` to exclude production-only configs.[Source: architecture/test-strategy-and-standards-revision-3.md#critical-plugin-configuration-story-46-lessons]
- **Test Scope**: This story requires deployment validation only (AC2), NOT full workflow execution testing; E2E execution tests can be deferred to Epic 8 when real use cases exist.[Source: docs/prd/epic-6-core-framework-hooks-flowable-prep-revision-3.md#story-66]

### Process Variable Documentation Requirements
The BPMN documentation must define these variables for migration developers:

**Input Variables (Set at process start)**:
- `tenantId` (String, MANDATORY): Tenant identifier for isolation, propagated to all service tasks
- `prescriptPlaybookPath` (String): Path to Ansible playbook for pre-command automation
- `coreCommandClassName` (String): Fully-qualified name of Axon command class (e.g., `com.axians.eaf.api.widget.commands.CreateWidgetCommand`)
- `coreCommandConstructorParameters` (Map<String, Any>): Constructor arguments for command instantiation
- `eventBusinessKey` (String): Correlation key for AxonEventSignalHandler to match event to process instance
- `postscriptPlaybookPath` (String): Path to Ansible playbook for post-command automation
- `compensationCommandClassName` (String): FQN of compensating command for error recovery

**Output Variables (Set by service tasks)**:
- `ansibleExitCode` (Int): Exit code from Ansible playbook execution (0 = success)
- `ansibleStdout` (String): Standard output from playbook (truncated to 4KB for metadata size limits)
- `ansibleStderr` (String): Standard error from playbook (truncated to 4KB)

**Internal Variables (Set during error handling)**:
- `commandClassName` (String): Current command class for DispatchAxonCommandTask (updated by SetCommandTypeDelegate during compensation)
