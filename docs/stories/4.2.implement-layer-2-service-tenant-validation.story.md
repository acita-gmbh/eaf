# Story 4.2: Implement Layer 2 (Service Layer): Tenant Boundary Validation

## Status
Approved

## Story
**As a** Core Developer,
**I want** the Axon Command Handlers (Service Layer) to validate that the active tenant (from `TenantContext`) is authorized to operate on the requested aggregate,
**so that** tenants cannot maliciously issue commands against another tenant's data (defense-in-depth).

## Acceptance Criteria
1. All new aggregates (like `CreateWidgetCommand`) must be automatically assigned the `tenant_id` from the `TenantContext` when created.
2. When loading an existing aggregate (e.g., for an Update command), the framework must verify that the aggregate's stored tenant ID matches the `TenantContext` tenant ID.
3. If the tenant IDs do not match, a Security Exception is thrown, the transaction is rolled back, and the operation fails (e.g., 403 Forbidden or 404 Not Found).
4. An integration test (using the Keycloak Testcontainer from Epic 3) verifies that User A (Tenant A) *cannot* issue a valid command against an aggregate belonging to Tenant B.

### TenantContext Integration Pattern

**Injection in Aggregates** [Source: coding-standards-revision-2.md#Multi-Tenancy Patterns]:

**Option A: Static Access** (Current TenantContext design supports this):
```kotlin
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId() // Static access, fail-closed
    require(command.tenantId == currentTenant) {
        "Tenant isolation violation: command=${command.tenantId}, context=$currentTenant"
    }
    // ...
}
```

**Option B: Injected TenantContext** (Requires aggregate to be Spring-managed):
```kotlin
@Aggregate
class Widget(@Autowired private val tenantContext: TenantContext) {
    @CommandHandler
    constructor(command: CreateWidgetCommand, tenantContext: TenantContext) {
        this.tenantContext = tenantContext
        val currentTenant = tenantContext.getCurrentTenantId()
        // ...
    }
}
```

**DECISION**: Use **Option A (Static Access)** because:
- TenantContext is Spring @Component with single instance
- Aggregates in Axon are NOT Spring beans (event sourced, reconstructed from events)
- Constructor injection not possible in aggregate constructors
- Story 4.1 TenantContext designed for static access pattern

### Error Handling Pattern

**Arrow Either for Domain Errors** [Source: coding-standards-revision-2.md#Error Handling Patterns]:
```kotlin
sealed class WidgetError {
    data class TenantIsolationViolation(
        val requestedTenant: String,
        val actualTenant: String
    ) : WidgetError() {
        override val message = "Tenant isolation violation: requested=$requestedTenant, actual=$actualTenant"
    }

    data class SecurityException(
        val reason: String
    ) : WidgetError()
}

// Usage in update command handlers:
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = tenantContext.getCurrentTenantId()

    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(
            requestedTenant = command.tenantId,
            actualTenant = currentTenant
        )
    }

    apply(WidgetUpdatedEvent(...))
}
```

class WidgetTest : BehaviorSpec({
    val meterRegistry = SimpleMeterRegistry()

### Security Considerations

**Fail-Closed Design Principle** [Source: security.md, Story 4.1]:
- Missing tenant context → `IllegalStateException` from `getCurrentTenantId()`
- Tenant mismatch → `SecurityException` or `TenantIsolationViolation` error
- Transaction automatically rolled back on exception (Axon default behavior)

**Defense-in-Depth Philosophy** [Source: security.md#Multi-Tenancy]:
> "Even if Layer 1 (Request Filter) is bypassed, Layer 2 (Service Validation) and Layer 3 (Database RLS) must still enforce tenant isolation. Each layer operates independently and fails closed."

**Logging Security** [Source: coding-standards-revision-2.md]:
- Log tenant mismatches at WARN level for security monitoring
- Include aggregateId, commandTenant, contextTenant in structured logs
- DO NOT log sensitive data (JWT tokens, user details)
