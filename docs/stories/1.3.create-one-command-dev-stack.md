# Story 1.3: Create One-Command Dev Stack (Docker Compose)

## Status
Done

## Story
**As a** New Developer,
**I want** a single command (e.g., `./scripts/init-dev.sh`) to launch the complete local development stack,
**so that** I can be productive on day one, meeting the <1 month productivity goal.

## Acceptance Criteria
1. A root-level `compose.yml` (or equivalent Compose file set) exists in the repository.
2. Running `./scripts/init-dev.sh` (or equivalent command) successfully starts PostgreSQL 16.1+ and Keycloak 26.0.0 using Docker Compose.
3. All services start with default users, passwords, ports, and data volumes configured for local development.

### Component Specifications
- The onboarding workflow expects `scripts/init-dev.sh` to own prerequisite checks, Compose startup, migrations, Keycloak configuration, quality gates, and optional frontend launch. `[Source: architecture/development-workflow.md#one-command-onboarding]`
- Backend readiness is determined via `http://localhost:8080/actuator/health`, while the admin portal runs on port 3000 when Node.js is available. `[Source: architecture/development-workflow.md#start-application]`
- Monitoring surfaces (Grafana on 3001, Prometheus on 9090) are part of the standard local developer experience; the Compose manifest now honors `GRAFANA_PORT` to avoid host port conflicts while defaulting to 3001. `[Source: architecture/development-workflow.md#one-command-onboarding]`

### File Locations
- Root `compose.yml` co-locates with other top-level automation assets per the canonical monorepo layout. `[Source: architecture/high-level-architecture.md#project-structure-guide]`
- Developer automation scripts live under `scripts/`, alongside SQL and Keycloak setup helpers invoked during onboarding. `[Source: architecture/high-level-architecture.md#project-structure-guide]`
- Shared Testcontainers utilities remain in `shared/testing/` to keep integration environments consistent with the local stack orchestrated by the script. `[Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]`

### Testing Requirements
- Constitutional TDD mandates executing fast Nullable-pattern tests, Testcontainers-backed integration tests, Jacoco ≥80% line coverage, and Pitest ≥80% mutation coverage during onboarding. `[Source: architecture/test-strategy-and-standards-revision-3.md#testing-philosophy]`
- The workflow prescribes running `./gradlew test`, `./gradlew integrationTest`, and related quality tasks automatically after containers are ready. `[Source: architecture/development-workflow.md#run-tests]`

### Technical Constraints
- Docker 24.x (or Podman 4.x) and multi-architecture images are required for the stack to run reliably across developer machines. `[Source: architecture/tech-stack.md#container-runtime]`
- Network bindings must expose ports 8080, 5432, 8180, 6379, 9090, and 3000/3001 to match the prescribed developer tooling surface. `[Source: architecture/tech-stack.md#network-requirements]`
- Service versions are pinned: PostgreSQL 16.1+, Keycloak 26.0.0, Redis 7.2-alpine, ensuring parity with the Testcontainers baselines. `[Source: architecture/tech-stack.md#postgresql-161-primary-database]` `[Source: architecture/tech-stack.md#keycloak-2600-identity-provider]` `[Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]`
- `application-dev.yml` configures Spring Boot to use the local PostgreSQL container and disables Axon Server auto-connect so the licensing server can boot without additional infrastructure. `[Source: architecture/high-level-architecture.md#system-architecture]`
