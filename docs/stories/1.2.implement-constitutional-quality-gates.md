# Story 1.2: Implement Constitutional Quality Gates

## Status
Approved

## Story
**As a** Core Developer,
**I want** automated quality gates for static analysis and testing integrated into the Gradle build,
**so that** all code (human or AI-generated) strictly adheres to our mandatory standards.

## Acceptance Criteria
1. The Gradle build integrates `ktlint` (1.4.0) and `Detekt` (1.23.7) with zero-tolerance failure policies; this configuration MUST be applied to modules via the convention plugins defined in Story 1.1.
2. The build process enforces the "Constitutional TDD" (RED-GREEN-Refactor) philosophy.
3. The build is configured to run Kotest and Testcontainers for the `integrationTest` source set; this configuration MUST also be applied via the convention plugins.

## Dev Notes
### Previous Story Insights
- Story 1.1 established `build-logic/` as the canonical home for convention plugins; reuse those entry points to apply all new quality gate automation. `[Source: architecture/high-level-architecture.md#project-structure-guide]`

### Quality Gate Requirements
- ktlint 1.4.0, Detekt 1.23.7, Konsist 0.17.3, and Pitest 1.15.0 are mandatory and must run with zero violations to pass the build. `[Source: architecture/tech-stack.md#quality-gate-configuration]`
- Version catalog entries remain the single source of truth for all tool versions; convention plugins may only read from `libs.versions.toml`. `[Source: architecture/coding-standards-revision-2.md#critical-requirements-must-follow]`

### Constitutional TDD Enforcement
- Constitutional TDD requires test-first development, Nullable Pattern fast tests, and real integration flows; enforcing `test`, `integrationTest`, `konsistTest`, and `pitest` in `check` keeps teams aligned with this mandate. `[Source: architecture/test-strategy-and-standards-revision-3.md#constitutional-tdd]`
- Coverage gates must assure â‰Ą85% line coverage, â‰Ą80% mutation coverage, and â‰Ą80% branch coverage across all modules. `[Source: architecture/test-strategy-and-standards-revision-3.md#quality-gates-and-coverage]`

### Testing Infrastructure
- Integration tests must run against PostgreSQL, Redis, and Keycloak Testcontainers with reuse enabled for performance. `[Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]`
- Nullable Pattern helpers in `framework/core` support fast business logic tests that precede container-heavy suites. `[Source: architecture/test-strategy-and-standards-revision-3.md#nullable-design-pattern]`

### File Locations
- Convention plugin updates live in `build-logic/src/main/kotlin/conventions/`, maintaining separate Kotlin, Spring Boot, Testing, and Quality gate entry points. `[Source: architecture/tech-stack.md#build-system]`
- Shared Testcontainers scaffolding belongs under `shared/testing/` to keep infrastructure utilities reusable across products. `[Source: architecture/high-level-architecture.md#project-structure-guide]`

### Technical Constraints
- Zero-tolerance policies forbid wildcard imports, generic exceptions, and hard-coded versions; failing tasks should clearly surface these violations. `[Source: architecture/coding-standards-revision-2.md#critical-requirements-must-follow]`
- Kotlin 2.0.10 and JVM 21 remain pinned across the build, so convention plugins must enforce the JVM toolchain. `[Source: architecture/tech-stack.md#critical-version-constraints]`

### Testing Requirements
- Kotest is the exclusive testing framework; never introduce JUnit annotations or runners. `[Source: architecture/test-strategy-and-standards-revision-3.md#testing-framework-standards]`
- Integration flow coverage must include CQRS command handling, event sourcing, and tenant isolation slices validated through Testcontainers runs. `[Source: architecture/test-strategy-and-standards-revision-3.md#testing-philosophy]`

## Testing
- Run `./gradlew clean check` to execute ktlint, Detekt, Konsist, unit, integration, and mutation suites under the enforced thresholds. `[Source: architecture/development-workflow.md#run-quality-checks]`

- Verify Testcontainers reuse by running `./gradlew integrationTest` twice and ensuring container spin-up is skipped on repeat executions. `[Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]`
