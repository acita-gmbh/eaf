# Story 4.1: Implement Layer 1 (Request Layer): TenantContext Filter

## Status
Ready for Review

## Story
**As a** Core Developer,
**I want** a request filter that extracts the validated tenant ID (from the JWT) and places it into a secure, request-scoped `TenantContext`,
**so that** all downstream application logic and database sessions can reliably access the active tenant ID.

## Acceptance Criteria
1. A `TenantContext` (likely using `ThreadLocal`) is created to hold the active `tenant_id`.
2. A Spring `Filter` is implemented that runs *after* the AuthN filter (from Epic 3).
3. The filter extracts the `tenant_id` claim (which was validated in Story 3.3) from the security principal and populates the `TenantContext`.
4. The filter implements a **"Fail-Closed" design**: If the JWT is valid but the `tenant_id` claim is missing, the request is immediately rejected (403 Forbidden).
5. The filter ensures the `TenantContext` is cleared (e.g., in a `finally` block) after the request completes to prevent context leakage.

### TenantContext Implementation Patterns
**Current Implementation Analysis**: The existing `TenantContext.kt` is an `object` that extracts tenant ID directly from Spring Security context via `SecurityContextHolder.getContext().authentication` and JWT token claims. This approach requires JWT to be present in SecurityContext for tenant extraction. [Source: framework/security/src/main/kotlin/com/axians/eaf/framework/security/tenant/TenantContext.kt]

**Refactoring Scope**: Convert from SecurityContext-based extraction to stack-based ThreadLocal implementation with comprehensive memory-leak protection:

**Stack-Based ThreadLocal API Design**:
- **Change from `object` to `class`** with ThreadLocal<Deque<WeakReference<String>>> storage for multi-request isolation
- **`setCurrentTenantId(tenantId: String)`** - pushes tenant ID onto ThreadLocal stack (stores as WeakReference) [Source: architecture/coding-standards-revision-2.md#multi-tenancy-patterns]
- **`current(): String?`** - peeks current tenant ID from stack top or returns null if stack empty
- **`getCurrentTenantId(): String`** - existing method signature maintained, calls current() and throws exception if null (fail-closed design)
- **`clearCurrentTenant()`** - pops one tenant ID from stack, fully clears ThreadLocal when stack becomes empty
- **Preserve existing `error()` behavior** - maintain fail-closed design when tenant context is missing

**Memory-Leak Protection Strategy**:
- **WeakReference Storage**: All tenant IDs stored as `WeakReference<String>` to allow garbage collection
- **Stack Management**: Use `ThreadLocal<Deque<WeakReference<String>>>` (ArrayDeque implementation recommended)
- **Automatic Cleanup**: Wire cleanup call from servlet filter request completion (filter finally block)
- **Complete ThreadLocal Removal**: When stack becomes empty, call `ThreadLocal.remove()` to prevent pooled thread contamination
- **Production Monitoring**: Optional lightweight hook to emit metric/alert if stack depth > 0 after request completion

**Implementation Requirements**:
- **Thread Safety**: All stack operations must be thread-safe within single thread context
- **Null Handling**: Graceful handling of null tenant IDs (reject and fail-closed)
- **Stack Depth Monitoring**: Track maximum stack depth for performance analysis
- **Cleanup Verification**: Unit tests must validate complete ThreadLocal cleanup
