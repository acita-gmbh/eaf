### Security Integration Architecture
The Widget API currently uses temporary manual JWT extraction (`extractTenantFromJwt` method) that returns hardcoded tenant values. This story replaces that implementation with proper Spring Security integration using the 10-layer JWT validation system from Story 3.3. [Source: architecture/security.md#3-layer-tenant-isolation]

**Current Implementation**: WidgetController.extractTenantFromJwt() returns "test-tenant" or "default-tenant" based on Authorization header format [Source: current WidgetController code lines 106-116]

**Target Architecture**:
- Layer 1: JwtValidationFilter validates JWT using 10-layer system [Source: architecture/security.md#10-layer-jwt-validation-system]
- Layer 2: Spring Security populates SecurityContext with authenticated user [Source: architecture/security.md#3-layer-tenant-isolation]
- Layer 3: TenantContext.getCurrentTenantId() extracts tenant from SecurityContext [Source: stories/3.3#tenant-context-propagation]

### API Endpoint Security Requirements
**Secured Endpoints**:
- `POST /widgets` - Requires valid JWT with tenant_id claim [Source: architecture/security.md#layer-4-claim-schema-validation]
- `GET /widgets/{id}` - Requires valid JWT with tenant_id claim [Source: architecture/security.md#layer-4-claim-schema-validation]
- `GET /widgets` - Currently implemented but not mentioned in AC, should follow same security pattern for consistency

**Security Configuration**:
- SecurityConfiguration must add endpoint protection for /widgets/** paths [Source: architecture/component-specifications.md#security-components]
- JWT authentication required for all Widget API operations [Source: architecture/security.md#implementation]
- Proper exception handling for 401 Unauthorized responses [Source: architecture/security.md#fail-closed-design]
