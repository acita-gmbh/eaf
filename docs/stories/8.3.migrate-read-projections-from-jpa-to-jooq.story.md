# Story 8.3: Migrate Read Projections from JPA to jOOQ

## Status
Approved

## Story
**As a** Developer,
**I want** to replace JPA-based CQRS read projections with jOOQ DSL,
**so that** the read model achieves optimal performance and aligns with architectural specifications.

## Business Context
During Epic 8 investigation, a critical architectural violation was identified:

**Architecture Requirement**: docs/architecture/tech-stack.md and docs/architecture/data-models.md explicitly specify **"jOOQ for read projections"**

**Current Implementation**:
- WidgetProjection uses JPA @Entity annotation
- WidgetProjectionRepository extends JpaRepository<WidgetProjection, String>
- WidgetQueryHandler relies on Spring Data JPA query methods

**Root Cause**: Epic 2 (Walking Skeleton) used JPA for rapid prototyping; jOOQ migration was deferred as technical debt

**Impact**:
- Architectural non-compliance (tech stack violation)
- Sub-optimal read query performance (JPA overhead vs jOOQ type-safe DSL)
- Missed benefits of jOOQ's compile-time query validation

**Why Now**: Epic 9 (Licensing Server MVP) will replicate Widget patterns. Must fix Widget implementation first to avoid propagating architectural debt.

### jOOQ Repository Pattern

**Example Implementation:**
```kotlin
@Repository
class JooqWidgetProjectionRepository(
    private val dsl: DSLContext
) {
    fun findByWidgetIdAndTenantId(widgetId: String, tenantId: String): WidgetProjection? {
        return dsl.selectFrom(WIDGET_PROJECTION)
            .where(WIDGET_PROJECTION.WIDGET_ID.eq(widgetId))
            .and(WIDGET_PROJECTION.TENANT_ID.eq(tenantId))
            .fetchOneInto(WidgetProjection::class.java)
    }

    fun findByTenantIdOrderByCreatedAtDesc(tenantId: String): List<WidgetProjection> {
        return dsl.selectFrom(WIDGET_PROJECTION)
            .where(WIDGET_PROJECTION.TENANT_ID.eq(tenantId))
            .orderBy(WIDGET_PROJECTION.CREATED_AT.desc())
            .fetchInto(WidgetProjection::class.java)
    }
```

### Projection Event Handler Pattern

**Example for WidgetCreatedEvent:**
```kotlin
@Component
@ProcessingGroup("widget-projection")
class WidgetProjectionHandler(
    private val dsl: DSLContext
) {
    @EventHandler
    fun on(event: WidgetCreatedEvent) {
        dsl.insertInto(WIDGET_PROJECTION)
            .set(WIDGET_PROJECTION.WIDGET_ID, event.widgetId)
            .set(WIDGET_PROJECTION.TENANT_ID, event.tenantId)
            .set(WIDGET_PROJECTION.NAME, event.name)
            .set(WIDGET_PROJECTION.DESCRIPTION, event.description)
            .set(WIDGET_PROJECTION.VALUE, event.value)
            .set(WIDGET_PROJECTION.CATEGORY, event.category)
            .set(WIDGET_PROJECTION.METADATA, objectMapper.writeValueAsString(event.metadata))
            .set(WIDGET_PROJECTION.CREATED_AT, Instant.now())
            .set(WIDGET_PROJECTION.UPDATED_AT, Instant.now())
            .execute()
    }

    @EventHandler
    fun on(event: WidgetUpdatedEvent) {
        dsl.update(WIDGET_PROJECTION)
            .set(WIDGET_PROJECTION.NAME, event.name)
            .set(WIDGET_PROJECTION.DESCRIPTION, event.description)
            .set(WIDGET_PROJECTION.VALUE, event.value)
            .set(WIDGET_PROJECTION.CATEGORY, event.category)
            .set(WIDGET_PROJECTION.METADATA, objectMapper.writeValueAsString(event.metadata))
            .set(WIDGET_PROJECTION.UPDATED_AT, Instant.now())
            .where(WIDGET_PROJECTION.WIDGET_ID.eq(event.widgetId))
            .and(WIDGET_PROJECTION.TENANT_ID.eq(event.tenantId))
            .execute()
    }
}
```

### Migration Strategy

**Recommended Approach:**
1. **Parallel Implementation**: Keep JPA, add jOOQ side-by-side
2. **Feature Flag**: Use Spring profile to toggle implementations
   - @Profile("jpa-projection") for old implementation
   - @Profile("!jpa-projection") for new jOOQ implementation
3. **Validation Tests**: Compare JPA vs jOOQ outputs for 100+ scenarios
4. **Gradual Cutover**: Default to jOOQ after validation passes

**Rollback Plan:**
- Revert Spring profile default if issues found
- JPA implementation remains in codebase for emergency rollback
- Remove JPA code only after 30 days of stable jOOQ operation

**Rollback Triggers** (When to revert to JPA):
- **Error Rate**: If application error rate increases >1% after jOOQ deployment
- **Performance Degradation**: If query latency p95 degrades >10% from JPA baseline
- **Data Inconsistency**: If projection data mismatches detected (validation test failures)
- **Critical Bug**: Any data corruption or tenant isolation violation
- **Stakeholder Request**: Product Owner or Architecture team determines risk too high

**Rollback Procedure:**
1. Revert Spring profile: Change default from `!jpa-projection` to `jpa-projection`
2. Restart application (picks up JPA implementation)
3. Verify error rate returns to baseline
4. Document rollback reason and create follow-up investigation story
