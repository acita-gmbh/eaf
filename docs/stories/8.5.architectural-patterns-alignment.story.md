# Story 8.5: Architectural Patterns Alignment

## Status
Done

## Story
**As a** Developer,
**I want** the codebase to consistently apply the Either and Nullable patterns,
**so that** the framework adheres to functional programming principles and enables faster, more robust testing.

## Business Context
During recent development, a focused architectural cleanup was performed to ensure consistent and correct usage of two key patterns:
1.  **Arrow Either Pattern**: For explicit and functional error handling, replacing exception-based control flow.
2.  **Nullable Design Pattern**: For providing fast, in-memory infrastructure substitutes in unit tests, adhering to the "Zero-Mocks Policy" for business logic.

This work was critical for aligning the codebase with the established architectural principles of functional error handling and efficient testing, as detailed in `docs/architecture/coding-standards-revision-2.md` and `docs/architecture/test-strategy-and-standards-revision-3.md`.

This story formalizes the completion of this architectural alignment effort.

## Acceptance Criteria
1.  Consistent application of the Arrow `Either` type for domain operations and error handling across relevant modules.
2.  Correct usage of the `Nullable` Design Pattern for test doubles, particularly for `JwtDecoder` and other infrastructure components in unit tests.
3.  Elimination of generic exceptions in business logic, replaced by domain-specific `Either` error types.
4.  All existing tests (unit and integration) continue to pass after the pattern alignment.
5.  Codebase adheres to the "Zero-Mocks Policy" for business logic, leveraging `Nullable` implementations for infrastructure.
6.  Documentation (e.g., `docs/architecture/coding-standards-revision-2.md`, `docs/architecture/test-strategy-and-standards-revision-3.md`) accurately reflects and reinforces the usage of these patterns.

  - [x] Subtask 1.4: Verify `Either` usage is consistent with `coding-standards-revision-2.md`.

  - [x] Subtask 2.3: Ensure `Nullable` implementations are used where appropriate to avoid mocking business logic.
  - [x] Subtask 2.4: Verify `Nullable` usage is consistent with `test-strategy-and-standards-revision-3.md`.
