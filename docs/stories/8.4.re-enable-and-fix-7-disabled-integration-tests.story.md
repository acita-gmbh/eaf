## Story
**As a** QA Engineer,

## Business Context
During Story 4.6 (Re-enable Widget Integration Tests), 7 comprehensive integration tests were disabled in `kotlin-disabled/` folders due to Kotest + Spring Boot integration complexity.

**IMPORTANT**: This story should be implemented after **Story 8.3 (Migrate Read Projections from JPA to jOOQ)**. Some re-enabled tests may query projections, and having the jOOQ implementation in place will prevent test failures.

**Current State:**
- **7 disabled tests** not executing in CI pipeline
- **Test coverage gap**: No end-to-end validation for:
  - Widget lifecycle (creation, API, event store, walking skeleton, projection processing)
  - Observability (logging context, structured logging)
- **Solution pattern exists**: TenantContextFilterIntegrationTest.kt demonstrates working @SpringBootTest pattern
- **Intentional technical debt**: Disabled to unblock Epic 4 progress, documented in git history

**Why These Tests Matter:**
- **WidgetWalkingSkeletonIntegrationTest**: Complete CQRS flow validation (command → event store → projection → query)
- **WidgetApiIntegrationTest**: REST API validation with authentication
- **WidgetEventStoreIntegrationTest**: Event persistence and retrieval
- **WidgetEventProcessingIntegrationTest**: Projection handler validation
- **LoggingContextIntegrationTest**: Structured logging with tenant context
- **StructuredLoggingIntegrationTest**: JSON log format validation

**Impact of Re-enabling:**
- Closes test coverage gap from Story 4.6
- Provides regression protection for critical flows
- Validates multi-tenancy, authentication, CQRS patterns end-to-end

### Story 4.6 Reference

**Solution Pattern Discovered in Story 4.6:**
- **File**: `framework/security/src/integration-test/kotlin/com/axians/eaf/framework/security/filters/TenantContextFilterIntegrationTest.kt`
- **Pattern**: @Autowired field injection + init block (NOT constructor injection)
- **Critical**: Plugin order matters - `id("eaf.testing")` BEFORE `id("eaf.spring-boot")`

**Anti-Pattern to Avoid:**
```kotlin
// ❌ WRONG - Causes circular dependency
@SpringBootTest
class MyTest(
    private val mockMvc: MockMvc  // Constructor injection fails
) : FunSpec({ ... })
```

**Correct Pattern:**
```kotlin
// ✅ CORRECT - Field injection works
@SpringBootTest
@ActiveProfiles("test")
class MyTest : FunSpec() {
    @Autowired
    private lateinit var mockMvc: MockMvc

    init {
        extension(SpringExtension())
        test("my test") { /* mockMvc available */ }
    }
}
```

### Testcontainers Configuration Pattern

**Standard Pattern** (from working tests):
```kotlin
@SpringBootTest
@ActiveProfiles("test")
class MyIntegrationTest : FunSpec() {
    @Autowired
    private lateinit var dependency: SomeDependency

    init {
        extension(SpringExtension())

        test("8.3-INT-001: Test description") {
            // Test logic using autowired dependencies
        }
    }

    companion object {
        @Container
        private val postgres = PostgreSQLContainer<Nothing>("postgres:16.1")
            .withDatabaseName("eaf_test")
            .withUsername("eaf")
            .withPassword("eaf")

        @Container
        private val keycloak = KeycloakContainer("quay.io/keycloak/keycloak:25.0.6")
            .withRealmImportFile("keycloak-realm-eaf-test.json")

        @JvmStatic
        @DynamicPropertySource
        fun configureProperties(registry: DynamicPropertyRegistry) {
            postgres.start()
            keycloak.start()

            registry.add("spring.datasource.url") { postgres.jdbcUrl }
            registry.add("spring.datasource.username") { postgres.username }
            registry.add("spring.datasource.password") { postgres.password }
            registry.add("spring.security.oauth2.resourceserver.jwt.issuer-uri") {
                "${keycloak.authServerUrl}/realms/eaf"
            }
        }
    }
}
```

### Investigation Strategy

**IMPORTANT**: Story 8.3 (`Migrate Read Projections from JPA to jOOQ`) discovered the likely root cause for the integration test failures: a `ClassNotFoundException` related to OpenTelemetry when using `@SpringBootTest`. A successful workaround was developed.

**Strategy**: Apply the known solution pattern from Story 8.3 instead of performing a new investigation.

**Solution Pattern (from Story 8.3):**
- Create a minimal `@TestConfiguration` for each test suite.
- Exclude problematic auto-configurations like `org.springframework.boot.actuate.autoconfigure.observation.ObservationAutoConfiguration`.
- Use `@Import` to load only the necessary configuration for the test.

**Per-Test Approach:**
1. Apply the minimal test configuration pattern to the disabled test.
2. Move the test from `kotlin-disabled/` to `kotlin/`.
3. Run the test locally to confirm the fix.
4. Document the test's purpose and coverage area.

**Timebox**: 1 day maximum per test
- If a test still fails with the known pattern, document the new blocker, create a follow-up ticket, and move on to the next test.
- If more than 2 tests are unfixable, escalate to the Product Owner and Architect to re-evaluate the story's scope and feasibility.
