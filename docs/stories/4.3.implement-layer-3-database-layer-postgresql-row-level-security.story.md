# Story 4.3: Implement Layer 3 (Database Layer): PostgreSQL Row-Level Security

## Status
Done

## Story
**As a** Core Developer,
**I want** PostgreSQL Row-Level Security wired into every tenant-owned table and activated per request via the TenantContext session variable,
**so that** the database itself enforces tenant isolation even if upper layers are bypassed.

## Acceptance Criteria
1. A database interceptor (e.g., a JDBC interceptor or JPA listener) is created that sets the current PostgreSQL session variable (e.g., `SET app.current_tenant = ?`) using the ID from the `TenantContext` (from 4.1) at the start of every transaction.
2. RLS Policies are enabled via migration scripts on the `domain_event_entry` (Event Store) and the canonical `widget_projection` (Read Model) tables.
3. The RLS policies strictly enforce that all SELECT, INSERT, UPDATE, and DELETE operations can only apply to rows where the `tenant_id` column matches the `app.current_tenant` session variable.
4. Integration tests verify that even a direct SQL query (simulating an SQL injection vulnerability) attempting to read data *without* the session variable correctly set returns zero rows.

## Dev Notes

### Previous Story Insights
- Layer 2 introduced tenant validation inside Axon command handlers and mandates generic "Access denied" messages to avoid tenant enumeration; reuse the same fail-closed error patterns to keep diagnostic output consistent. [Source: architecture/security.md#security-best-practices-for-multi-tenant-systems]

### Data Models
- Event store partitions already include `tenant_id` and BRIN indexes, so RLS must respect the existing schema (table: `eaf_event_store.domain_event_entry`). [Source: architecture.md#database-schema]
- Widget projections use the `widget_projection` table with tenant-aware unique constraints and an RLS policy referencing `current_setting('app.current_tenant')::UUID`; use this canonical naming for migrations and policy checks. [Source: architecture/coding-standards-revision-2.md#database-constraints][Source: framework/persistence/src/main/kotlin/com/axians/eaf/framework/persistence/entities/WidgetProjection.kt]

### Tenant Session Enforcement
- Story 4.3 completes the documented 3-layer tenant isolation model by linking TenantContext (Layer 1) and service checks (Layer 2) to database RLS. [Source: architecture.md#security-architecture]
- TenantContext is ThreadLocal-backed and fail-closed (`getCurrentTenantId()` throws when unset); the interceptor must consume it per transaction and log/abort if absent. [Source: architecture.md#security-architecture]

### Component Specifications
- Persistence entities and repositories for projections live under `framework/persistence/src/main/kotlin/com/axians/eaf/framework/persistence/**`, with integration tests in `framework/persistence/src/integration-test/kotlin`. [Source: architecture/component-specifications.md#projection-components]
- Security utilities (TenantContext and filters) are in `framework/security/src/main/kotlin/com/axians/eaf/framework/security/**`; reuse these modules instead of duplicating context logic. [Source: architecture/component-specifications.md#security-components]

### File Locations
- Place any JDBC/Axon interceptor or configuration classes inside an existing persistence module package (e.g., `framework/persistence/src/main/kotlin/com/axians/eaf/framework/persistence/config/`) to comply with documented structure. [Source: architecture/component-specifications.md#module-structure-standards]
- SQL migration scripts must align with the repo’s Flyway entry points invoked by `./gradlew flywayMigrate` during environment setup. [Source: architecture/development-workflow.md#initialize-database]

### Testing Requirements
- Integration suites rely on Spring Boot + Kotest with PostgreSQL Testcontainers; reuse the existing pattern that verifies tenant isolation by toggling TenantContext. [Source: architecture/test-strategy-and-standards-revision-3.md#integration-test-example]
- Tests must achieve Constitutional TDD goals (≥30% integration coverage) and avoid in-memory databases; only the documented Testcontainer stack is acceptable. [Source: architecture/test-strategy-and-standards-revision-3.md#testcontainers-configuration]

### Technical Constraints
- PostgreSQL 16.1+ is mandatory; ensure RLS statements remain compatible with that baseline and avoid unsupported syntax. [Source: architecture/tech-stack.md#postgresql-161-primary-database]
- Do not introduce H2 or alternative databases; architecture forbids them for tenant-critical flows. [Source: architecture/tech-stack.md#postgresql-161-primary-database]

- Architecture shards still reference the legacy `product_projection`; align those documents with `widget_projection` during this story to keep persistence naming consistent. [Source: architecture/index.md#quick-navigation][Source: framework/persistence/src/main/kotlin/com/axians/eaf/framework/persistence/entities/WidgetProjection.kt]

#### Executive Summary

A comprehensive RLS + Axon + Connection Pooling validation prototype was created and executed, revealing a **critical PostgreSQL session variable behavior** that required deep research to resolve. Five independent research investigations reached unanimous consensus on root cause and solution.

**Key Finding**: PostgreSQL custom GUC variables, once initialized via `SET LOCAL`, become permanently registered for the session with an **empty string default value** rather than reverting to `NULL` after transaction commit/rollback. This creates a fail-open security vulnerability that must be addressed.

#### Risk Validation Status

##### ✅ SEC-001: RLS Policy Bypass Risk - VALIDATED WITH FIX REQUIRED

**Finding**: RLS policies syntactically correct and activated, but session variable management requires defensive pattern.

**Solution**: Use `NULLIF()` wrapper in RLS policies:
```sql
-- SECURE PATTERN
CREATE POLICY tenant_isolation_policy ON sensitive_table
USING (tenant_id = NULLIF(current_setting('app.current_tenant', true), '')::UUID);
```

**Validation**:
- Pattern confirmed by PostgREST, Ruby on Rails, AWS SaaS Factory, Crunchy Data
- 0% performance overhead (inline function)
- Enforces fail-closed semantics at database layer

##### ✅ SEC-002: Session Variable Leakage Risk - CRITICAL ISSUE IDENTIFIED AND RESOLVED

**Finding**: PostgreSQL custom GUC variables follow a four-state lifecycle:
1. **Undefined** (fresh session): Returns `NULL` ✅
2. **Initialized** (after first SET): Variable registered with empty string default
3. **Cleared** (after COMMIT): Returns `''` (empty string) ❌ **VULNERABILITY**
4. **Session end only**: Only way to return to NULL state

**Root Cause**: Tom Lane (PostgreSQL core developer, 2024): "Undocumented, unsupported abuse of behavior meant for loadable extensions"

**Solution**: Two-part defense-in-depth approach:
1. **Primary**: NULLIF wrapper in RLS policies (database-enforced)
2. **Secondary**: Spring AOP aspect for explicit `RESET` after transactions

**Performance Impact**:
- NULLIF: 0% overhead
- Explicit RESET: <1% overhead
