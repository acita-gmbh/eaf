# Story 2.1: Define the 'Widget' Domain Aggregate

## Status
Approved

## Story
**As a** Core Developer,
**I want** to define a simple 'Widget' domain aggregate within its own Spring Modulith module,
**so that** we have a testable entity for proving the CQRS/ES command flow.

## Acceptance Criteria
1. A new module (e.g., `framework.widget`) is created and passes the Spring Modulith boundary checks (enforced by the gates from Epic 1).
2. The `Widget` aggregate is defined using Axon annotations (@Aggregate), adhering to Hexagonal principles.
3. The aggregate logic includes a `CreateWidgetCommand`, an `@CommandHandler`, a `WidgetCreatedEvent`, and an `@EventSourcingHandler`.
4. All logic is validated using "Constitutional TDD" (Kotest), specifically using the "Nullable Design Pattern" for fast domain tests (per NFR8).

### Data Models

- Use Arrow Either<DomainError, T> pattern for domain validation and error handling in command handlers. [Source: architecture/coding-standards-revision-2.md#exception-handling]
- Implement tenant context propagation in all commands and events using 3-layer tenant isolation pattern. [Source: architecture/coding-standards-revision-2.md#multi-tenancy-patterns]

### API Specifications
- Commands and events should be placed in `shared/shared-api/` following the shared API pattern for CQRS types. [Source: architecture/data-models.md#product-aggregate]
- Use explicit imports only - no wildcard imports allowed per coding standards. [Source: architecture/coding-standards-revision-2.md#import-management]

### Component Specifications
- Widget module must be a proper Spring Modulith module with @ApplicationModule configuration and boundary enforcement via Konsist tests. [Source: architecture/coding-standards-revision-2.md#spring-modulith-configuration]
- Module should depend only on framework.core and shared-api to maintain proper layering. [Source: architecture/high-level-architecture.md#system-architecture]

### File Locations
- Create module at `framework/widget/` following existing framework structure pattern. [Source: architecture/high-level-architecture.md#project-structure]
- Use convention plugins from `build-logic/` for consistent Gradle configuration. [Source: architecture/tech-stack.md#gradle-version-catalogs]
- Place aggregate class in `framework/widget/src/main/kotlin/com/axians/eaf/framework/widget/domain/Widget.kt`
- Place commands/events in `shared/shared-api/src/main/kotlin/com/axians/eaf/api/widget/`

### Testing Requirements
- MANDATORY: Use Kotest framework exclusively - JUnit is explicitly forbidden. [Source: architecture/test-strategy-and-standards-revision-3.md#kotest-framework-mandatory]
- Implement Nullable Design Pattern for fast domain tests (createNull() factory pattern). [Source: architecture/test-strategy-and-standards-revision-3.md#nullable-design-pattern]
- Use Testcontainers for integration tests that validate full Axon event sourcing flow. [Source: architecture/test-strategy-and-standards-revision-3.md#testing-pyramid]
- Follow Constitutional TDD: test-first development with 85% line coverage and 80% mutation coverage. [Source: architecture/test-strategy-and-standards-revision-3.md#constitutional-tdd]
- Never mock business logic - only infrastructure adapters. [Source: architecture/test-strategy-and-standards-revision-3.md#testing-philosophy]
