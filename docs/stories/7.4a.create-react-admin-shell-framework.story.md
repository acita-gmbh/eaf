## Status
Approved

## Dependencies

## Story
**As a** Framework Developer,
**I want** to create a reusable React-Admin shell framework in `framework/admin-shell/`,
**so that** product UI modules can leverage shared authentication, theming, and infrastructure without code duplication.

## Acceptance Criteria
1. `framework/admin-shell/` npm package created with React-Admin 5.4.0, Material-UI 5.x, TypeScript 5.x, and Vite 5.x bundler.
2. Admin shell provides exportable infrastructure: AdminShell component, dataProvider (JWT auth, tenant injection, RFC 7807 error mapper), authProvider (Keycloak OIDC), Axians theme, shared UI components (EmptyState, LoadingSkeleton).
3. Admin shell exposes a resource registration API allowing product ui-modules to register their resources dynamically.
4. Shell is publishable as `@axians/eaf-admin-shell` npm package with zero product-specific domain logic.

### Framework Admin Shell Patterns

**AdminShell Component Implementation Example**:
```typescript
// framework/admin-shell/src/AdminShell.tsx
import { Admin, Resource } from 'react-admin';
import type { ReactElement } from 'react';

export interface ResourceConfig {
  name: string;
  list?: React.ComponentType;
  create?: React.ComponentType;
  edit?: React.ComponentType;
  show?: React.ComponentType;
  icon?: React.ComponentType;
  options?: { label?: string };
}

export interface AdminShellProps {
  resources: ResourceConfig[];
  apiBaseUrl?: string;
  keycloakConfig?: KeycloakConfig;
  customTheme?: Theme;
}

export const AdminShell: React.FC<AdminShellProps> = ({
  resources,
  apiBaseUrl = 'http://localhost:8080/api/v1',
  keycloakConfig = defaultKeycloakConfig,
  customTheme
}) => {
  const dataProvider = createDataProvider(apiBaseUrl);
  const authProvider = createAuthProvider(keycloakConfig);
  const theme = customTheme || eafTheme;

  return (
    <Admin dataProvider={dataProvider} authProvider={authProvider} theme={theme}>
      {resources.map((resource) => (
        <Resource
          key={resource.name}
          name={resource.name}
          list={resource.list}
          create={resource.create}
          edit={resource.edit}
          show={resource.show}
          icon={resource.icon}
          options={resource.options}
        />
      ))}
    </Admin>
  );
};
```

### Security Considerations

**JWT Storage** (IMPORTANT):
- Current implementation uses localStorage for JWT tokens
- **Risk**: XSS attacks can steal tokens from localStorage
- **Mitigation Strategy** (MVP):
  1. **Short-lived tokens**: access_token expires in 15 minutes, refresh_token expires in 8 hours
  2. **Automatic refresh**: authProvider automatically refreshes access_token using refresh_token when <5 minutes remain
  3. **XSS Protection**: Implement Content Security Policy (CSP) headers via backend
  4. **Input Sanitization**: Use DOMPurify for any user-generated content rendering
  5. **Security Advisory**: README.md documents localStorage risk, recommends httpOnly cookies for production

**Future Enhancement (Post-MVP)**:
- Use httpOnly cookies (requires backend session support)
- Implement BFF (Backend-for-Frontend) pattern for token management

**Token Refresh Implementation** (authProvider):
```typescript
const authProvider = {
  checkAuth: async () => {
    const token = localStorage.getItem('token');
    if (!token) return Promise.reject();

    const decoded = jwtDecode(token);
    const expiresIn = decoded.exp * 1000 - Date.now();

    // Auto-refresh if <5 minutes remaining
    if (expiresIn < 5 * 60 * 1000) {
      const refreshToken = localStorage.getItem('refresh_token');
      const newTokens = await refreshAccessToken(refreshToken);
      localStorage.setItem('token', newTokens.access_token);
    }

    return Promise.resolve();
  }
};
```

**XSS Protection** (dataProvider):
```typescript
import DOMPurify from 'dompurify';

const dataProvider = {
  create: async (resource, params) => {
    // Sanitize all string inputs
    const sanitizedData = Object.entries(params.data).reduce((acc, [key, value]) => {
      acc[key] = typeof value === 'string' ? DOMPurify.sanitize(value) : value;
      return acc;
    }, {});

    return httpClient(`/${resource}`, {
      method: 'POST',
      body: JSON.stringify(sanitizedData),
    });
  }
};
```

**Tenant Isolation**:
- Tenant ID extracted from JWT `tenant_id` claim (Layer 1 filter pattern)
- **Fail-closed**: If tenant ID missing, reject request (don't default to any tenant)
- Automatically injected in `X-Tenant-ID` header on every API call
- **Validation**: dataProvider checks tenant ID exists before every request, throws error if missing

**CSRF Protection**:
- JWT tokens in Authorization header (not cookies) inherently resistant to CSRF
- No additional CSRF tokens needed for stateless JWT authentication
- Backend validates JWT signature (10-layer validation from Story 3.3)

### Top Issues Identified

#### ISSUE-001: JWT localStorage Storage (XSS Vulnerability) — **HIGH**
- **Risk**: SEC-001 (score 6) — XSS attacks can steal tokens from localStorage
- **Mitigation (MVP)**: Short-lived tokens (15min), DOMPurify sanitization, backend CSP headers, README advisory
- **Mitigation (Post-MVP)**: httpOnly cookies + BFF pattern
- **Tests Required**: 7.4a-UNIT-P0-001 (XSS), 7.4a-UNIT-P0-005 (expiration), 7.4a-UNIT-P0-006 (refresh)

#### ISSUE-002: Tenant Context Propagation Failure — **HIGH**
- **Risk**: SEC-002 (score 6) — Missing tenant_id → cross-tenant data exposure
- **Mitigation**: Fail-closed validation in dataProvider (reject if missing), backend 3-layer enforcement
- **Tests Required**: 7.4a-UNIT-P0-003 (tenant propagation)

#### ISSUE-003: Keycloak OIDC Flow Complexity — **HIGH**
- **Risk**: SEC-003 (score 6) — Custom OIDC implementation (no library) increases protocol error risk
- **Mitigation**: Comprehensive unit/integration testing (6 tests), security architect code review
- **Future Enhancement**: Consider oidc-client-ts library
- **Tests Required**: 7.4a-UNIT-P0-004, P0-005, P0-006, P1-004, P1-005
