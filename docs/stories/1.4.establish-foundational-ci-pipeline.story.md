# Story 1.4: Establish Foundational CI Pipeline

## Status
Done

## Story
**As a** Core Developer,
**I want** a foundational CI (Continuous Integration) pipeline (e.g., GitHub Actions workflow),
**so that** all pull requests are automatically validated against our mandatory quality gates.

## Acceptance Criteria
1. The CI pipeline triggers automatically on all pull requests targeting the main branch.
2. The CI pipeline executes all required build steps: `compile`, `check` (ktlint/Detekt), and `test` (Kotest unit tests).
3. The CI pipeline executes the full `integrationTest` suite (using Testcontainers).
4. A pull request is algorithmically blocked from merging if any quality gate (linting, Detekt, or any test) fails.

### Testing Requirements
- CI must execute ktlint, Detekt, Konsist, unit tests, integration tests, and Jacoco verification with ≥85% line and ≥80% mutation coverage to honor Constitutional TDD gates. `[Source: architecture/tech-stack.md#ci-cd-pipeline-tools]` `[Source: architecture/test-strategy-and-standards-revision-3.md#quality-gates-and-coverage]`
- Integration tests rely on Testcontainers, so runners need Docker privileges and should reuse cached containers where possible. `[Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]`
- Developers should continue running `./gradlew clean build` locally before pushing to keep CI green and prevent avoidable pipeline reruns. `[Source: architecture/development-workflow.md#run-quality-checks]`
- Running `./gradlew build` in CI ensures Gradle completes the compile phase before executing downstream quality gates, aligning with local `./gradlew clean build` guidance. `[Source: architecture/development-workflow.md#run-quality-checks]`

### Technical Constraints
- Hosted runners must use Java 21 and the pinned tooling versions (ktlint 1.4.0, Detekt 1.23.7, Konsist 0.17.3, Pitest 1.15.0) to meet zero-violation guarantees. `[Source: architecture/development-workflow.md#github-actions-workflow]` `[Source: architecture/tech-stack.md#quality-enforcement]`
- Workflow steps must fail fast—Gradle commands exit non-zero on violations—so failing checks automatically block merges per policy. `[Source: architecture/tech-stack.md#quality-enforcement]`
