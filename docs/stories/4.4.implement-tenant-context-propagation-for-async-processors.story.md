# Story 4.4: Implement Tenant Context Propagation for Async Processors

## Status
Ready for Review

## Story
**As a** Core Developer,
**I want** the `TenantContext` to be reliably propagated to all asynchronous Axon Event Processors,
**so that** the projection handlers (our read-side) can successfully write data to the RLS-protected database tables.

## Acceptance Criteria
1. A mechanism (such as an Axon Message Interceptor or custom Unit of Work component) is implemented to read the `tenant_id` from the metadata of every event being processed by a Tracking Event Processor.
2. This mechanism correctly populates the `TenantContext` (from Story 4.1) *before* the asynchronous `@EventHandler` logic executes.
3. The database interceptor (from Story 4.3) successfully reads this propagated `TenantContext` and sets the PostgreSQL session variable (`app.current_tenant_id`) for the projection's transaction.
4. The end-to-end integration tests are updated to verify that the `widget_projection` table *is successfully written to* (passing the RLS check from 4.3) and that the data is correct.
5. The implementation achieves ≥85% line coverage and ≥80% mutation coverage as measured by JaCoCo and Pitest, with all critical paths (interceptor, cleanup, RLS integration) covered by integration tests.

## Dev Notes

### Previous Story Insights (Story 4.3)
- RLS enforcement depends on `TenantContext` being populated *before* each database transaction; async processors currently lack this guarantee, which is why Story 4.4 exists. [Source: docs/stories/4.3.implement-layer-3-database-layer-postgresql-row-level-security.story.md]
- The session-variable interceptor standardizes on `app.current_tenant`; acceptance criteria still reference `app.current_tenant_id`, so expect to reconcile terminology once propagation is implemented. [Source: docs/stories/4.3.implement-layer-3-database-layer-postgresql-row-level-security.story.md]
- Several RLS subtasks remain unchecked (prototype findings, test hardening); keep those risks in mind when adding async coverage. [Source: docs/stories/4.3.implement-layer-3-database-layer-postgresql-row-level-security.story.md#tasks--subtasks]

### Tenant Context & Propagation
- `TenantContext` is a ThreadLocal + MDC carrier with explicit `set`, `current`, `clear`, and coroutine helpers; interceptors must invoke `set`/`clear` exactly once per worker thread. [Source: architecture/components.md#tenant-context-management]
- Micrometer integration via `TenantContextPropagationComponent` resolves `tenantId` from contextual data, enabling reuse for Axon worker pools when properly registered. [Source: architecture/components.md#tenant-context-management]

### Command & Event Metadata
- `TenantCommandInterceptor` already injects the active tenant into command metadata using `message.andMetaData(mapOf("tenantId" to tenantId))`; reuse the same key when enriching event messages. [Source: architecture/components.md#command-interceptors]
- `AxonConfiguration` centralizes bus configuration and exposes the hook points (`commandGateway`, `eventProcessingConfigurer`) needed to bind new interceptors and correlation data providers. [Source: architecture/components.md#axon-configuration]

### Data Models & Projections
- `widget_projection` entities live under the persistence module and expect the `tenant_id` column to be populated on writes; projections are implemented via event handlers within the widget module. [Source: architecture/data-models.md#read-projections][Source: architecture/component-specifications.md#projection-components]
- The projection tests should assert both successful writes and tenant isolation behavior, matching existing examples that rely on `TenantContext` to filter results. [Source: architecture/test-strategy-and-standards-revision-3.md#integration-testing-with-testcontainers]

### Component Locations
- Security and tenancy helpers belong under `framework/security/src/main/kotlin/com/axians/eaf/framework/security/tenant/`. [Source: architecture/component-specifications.md#security-components]
- CQRS interceptors and Axon config classes reside in `framework/cqrs/src/main/kotlin/com/axians/eaf/framework/cqrs/`. [Source: architecture/component-specifications.md#module-structure-standards]
- Projection handlers still live in `framework/widget/src/main/kotlin/com/axians/eaf/framework/widget/projections/`; ensure async context arrives before they execute. [Source: architecture/component-specifications.md#projection-components]

### Technical Constraints
- Implementation must remain on Kotlin 2.2.20, Spring Boot 3.5.6, Axon 4.12.1, and PostgreSQL 16.1+ as mandated by the tech stack. [Source: architecture/tech-stack.md#critical-version-constraints][Source: architecture/tech-stack.md#framework-stack]

### Project Structure Notes
- `framework/cqrs`, `framework/security`, and `framework/widget` modules exist but their `src/main/kotlin` directories are currently empty; plan to scaffold the documented packages before adding interceptors or handlers.
- The repository does **not** contain the `framework/tenancy` module referenced in some docs—reuse the security module’s `tenant` package for propagation code and document the deviation.
- Ensure new tests land under `framework/widget/src/integration-test/kotlin`; the existing directory structure is present even though it lacks Kotlin sources.
