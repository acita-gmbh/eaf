# Story 8.6: Enable Mutation Testing (Pitest) for Framework Modules

## Status
Ready for Review

## Story
**As a** Developer,
**I want** mutation testing (pitest) enabled and functional on framework modules where business logic resides,
**so that** we can validate test quality beyond simple line coverage and ensure our tests actually detect bugs.

## Business Context

**Current State - Configured But Inactive:**

Mutation testing (pitest) is comprehensively configured in the `QualityGatesConventionPlugin` with appropriate settings:
- ✅ Pitest version 1.19.0-rc.1 (Gradle 9.1.0 compatible)
- ✅ Kotest integration (`testPlugin = "kotest"`)
- ✅ STRONGER mutators for aggressive testing
- ✅ 80% mutation threshold, 85% coverage threshold
- ✅ Multi-threaded execution

**However**: Pitest is **NOT RUNNING** anywhere in the codebase due to architectural plugin mismatch:

**Problem: Hexagonal Architecture Mismatch**

1. **Framework Modules** (where business logic lives):
   - `framework/security/` - JWT validation, tenant isolation (substantial logic)
   - `framework/cqrs/` - Event sourcing, tenant context propagation
   - `framework/core/` - Domain patterns, utilities
   - **Plugin Status**: Only apply `eaf.kotlin-common`, `eaf.testing`
   - **Result**: ❌ No `eaf.quality-gates` → No pitest task

2. **Product Modules** (thin wrappers):
   - `products/widget-demo/` - Only `@SpringBootApplication` + config
   - `products/licensing-server/` - Only `@SpringBootApplication`
   - **Plugin Status**: Apply `eaf.quality-gates`
   - **Result**: ✅ Pitest task exists BUT **intentionally disabled** (lines 160-165)
   - **Reason**: "No production code to mutation test (just @SpringBootApplication)"

**CI Behavior:**
- `mutation-testing` job runs: `./gradlew pitest`
- Both product modules SKIPPED (enabled = false)
- Build succeeds because `failWhenNoMutations.set(false)`
- **CI reports**: ✅ PASS (green) but **0 mutations tested**

**Impact:**
- Quality gate exists in name only
- No validation that tests actually detect bugs
- False sense of quality assurance
- Configured thresholds (80% mutation, 85% coverage) never enforced

**Why This Matters:**
- Line coverage can reach 100% with tests that never assert anything
- Mutation testing validates test **quality** by injecting bugs
- Industry best practice for critical code (security, multi-tenancy)
- Constitutional TDD principle: tests must fail when code breaks

**Why Now:**
- Epic 8 focuses on code quality and alignment
- Recent security fixes (PR #60) added 18 new tests - validate they're effective
- Framework modules now have substantial testable logic
- Before Epic 9 (Production MVP), ensure test suite quality

## Dev Notes

### Mutation Testing Overview

**What is Mutation Testing?**
Mutation testing validates test **quality** by introducing small bugs (mutants) into production code and verifying tests catch them.

**Example Mutations:**
- Change `>` to `>=` (boundary condition)
- Change `&&` to `||` (boolean logic)
- Return `null` instead of actual value
- Remove method calls (verify side effects tested)
- Negate conditionals (`if (x)` → `if (!x)`)

**Why Line Coverage Isn't Enough:**
```kotlin
// 100% line coverage but useless test:
fun validateTenant(id: String): Boolean {
    return id == currentTenant  // Line executed
}

// Bad test (passes but doesn't validate):
test("should validate tenant") {
    validateTenant("any-id")  // ✅ Line covered, ❌ No assertion!
}

// Mutation testing would catch this - mutant survives
```

### Implementation Options

**Option A: Add quality-gates Plugin** (Recommended)
```kotlin
// framework/security/build.gradle.kts
plugins {
    id("eaf.kotlin-common")
    id("eaf.observability")
    id("eaf.testing")
    id("eaf.quality-gates") // ADD THIS
}
```

**Pros:**
- Enables pitest + jacoco + detekt
- Consistent with product modules
- Automatic threshold enforcement

**Cons:**
- Adds detekt (currently disabled for Kotlin 2.2.20 compatibility)
- May need to disable detekt initially

**Option B: Apply Pitest Plugin Directly**
```kotlin
// framework/security/build.gradle.kts
plugins {
    id("info.solidsoft.pitest") version "1.15.0"
}

pitest {
    // Duplicate configuration from QualityGatesConventionPlugin
    pitestVersion.set("1.19.0-rc.1")
    testPlugin.set("kotest")
    // ... etc
}
```

**Pros:**
- Granular control
- No dependency on quality-gates plugin

**Cons:**
- Configuration duplication
- Harder to maintain consistency

**Recommendation**: Use Option A, disable detekt temporarily if needed

### Common Surviving Mutants

**Category 1: Boundary Conditions**
```kotlin
// Original: if (value > MAX)
// Mutant: if (value >= MAX)
// Fix: Add test at exactly MAX value
```

**Category 2: Null Handling**
```kotlin
// Original: return value ?: default
// Mutant: return value ?: null
// Fix: Test both null and non-null paths
```

**Category 3: Boolean Logic**
```kotlin
// Original: if (a && b)
// Mutant: if (a || b)
// Fix: Test all combinations (TT, TF, FT, FF)
```

**Category 4: Return Values**
```kotlin
// Original: return Either.right(result)
// Mutant: return Either.left(error)
// Fix: Assert on Either value, not just that it returns
```

### Risk Assessment

**Risk 1: Detekt Compatibility** (MEDIUM)
- quality-gates plugin includes detekt
- Detekt 1.23.8 incompatible with Kotlin 2.2.20
- **Mitigation**: Disable detekt in framework modules until detekt 2.0.0 stable

**Risk 2: Long Execution Time** (LOW)
- Mutation testing can be slow (exponential with code size)
- **Mitigation**: 45-minute CI timeout, multi-threaded execution, can skip in local dev

**Risk 3: Flaky Mutants** (LOW)
- Some mutants may be equivalent (unfixable)
- **Mitigation**: Document and accept equivalent mutants with rationale
