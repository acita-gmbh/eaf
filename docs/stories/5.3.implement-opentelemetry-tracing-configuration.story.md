### Trace Context Propagation Patterns

**HTTP Requests** (Automatic):
- Spring Boot auto-instrumentation handles W3C Trace Context headers (traceparent, tracestate)
- No manual code required - enabled via `opentelemetry-spring-boot-starter`

**Axon Commands** (Manual):
```kotlin
@Component
class TracingCommandInterceptor : MessageDispatchInterceptor<CommandMessage<*>> {
    override fun handle(messages: List<CommandMessage<*>>): BiFunction<Int, CommandMessage<*>, CommandMessage<*>> {
        return BiFunction { _, message ->
            val span = Span.current()
            val traceContext = mapOf(
                "trace_id" to span.spanContext.traceId,
                "span_id" to span.spanContext.spanId,
                "trace_flags" to span.spanContext.traceFlags.asHex()
            )
            message.andMetaData(traceContext)
        }
    }
}
```
[Source: docs/architecture/components.md#command-interceptors]

**Axon Events** (Manual with Async Support):
```kotlin
@Component
class TracingEventInterceptor : MessageHandlerInterceptor<EventMessage<*>> {
    override fun handle(unitOfWork: UnitOfWork<out EventMessage<*>>, chain: InterceptorChain): Any {
        val message = unitOfWork.message
        val traceId = message.metaData["trace_id"] as? String
        val spanId = message.metaData["span_id"] as? String

        // Restore trace context for async event handlers
        return if (traceId != null && spanId != null) {
            val context = SpanContext.createFromRemote(traceId, spanId, TraceFlags.getSampled(), TraceState.getDefault())
            val span = tracer.spanBuilder("event-handler").setParent(Context.current().with(Span.wrap(context))).startSpan()
            span.makeCurrent().use { chain.proceed() }
        } else {
            chain.proceed()
        }
    }
}
```

### Tenant-Aware Span Tagging

Following the tenant tagging strategy from Story 5.2 CustomMetrics:
- **Graceful Fallback**: Use `tenantContext.current() ?: "system"` for infrastructure observability
- **Span Attributes**: `tenant_id`, `service_name`, `environment` (standard OTel semantic conventions)
- **Purpose**: Enable multi-tenant filtering in trace analysis without breaking system-level operations

```kotlin
@Component
class TenantAwareTracingFilter(
    private val tenantContext: TenantContext
) : OncePerRequestFilter() {
    override fun doFilterInternal(request: HttpServletRequest, response: HttpServletResponse, chain: FilterChain) {
        val span = Span.current()
        val tenantId = tenantContext.current() ?: "system"
        span.setAttribute("tenant_id", tenantId)
        chain.doFilter(request, response)
    }
}
```

### Infrastructure Interceptor Exception Pattern

If tracing interceptors need to catch exceptions for observability (record span status), follow the legitimate generic catch pattern:
```kotlin
@Suppress("TooGenericExceptionCaught")
override fun handle(unitOfWork: UnitOfWork<*>, chain: InterceptorChain): Any {
    val span = tracer.spanBuilder("command-handler").startSpan()
    return try {
        span.makeCurrent().use { chain.proceed() }
    } catch (ex: Exception) {
        // Infrastructure observability only - record span error then re-throw
        span.recordException(ex)
        span.setStatus(StatusCode.ERROR)
        throw ex
    } finally {
        span.end()
    }
}
```

### Troubleshooting Common OpenTelemetry Integration Issues

**Symptom: Traces not appearing in OTLP collector/backend**
- **Cause 1**: OTLP exporter endpoint misconfigured or unreachable
  - **Solution**: Verify `otel.exporter.otlp.endpoint=http://localhost:4318` matches Docker Compose service port; check collector container logs
- **Cause 2**: Sampling probability set to 0.0
  - **Solution**: Ensure `management.tracing.sampling.probability=1.0` in application.yml for development/testing
- **Cause 3**: OpenTelemetry SDK not initialized before application startup
  - **Solution**: Verify `TracingConfiguration.kt` is discovered by Spring Boot auto-configuration; check `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` if using custom starter pattern

**Symptom: Trace context not propagated across Axon command/event boundaries**
- **Cause**: Axon metadata not carrying trace context
  - **Solution**: Verify `TracingCommandInterceptor` and `TracingEventInterceptor` are registered in AxonConfiguration; confirm metadata contains "trace_id", "span_id", "trace_flags" keys using debug logging
- **Root Cause**: Interceptor execution order matters - tracing interceptors must run BEFORE tenant validation interceptors
  - **Solution**: Register tracing interceptors first in interceptor chain configuration

**Symptom: `trace_id` missing from JSON logs**
- **Cause 1**: MDC enrichment not executing
  - **Solution**: Verify `EafMDCConfiguration` bean is active; check that `Span.current()` returns a valid span context (not invalid/no-op span)
- **Cause 2**: Logback encoder not configured to include MDC fields
  - **Solution**: Ensure `<mdcIncludedFields>trace_id,span_id,tenant_id</mdcIncludedFields>` is present in logback-spring.xml encoder configuration

**Symptom: Trace context lost in async/coroutine operations**
- **Cause**: ThreadLocal context not propagated to new threads
  - **Solution**: Leverage Micrometer Context Propagation (already configured in Story 4.4) which automatically bridges OpenTelemetry context to async boundaries; verify `spring-boot-starter-actuator` includes context propagation dependencies
- **Advanced**: If custom ExecutorService/thread pools are used, wrap with OpenTelemetry's `Context.taskWrapping()` decorator

**Symptom: High memory usage or performance degradation**
- **Cause**: 100% sampling with high request volume
  - **Solution**: Reduce `management.tracing.sampling.probability` to 0.1 (10%) for production environments; use adaptive sampling strategies for production
- **Cause**: Span data accumulation when exporter is unavailable
  - **Solution**: Configure batch span processor with reasonable limits (`otel.bsp.max.queue.size=2048`, `otel.bsp.max.export.batch.size=512`)

**Symptom: Tests failing with "no current span" errors**
- **Cause**: Test environment doesn't initialize OpenTelemetry SDK
  - **Solution**: Ensure `@SpringBootTest` tests include `TracingConfiguration` in component scan; use `Tracer.spanBuilder().startSpan()` to create explicit span context in unit tests
- **Alternative**: For fast unit tests using Nullable Pattern, create a `NullableTracer` that returns no-op spans

**Reference**: These patterns are common across OpenTelemetry Spring Boot integrations; consult [OpenTelemetry Java Instrumentation docs](https://opentelemetry.io/docs/instrumentation/java/) for advanced troubleshooting.
