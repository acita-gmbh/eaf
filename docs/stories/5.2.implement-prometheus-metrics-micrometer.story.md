# Story 5.2: Implement Prometheus Metrics (Micrometer)

## Status
Draft

## Story
**As an** Operator,
**I want** all services built on the EAF to expose standardized application metrics,
**so that** we can monitor system health via a Prometheus-compatible endpoint.

## Acceptance Criteria
1. The Spring Boot Actuator, Micrometer, and Prometheus registry dependencies are added and managed via the convention plugins.
2. The `/actuator/prometheus` endpoint is enabled, secured (requires AuthN from Epic 3), and exposed.
3. Core metrics (JVM, HTTP server requests, Axon message processing) are exposed by default.
4. All relevant metrics are automatically tagged with `tenant_id` (where applicable) and `service_name` to allow for granular filtering.

## Dev Notes

### Previous Story Insights
- Story 5.1 introduced the `framework/observability` module with structured logging, creating the MDC context wiring and convention plugin pattern we can extend for metrics; reuse its tenant-aware context strategy to ensure tagging consistency. [Source: docs/stories/5.1.implement-standardized-structured-logging.story.md#dev-agent-record]

### Data Models
- `TenantContext` exposes helpers for retrieving the active tenant, user tier, and coroutine propagation; leverage these utilities when tagging metrics to prevent cross-tenant leakage. [Source: docs/architecture/components.md#tenant-context-management]

### API Specifications
- Spring Boot Actuator is already part of the baseline stack; only `/actuator/health` should remain unauthenticated while additional endpoints (including `/actuator/prometheus`) must require JWT authentication per security guidance. [Source: docs/architecture/components.md#security-configuration]

### Component Specifications
- Metrics configuration is expected under `framework/observability/src/main/kotlin/com/axians/eaf/framework/observability/MetricsConfiguration.kt`, providing beans for `PrometheusMeterRegistry` and custom counters/timers for Axon flows. [Source: docs/architecture/components.md#observability-components]
- Tenant-aware Micrometer context propagation is established through the documented `TenantContextPropagationComponent`, ensuring async workers inherit tenant metadata. [Source: docs/architecture/components.md#tenant-context-management]

### File Locations
- Observability cross-cutting components belong in `framework/observability/`, with module boundaries enforced by Spring Modulith metadata. [Source: docs/architecture/high-level-architecture.md#project-structure]
- Security filter-chain adjustments occur in `framework/security/src/main/kotlin/com/axians/eaf/framework/security/config/SecurityFilterChainConfiguration.kt`, which currently permits all `/actuator/**` requests and must be tightened. [Source: docs/architecture/components.md#security-configuration]
- Product-level configuration for management endpoints lives in `products/{application}/src/main/resources/application.yml`. [Source: docs/architecture/high-level-architecture.md#project-structure]

### Testing Requirements
- Follow Constitutional TDD using Kotest 6.0.3 with Testcontainers; integration tests validating infrastructure changes belong in `src/integration-test/kotlin`. [Source: docs/architecture/test-strategy-and-standards-revision-3.md#testing-framework-standards]
- Maintain ≥85% line and ≥80% mutation coverage; reuse patterns from Epic 4 stories for infrastructure verification. [Source: docs/architecture/test-strategy-and-standards-revision-3.md#constitutional-tdd]

### Technical Constraints
- All new dependencies must be added through the Gradle version catalog and applied via convention plugins—inline version declarations are forbidden. [Source: docs/architecture/coding-standards-revision-2.md#critical-requirements]
- Metrics must respect multi-tenant isolation: fail closed when `TenantContext` is missing and avoid leaking tenant identifiers across threads. [Source: docs/architecture/components.md#tenant-context-management]
- Compose-based developer stack exposes Prometheus on port 9090; ensure the service remains compatible with the documented local environment. [Source: docs/architecture/development-workflow.md#services-available]

### Project Structure Notes
- `docs/architecture/component-specifications.md` does not yet list the observability metrics classes; update the shard in Task 2.4 so file locations remain discoverable for future work. [Source: docs/architecture/component-specifications.md]
