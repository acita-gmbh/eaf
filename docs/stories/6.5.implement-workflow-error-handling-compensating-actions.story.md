# Story 6.5: Implement Workflow Error Handling (Compensating Actions)

## Status
Approved

## Story
**As a** Core Developer,
**I want** the framework to support compensating commands for failed workflows,
**so that** we can manage the distributed transaction risk identified in our analysis (Risk Assessment 5).

## Acceptance Criteria
1. A "compensating command" (e.g., `CancelWidgetCreationCommand`) and corresponding event handler are added to the Widget aggregate (from Epic 2).
2. A BPMN "Error Boundary Event" is configured in the workflow.
3. If a downstream step (like the Ansible Task) fails, the BPMN error path must successfully trigger the `DispatchAxonCommandTask` (from 6.2) to send the `CancelWidgetCreationCommand`, reversing the initial transaction.

### Key Implementation Decisions

**1. Generic Delegate + Variable Management Pattern**:
- DispatchAxonCommandTask remains generic (supports any command type via `commandType` variable)
- BPMN compensation flows require explicit `commandType` update before compensation task
- Solution: SetCommandTypeDelegate (custom JavaDelegate) sets variable without scripting engines
- **Architectural Benefit**: No Groovy/JavaScript runtime dependencies in test classpath

**2. Framework Test Purity**:
- E2E compensation test uses TestEntity aggregate (framework test infrastructure)
- Widget aggregate remains in products module (architectural boundary respected)
- SetCommandTypeDelegate created in axonIntegrationTest scope (test-only)
- **Lesson**: Framework tests must NOT depend on products module

**3. FlowableMetrics Optional Injection**:
- Changed from required to optional (`FlowableMetrics?`) in DispatchAxonCommandTask
- Reason: AxonIntegrationTestConfig explicitly doesn't provide FlowableMetrics bean
- Uses safe-call operator (`?.`) for telemetry recording
- **Benefit**: Delegate works in both production (with metrics) and test (without metrics) contexts

**4. Idempotency Strategy**:
- Widget.handle(CancelWidgetCreationCommand) checks if `status == CANCELLED`
- Returns `Unit.right()` without emitting duplicate event
- TestEntityAggregate simplified (no status field for test infrastructure)
- **Production Pattern**: Status-based idempotency guard

**5. BPMN Element Ordering (Critical Learning)**:
- BPMN 2.0 XSD requires: `<documentation>` BEFORE `<extensionElements>`
- Violating this order causes SAXParseException during process deployment
- Validated via Consult7 with Gemini thinking mode
- **Tool Used**: mcp__consult7__consultation with schema analysis

### Security Hardening (2025-10-02 18:50 UTC)

**SEC-VULN-001 - Code Injection via Insufficient Reflection Whitelist** (RESOLVED):

**Finding** (Security Review):
- Pure reflection with `Class.forName(commandClassName)` used package prefix validation only
- `commandClassName.startsWith("com.axians.eaf.")` insufficient - allows ANY class in namespace

- Added `ALLOWED_COMMAND_CLASSES` companion object with explicit whitelist
- Fail-closed validation: Only 5 specific command classes allowed
- Replaced `startsWith()` check with `in ALLOWED_COMMAND_CLASSES`
- Defense-in-depth: Protects even if perimeter controls (CI/CD, code review) fail
