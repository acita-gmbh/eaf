# Story 9.1: Implement React-Admin Consumer Application

## Status
Done

## Story
**As an** Administrator,
**I want** a functional React-Admin portal that integrates the framework shell with product UI modules,
**so that** I can manage widgets through a modern web interface.

## Business Context
Epic 7.4 implemented a **micro-frontend architecture** (validated by Winston) that mirrors the backend framework/product separation from Story 4.5:

**Current State - Partial Implementation:**
- ✅ **Framework Admin Shell** (`framework/admin-shell/`) - Published npm library @axians/eaf-admin-shell
  - Provides: Authentication, theming, data providers, shared components
  - Version: 0.1.0
  - Dependencies: react-admin 5.4.0, Material UI 5.x
  - Build: Vite library mode with TypeScript

- ✅ **UI Resource Generator** (`eaf scaffold ui-resource`) - CLI tool functional
  - Generates product-specific UI modules
  - Creates List, Edit, Create, Show components
  - Imports framework shell components

- ❌ **Consumer Application** (`apps/admin/`) - MISSING (THIS STORY)
  - Only placeholder files: build.gradle.kts (737 bytes), package-lock.json (84 bytes empty)
  - No package.json, no src/, no React app
  - This is the integration layer that ties everything together

**Architecture Pattern** (docs/architecture/frontend-architecture-decision.md):
- **Option 3 (Hybrid)**: Framework shell + Product UI modules + Consumer app
- Mirrors backend separation: framework (publishable) / products (domain-specific)
- Consumer app is thin integration layer that dynamically registers product resources

**Backend Infrastructure Ready:**
- REST API: WidgetController with full CRUD
- Authentication: Keycloak OIDC integration
- Projections: Widget read model (JPA now, jOOQ after Story 8.2)

**Why Now:** Epic 9 (Licensing Server MVP) Story 9.4 requires React-Admin UI for validation. This story is the foundational step.

### Accessibility Requirements (WCAG AA)

**PRD Requirement** (docs/prd.md): "Accessibility: WCAG AA"

**WCAG AA Standards to Validate:**

1. **Keyboard Navigation** (2.1.1 Keyboard):
   - All functionality available via keyboard only
   - No keyboard traps
   - Logical tab order

2. **Screen Reader Compatibility** (4.1.2 Name, Role, Value):
   - ARIA labels on all form controls
   - ARIA live regions for dynamic content
   - Semantic HTML structure

3. **Color Contrast** (1.4.3 Contrast):
   - Normal text: ≥4.5:1 contrast ratio
   - Large text (18pt+): ≥3:1 contrast ratio
   - UI components: ≥3:1 contrast ratio

4. **Focus Visible** (2.4.7 Focus Visible):
   - Clear focus indicators on all interactive elements
   - Focus never hidden or indistinguishable

5. **Labels or Instructions** (3.3.2 Labels or Instructions):
   - All form inputs have associated labels
   - Error identification clear and accessible

**Validation Tools:**
- **Browser Extensions**: axe DevTools, WAVE
- **Screen Readers**: NVDA (Windows), VoiceOver (Mac), JAWS
- **Contrast Checkers**: Chrome DevTools, WebAIM Contrast Checker

**React-Admin Accessibility Notes:**
- Material UI provides baseline ARIA support
- Custom components may need additional ARIA attributes
- Test all custom forms and interactions

**Severity**: MEDIUM (Configuration Management)

**Target**: Before Multi-Environment Deployment

- JWT issuer and audience hardcoded as constants
- Requires code recompilation to change realms or environments
- Cannot support dev/staging/production with different realms
- Generic `account` audience (Keycloak default) less secure than app-specific

**Current Configuration**:
```kotlin
private const val EXPECTED_ISSUER = "http://localhost:8180/realms/eaf-test"
```

**Required Actions**:

   ```kotlin
   @Value("\${eaf.security.jwt.expected-issuer}")

   ```

   ```yaml
   eaf:
   ```
