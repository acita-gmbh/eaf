# Story 2.4: Implement the Query Side (API)

## Status
Approved

## Story
**As a** Core Developer,
**I want** to query the "Widget" read model via a simple API,
**so that** I can retrieve the current state of the system and prove the "read-side" of CQRS is working.

## Acceptance Criteria
1. A Query (e.g., `FindWidgetQuery`) and a corresponding `@QueryHandler` are implemented.
2. A new API endpoint (e.g., `GET /widgets/{id}`) is created which dispatches the query.
3. The Query Handler interacts *only* with the `widget_projection` table (the read model), never the event store.
4. A final end-to-end integration test (using Kotest/Testcontainers) confirms the full "Walking Skeleton" flow: (1) POST command, (2) (Wait for projection), (3) GET query, (4) Assert the returned data matches the command data.

### Data Models
- WidgetProjection entity is fully implemented with tenant isolation, audit fields, and PostgreSQL jsonb support for metadata queries. [Source: stories/2.3.implement-read-side-projection.story.md#file-list]
- Use QueryGateway from Axon Framework to dispatch queries to query handlers, following established patterns. [Source: architecture/data-models.md#query-handlers]
- Query structures should follow the FindProductByIdQuery pattern with tenant context propagation. [Source: architecture/data-models.md#query-handlers]
- WidgetResponse DTO should include: id (String), name (String), description (String?), value (BigDecimal), category (String), metadata (Map<String, Any>?), createdAt (Instant) - excluding sensitive fields like tenantId and updatedAt
- Pagination responses should use Spring Data Page<T> structure with totalElements, totalPages, size, number, and content properties for consistent API contracts

### API Specifications
- REST API endpoints must follow RESTful principles with resource-oriented URLs and proper HTTP semantics. [Source: architecture/api-specification-revision-2.md#api-design-principles]
- GET endpoints require JWT Bearer token authentication with multi-tenant isolation via token claims. [Source: architecture/api-specification-revision-2.md#security-first]
- Error responses must use RFC 7807 Problem Details format with structured error information. [Source: architecture/api-specification-revision-2.md#error-handling]
- Pagination parameters: page (int, default 0), size (int, default 20, max 100), sort (String array, format: "field.direction"), category (String, optional filter), search (String, optional name filter)
- Pagination responses include: content (List<WidgetResponse>), totalElements (Long), totalPages (Int), size (Int), number (Int), first (Boolean), last (Boolean)
