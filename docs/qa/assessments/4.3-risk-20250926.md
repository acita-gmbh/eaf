## Critical Risks Requiring Immediate Attention

### 1. [SEC-001]: PostgreSQL RLS Policy Bypass

**Score: 9 (Critical)**
**Probability**: High (3) - Complex interaction with connection pooling and transaction management
**Impact**: High (3) - Complete tenant isolation failure, cross-tenant data access

**Description**: If the database interceptor fails to set `app.current_tenant` correctly before a transaction begins, or if the session variable is not properly scoped to the transaction, RLS policies will fail to enforce tenant isolation. This could occur due to:
- Interceptor not executing before query execution
- Connection pooling returning connections with stale/wrong session variables
- Transaction boundaries not aligned with session variable lifecycle
- Axon Framework's complex transaction management bypassing interceptor

**Affected Components**:
- Database interceptor (`framework/persistence/.../config/TenantSessionInterceptor`)
- RLS policies on `domain_event_entry` and `widget_projection`
- Connection pool configuration
- Axon transaction management hooks

**Mitigation**:
- **Preventive Actions**:
  - Implement interceptor at multiple levels (connection acquisition AND transaction start)
  - Add mandatory session variable reset on connection release/return to pool
  - Use PostgreSQL's `SET LOCAL` for transaction-scoped variables instead of `SET`
  - Add pre-query validation that session variable is set correctly
- **Detective Actions**:
  - Log every session variable set/reset operation with transaction ID
  - Add query-level assertion that `current_setting('app.current_tenant')` matches expected tenant
  - Implement metrics tracking session variable set failures
- **Testing Requirements**:
  - Simulate connection pool exhaustion and reuse scenarios
  - Test with concurrent transactions from different tenants
  - Verify session variable isolation across pooled connections
  - Test Axon event replay with RLS enabled
  - Simulate interceptor failure scenarios

**Residual Risk**: Medium - Even with comprehensive controls, edge cases in Axon's transaction management may exist
**Owner**: Dev Team
**Timeline**: Must be resolved before Story 4.3 implementation begins

### 2. [SEC-002]: Session Variable Leakage Across Pooled Connections

**Score: 9 (Critical)**
**Probability**: High (3) - Connection pooling is complex and error-prone
**Impact**: High (3) - Cross-tenant data contamination, audit trail corruption

**Description**: PostgreSQL session variables (set via `SET`) persist for the lifetime of a connection. When using connection pooling (HikariCP, PgBouncer), connections are reused across different requests and potentially different tenants. If the session variable is not properly reset when a connection is returned to the pool, the next request could inherit the previous tenant's context, leading to:
- User from Tenant A seeing data from Tenant B
- Events written to wrong tenant's partition
- Audit logs showing incorrect tenant association

**Affected Components**:
- HikariCP connection pool configuration
- Connection lifecycle hooks
- Database interceptor cleanup logic
- All database queries (event store and projections)

**Mitigation**:
- **Preventive Actions**:
  - Configure HikariCP with connection test query that validates session state
  - Implement `connectionInitSql` that explicitly resets session variables
  - Use `SET LOCAL` instead of `SET` to scope variables to transaction only
  - Add connection validation that fails fast if session variable incorrect
  - Consider using PostgreSQL's `reset_all` on connection acquisition
- **Detective Actions**:
  - Add connection pool metrics tracking tenant context per connection
  - Log warnings when session variable doesn't match expected tenant
  - Implement audit trail validation that detects tenant mismatches
- **Testing Requirements**:
  - Test with minimal connection pool (1-2 connections) and multiple tenants
  - Simulate rapid tenant switching on same connection
  - Verify session variable reset on connection return
  - Load test with high concurrency and tenant variance

**Residual Risk**: Low with `SET LOCAL` approach, Medium with `SET` approach
**Owner**: Dev Team
**Timeline**: Must be resolved before Story 4.3 implementation begins

### 3. [TECH-001]: Axon Framework Transaction Management Compatibility

**Score: 9 (Critical)**
**Probability**: High (3) - Axon has complex transaction boundaries and event replay mechanisms
**Impact**: High (3) - RLS could block legitimate operations, event sourcing broken

**Description**: Axon Framework 4.9.4 manages transactions in complex ways:
- Command handlers may span multiple transaction boundaries
- Event replay during projection rebuilds may not have tenant context
- Sagas and event processors may execute outside normal request lifecycle
- Axon's `@Transactional` boundaries may not align with interceptor execution

If RLS policies are enabled without proper integration with Axon's transaction management, legitimate operations could be blocked:
- Event replay fails because no tenant context available
- Projection updates fail due to missing session variable
- Aggregate loading blocked by RLS during command processing

**Affected Components**:
- Axon command handlers
- Event store (`domain_event_entry` table)
- Projection handlers
- Event replay mechanisms
- Saga implementations

**Mitigation**:
- **Preventive Actions**:
  - Review Axon documentation for recommended multi-tenancy patterns
  - Implement tenant context propagation in Axon's `UnitOfWork`
  - Add special handling for system operations (replay, admin)
  - Consider RLS policy exemptions for specific system users
  - Use Axon transaction lifecycle hooks to ensure interceptor execution
- **Detective Actions**:
  - Add metrics tracking RLS policy violations
  - Log all database permission errors with context
  - Implement health checks that verify RLS policy state
- **Testing Requirements**:
  - Test event replay with and without tenant context
  - Verify projection rebuilds work with RLS enabled
  - Test saga execution across tenant boundaries
  - Simulate aggregate loading under RLS constraints
  - Test dead letter queue processing (no tenant context)

**Residual Risk**: Medium - Axon's complexity means edge cases likely exist
**Owner**: Dev Team + Axon SME
**Timeline**: Must be resolved before Story 4.3 implementation begins

## High Risks Requiring Mitigation

### 4. [PERF-001]: RLS Performance Impact on Event Store Operations

**Score: 6 (High)**
**Probability**: Medium (2) - RLS adds overhead to every query
**Impact**: High (3) - Could violate p95 <200ms latency SLA

**Description**: Row-Level Security adds a WHERE clause condition to every query against protected tables. For the event store table (`domain_event_entry`), which is queried frequently during:
- Aggregate loading (SELECT events for aggregate ID)
- Event replay (SELECT all events in order)
- Projection updates (SELECT events since last token)

This additional filter could degrade query performance, especially if:
- RLS policy is not optimized
- Indexes don't include `tenant_id` as first column
- `current_setting()` function call adds overhead

**Affected Components**:
- `domain_event_entry` table and all queries
- Aggregate loading performance
- Projection lag metrics
- Command handler response times

**Mitigation**:
- **Preventive Actions**:
  - Ensure all indexes include `tenant_id` as leading column
  - Verify BRIN indexes are effective with RLS filters
  - Use PostgreSQL's `EXPLAIN ANALYZE` to verify query plans
  - Consider materialized session variable if `current_setting()` shows overhead
  - Benchmark query performance before/after RLS enablement
- **Testing Requirements**:
  - Load test command processing with RLS enabled
  - Measure aggregate loading times with large event streams
  - Test projection lag under high event volume
  - Verify p95 latency remains <200ms

**Residual Risk**: Low - Performance impact should be minimal with proper indexes
**Owner**: Dev Team
**Timeline**: Before Story 4.3 completion

### 5. [SEC-003]: SQL Injection via TenantContext Bypass

**Score: 6 (High)**
**Probability**: Medium (2) - Requires attacker to bypass multiple layers
**Impact**: High (3) - Complete tenant isolation failure

**Description**: If an attacker can manipulate the `TenantContext` or inject SQL that bypasses the interceptor, they could:
- Set a different tenant ID before queries execute
- Clear the session variable mid-transaction
- Inject SQL that reads `current_setting()` and uses it maliciously

This is mitigated by Layers 1 and 2, but must be tested to ensure defense-in-depth works.

**Affected Components**:
- TenantContext ThreadLocal storage
- Database interceptor
- All SQL query paths

**Mitigation**:
- **Preventive Actions**:
  - Ensure TenantContext is immutable within a request
  - Use parameterized queries exclusively (jOOQ, Axon do this)
  - Validate session variable matches TenantContext before committing
  - Add integrity check: compare tenant_id in written rows to session variable
- **Testing Requirements**:
  - Simulate SQL injection attempts
  - Test with manipulated ThreadLocal (if possible)
  - Verify jOOQ query generation doesn't allow injection
  - Test with malformed tenant IDs

**Residual Risk**: Low - Multiple layers provide defense
**Owner**: Dev Team + Security Review
**Timeline**: Before Story 4.3 completion

### 6. [DATA-001]: Event Store Data Corruption During Migration

**Score: 6 (High)**
**Probability**: Medium (2) - Migration scripts carry inherent risk
**Impact**: High (3) - Event sourcing broken, data loss

**Description**: Enabling RLS on existing tables with data requires careful migration:
- If RLS is enabled before policies are created, all queries fail
- If policies are incorrect, legitimate data becomes inaccessible
- If backfill of `tenant_id` is incorrect, events are orphaned
- If migration is not idempotent, re-runs could corrupt data

**Affected Components**:
- Flyway migration scripts
- `domain_event_entry` table
- `widget_projection` table
- Existing test data

**Mitigation**:
- **Preventive Actions**:
  - Create idempotent migration scripts with rollback support
  - Test migrations on copy of production-like data
  - Enable RLS in permissive mode first, then switch to restrictive
  - Validate all existing rows have valid `tenant_id` before enabling RLS
  - Add migration script that validates policy correctness
- **Testing Requirements**:
  - Test migration on empty database
  - Test migration on database with existing multi-tenant data
  - Test migration rollback and re-run
  - Verify no data loss after migration
  - Test with invalid/missing tenant_id values

**Residual Risk**: Low with comprehensive testing
**Owner**: Dev Team
**Timeline**: Before Story 4.3 completion
