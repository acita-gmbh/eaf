## Detailed Risk Register

| Risk ID | Detection Method | Mitigation Strategy | Residual Risk |
|---------|------------------|---------------------|---------------|
| OPS-001 | Integration jobs fail on `ubuntu-latest` without Docker socket access | Validate runner capability early using `docker info`; enable `services: docker`, or provision self-hosted runners; document fallback to `runs-on: ubuntu-latest` with `actions/setup-java` plus Testcontainers reuse flags (`TESTCONTAINERS_HOST_OVERRIDE`) | Medium (4) once mitigation verified |
| OPS-002 | Merge succeeds without CI enforcement after workflow merge | Repository administrators update branch protection to require `quality-gates`, `test`, `mutation-testing`, and `security-scan`; capture evidence in docs/architecture/development-workflow.md change log | Low (2) after enforcement documented |
| TECH-001 | Flaky failures or timeouts during `integrationTest` in matrix | Enable Testcontainers reuse, serialize matrix (e.g., `fail-fast: false`, `max-parallel: 1`), add health-check retries; monitor failure rate across OS targets | Medium (4) until stability trend validated |
| PERF-001 | Workflow duration metrics from GitHub Actions, developer pain reports | Use Gradle build cache, dependency caching, optional `--scan`; evaluate splitting heavy tasks or scheduling nightly mutation testing if runtime >15 min | Medium (4) pending runtime data |
| SEC-001 | Secrets scan, workflow review | Restrict new secrets use; prefer GitHub OIDC with short-lived tokens; document secret storage procedures in architecture docs | Low (2) maintained |

## Risk-Based Testing Strategy

### Priority 1 – High Risks (OPS-001, OPS-002)
- Run a dry-run of the new workflow on both `ubuntu-latest` and `macos-latest` using `gh workflow run` or `act` before merge.
- Confirm Testcontainers services start/stop successfully under GitHub-hosted runners; capture logs.
- Verify branch protection rules are updated and screenshot evidence stored alongside architecture docs.

### Priority 2 – Medium Risks (TECH-001, PERF-001)
- Execute repeated pipeline runs to observe flake rate; enable Testcontainers reusable mode and Gradle build cache to reduce runtime.
- Track job duration metrics; set alert if any job exceeds 15 minutes or matrix build exceeds 30 minutes.
- Validate pipeline behavior under concurrent PR load.

### Priority 3 – Low Risk (SEC-001)
- Schedule periodic secret reviews and ensure no plain-text credentials are introduced in future workflow updates.

## Monitoring & Follow-Up

- Add pipeline runtime and failure-rate dashboards to monitoring plan once workflow is live.
- Reassess risk after first week of CI operations or upon significant workflow changes.
- Document completion of manual branch protection updates in story change log or operational playbook.

## Residual Risk Considerations

- No risks scored ≥9 after mitigation planning; story remains within acceptable risk tolerance once actions above are executed.
- Residual risk primarily operational; ensure ops owner signs off after confirming branch protection and runner compatibility.
