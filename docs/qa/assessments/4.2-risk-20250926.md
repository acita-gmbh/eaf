# Risk Profile Assessment: Story 4.2 - Layer 2 Service Tenant Validation

**Story**: 4.2 - Implement Layer 2 (Service Layer): Tenant Boundary Validation
**Date**: 2025-09-26
**QA Reviewer**: Quinn (Test Architect)
**Assessment Type**: Pre-Implementation Risk Analysis

## Executive Summary

Story 4.2 implements critical tenant isolation at the service layer (Layer 2 of defense-in-depth). This story presents **HIGH SECURITY RISK** due to tenant isolation vulnerabilities if implemented incorrectly. The technical complexity is **MODERATE** with Axon Framework integration challenges. Overall risk score: **7/10** (High Priority).

**Key Risk Factors**:
- Security-critical feature: Tenant isolation bypass could expose cross-tenant data
- Complex integration: Axon aggregate lifecycle + ThreadLocal context propagation
- Transaction safety: Rollback guarantees required for security exceptions
- Testing complexity: Multi-tenant attack scenarios require sophisticated test setup

## Detailed Risk Analysis

### 1. Security Risks (CRITICAL)

#### 1.1 Tenant Isolation Bypass via TenantContext Manipulation
**Risk Score**: 4.0/10 (P0)
**Probability**: Medium (40%) - Developer might trust command.tenantId without TenantContext validation
**Impact**: Critical (10) - Complete tenant isolation breach, cross-tenant data exposure

**Attack Scenario**:
```kotlin
// VULNERABLE CODE (Example of what NOT to do)
@CommandHandler
constructor(command: CreateWidgetCommand) {
    // ❌ Only validates command.tenantId exists, NOT against TenantContext
    require(command.tenantId.isNotBlank()) { "Tenant ID required" }

    apply(WidgetCreatedEvent(
        widgetId = command.widgetId,
        tenantId = command.tenantId  // ❌ Attacker-controlled value
    ))
}
```

**Attack Vector**: Malicious user crafts JWT with valid signature but incorrect tenant claim, bypasses Layer 1, sends command with different tenantId.

**Mitigation Requirements**:
1. MUST compare `command.tenantId == TenantContext.getCurrentTenantId()`
2. MUST throw SecurityException if mismatch detected
3. MUST validate BOTH command tenant AND aggregate tenant (for update commands)
4. Integration test MUST verify cross-tenant command rejection

**Test Coverage Required**:
- Unit test: CreateWidgetCommand with mismatched tenant → SecurityException
- Unit test: UpdateWidgetCommand with command tenant mismatch → Either.left
- Integration test: Tenant A attempts to update Tenant B aggregate → 403 Forbidden

#### 1.2 Missing TenantContext Check in Command Handler
**Risk Score**: 6.0/10 (P0)
**Probability**: High (60%) - Current Widget.kt shows partial validation only (lines 65-69)
**Impact**: Critical (10) - Layer 2 becomes ineffective, defense-in-depth compromised

**Current Implementation Gap** (from story file):
```kotlin
// Current Widget.kt (lines 65-69) - PARTIAL VALIDATION ONLY
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> {
    // ⚠️ Has validation but doesn't use TenantContext
    // Story 4.2 must add TenantContext validation
}
```

**Required Implementation**:
```kotlin
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId() // Fail-closed

    require(command.tenantId == currentTenant) {
        "Tenant isolation violation: command=${command.tenantId}, context=$currentTenant"
    }

    apply(WidgetCreatedEvent(
        widgetId = command.widgetId,
        tenantId = command.tenantId  // Already validated
    ))
}
```

**Validation Checklist**:
- [ ] CreateWidgetCommand handler validates TenantContext
- [ ] UpdateWidgetCommand handler validates command.tenantId
- [ ] UpdateWidgetCommand handler validates this.tenantId (aggregate state)
- [ ] All handlers use fail-closed TenantContext.getCurrentTenantId()
- [ ] SecurityException thrown on any mismatch

#### 1.3 Race Condition in ThreadLocal Access
**Risk Score**: 1.6/10 (P1)
**Probability**: Low (20%) - ThreadLocal is thread-safe by design
**Impact**: High (8) - Wrong tenant context could leak to concurrent request

**Scenario**: Multiple concurrent commands on different aggregates within same HTTP request might share ThreadLocal state incorrectly if Axon uses thread pools.

**Mitigation**:
- Story 4.1 TenantContext uses stack-based ThreadLocal with WeakReference
- TenantContextFilter guarantees cleanup in finally block
- Integration test MUST verify concurrent request isolation

**Test Requirements**:
- Integration test: Concurrent commands for different tenants verify isolation
- Unit test: TenantContext cleanup verified in finally block

#### 1.4 Transaction Not Rolled Back on Security Failure
**Risk Score**: 2.7/10 (P0)
**Probability**: Medium (30%) - Axon defaults to rollback, but custom error handling might break
**Impact**: High (9) - Partial state persistence despite security violation

**Axon Framework Guarantee**: CommandHandler exceptions automatically rollback event store transaction.

**Risk Factor**: If using Either<Error, Result> pattern incorrectly:
```kotlin
// ❌ WRONG - Either.left doesn't throw exception
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> {
    val currentTenant = TenantContext.getCurrentTenantId()

    if (this.tenantId != currentTenant) {
        return Either.left(WidgetError.TenantIsolationViolation(...))
        // ⚠️ Transaction NOT rolled back, aggregate state persists
    }

    apply(WidgetUpdatedEvent(...))
}
```

**Required Pattern**:
```kotlin
// ✅ CORRECT - Use ensure() or throw exception
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = TenantContext.getCurrentTenantId()

    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(
            requestedTenant = command.tenantId,
            actualTenant = currentTenant
        )
    }

    apply(WidgetUpdatedEvent(...))
}
```

**Test Requirements**:
- Integration test: Verify aggregate state unchanged after failed tenant validation
- Integration test: Query aggregate after security exception, verify original state

### 2. Technical Risks (MODERATE)

#### 2.1 Axon Aggregate Constructor Cannot Use Either Return Type
**Risk Score**: 4.2/10 (P1)
**Probability**: High (70%) - Axon framework limitation documented
**Impact**: Medium (6) - Requires exception-based error handling instead of Either

**Constraint**: Axon `@CommandHandler` constructors MUST throw exceptions (cannot return Either).

**Impact on Story 4.2**:
- CreateWidgetCommand MUST use `require()` or `throw SecurityException`
- UpdateWidgetCommand CAN use Either<WidgetError, Unit> pattern
- Two different error handling patterns in same aggregate

**Mitigation**:
```kotlin
// CreateWidgetCommand - Exception-based (Axon constraint)
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId()
    require(command.tenantId == currentTenant) {
        "Tenant isolation violation"
    }
    apply(WidgetCreatedEvent(...))
}

// UpdateWidgetCommand - Either-based (preferred domain pattern)
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = TenantContext.getCurrentTenantId()
    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(...)
    }
    apply(WidgetUpdatedEvent(...))
}
```

**Documentation Requirements**:
- Dev Notes MUST explain why CreateWidgetCommand uses exceptions
- Comment in Widget.kt explaining Axon constructor constraint
- Test both patterns validate identical security guarantees

#### 2.2 TenantContext Static Access Breaks Unit Test Isolation
**Risk Score**: 2.5/10 (P2)
**Probability**: Medium (50%) - Static access pattern can cause test pollution
**Impact**: Medium (5) - Test failures due to shared state

**Challenge**: Story 4.2 uses static access pattern `TenantContext.getCurrentTenantId()` because aggregates cannot be Spring-managed.

**Test Isolation Risks**:
- Unit tests must explicitly set/clear TenantContext before/after each test
- Forgotten cleanup causes test pollution between test cases
- Parallel test execution might share ThreadLocal state

**Mitigation Strategy** (from story file):
```kotlin
class WidgetTest : BehaviorSpec({
    val meterRegistry = SimpleMeterRegistry()
    val tenantContext = TenantContext(meterRegistry)

    Given("a CreateWidgetCommand with matching tenant") {
        When("tenant context is set to tenant-a") {
            tenantContext.setCurrentTenantId("tenant-a")

            Then("widget creation should succeed") {
                val widget = Widget(command, tenantContext)
                widget.tenantId shouldBe "tenant-a"
            }

            // ✅ CRITICAL: Cleanup in Then block
            tenantContext.clearCurrentTenant()
        }
    }
})
```

**Test Requirements**:
- All unit tests MUST call `tenantContext.clearCurrentTenant()` in Then block
- Consider Kotest `beforeTest` and `afterTest` for automatic cleanup
- Add test: Verify TenantContext cleared between test cases

#### 2.3 Event Sourcing Replay Doesn't Validate Tenant Context
**Risk Score**: 2.1/10 (P1)
**Probability**: Low (30%) - Event sourcing replays from trusted event store
**Impact**: High (7) - Aggregate reconstruction with wrong tenant context

**Scenario**: When Axon replays events to reconstruct aggregate state, `@EventSourcingHandler` methods don't have access to TenantContext (no HTTP request context).

**Example**:
```kotlin
@EventSourcingHandler
fun on(event: WidgetCreatedEvent) {
    this.widgetId = event.widgetId
    this.tenantId = event.tenantId  // ✅ Stored in event, no context needed
}
```

**Risk Assessment**: LOW because:
1. Event store is trusted (already persisted, validated at write time)
2. TenantId stored in event payload itself
3. Replay is read-only operation

**Validation**:
- Integration test: Verify aggregate reconstructed correctly after event replay
- Unit test: Verify EventSourcingHandler applies tenantId from event

#### 2.4 UpdateWidgetCommand Partial Validation Incomplete
**Risk Score**: 5.6/10 (P1)
**Probability**: High (80%) - Story explicitly states "PARTIAL validation" exists
**Impact**: High (7) - Incomplete defense-in-depth at Layer 2

**From Story File** (lines 133-137):
> Current Widget Implementation Gaps:
> - ⚠️ UpdateWidgetCommand handler has PARTIAL validation (lines 65-69) but doesn't use TenantContext

**Required Enhancement**:
```kotlin
// Current (PARTIAL)
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> {
    // Some validation exists, but no TenantContext check
}

// Story 4.2 Required (COMPLETE)
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = TenantContext.getCurrentTenantId()

    // Validate 1: Command tenant matches context
    ensure(command.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(
            requestedTenant = command.tenantId,
            actualTenant = currentTenant
        )
    }

    // Validate 2: Aggregate tenant matches context
    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(
            requestedTenant = this.tenantId,
            actualTenant = currentTenant
        )
    }

    apply(WidgetUpdatedEvent(...))
}
```

**Test Coverage Required**:
- Unit test: UpdateWidgetCommand with command.tenantId mismatch
- Unit test: UpdateWidgetCommand with this.tenantId (aggregate) mismatch
- Integration test: Cross-tenant update attack scenario

### 3. Testing Risks (MODERATE)

#### 3.1 Integration Test Doesn't Verify Transaction Rollback
**Risk Score**: 4.0/10 (P1)
**Probability**: Medium (50%) - Easy to test exception, harder to test rollback
**Impact**: High (8) - False confidence in security guarantees

**Challenge**: Integration test must verify:
1. Command rejected with SecurityException
2. Aggregate state unchanged (transaction rolled back)
3. No events persisted to event store

**Required Test Pattern** (from story file):
```kotlin
Given("a widget created for Tenant B") {
    val widgetId = "widget-b-123"

    When("Tenant A tries to update Tenant B's widget") {
        // Setup: Create as Tenant B
        tenantContext.setCurrentTenantId("tenant-b")
        commandGateway.sendAndWait<String>(
            CreateWidgetCommand(
                widgetId = widgetId,
                tenantId = "tenant-b",
                name = "Original Name"
            )
        )
        tenantContext.clearCurrentTenant()

        // Attack: Update as Tenant A
        tenantContext.setCurrentTenantId("tenant-a")

        Then("command should be rejected") {
            shouldThrow<Exception> {
                commandGateway.sendAndWait<Unit>(
                    UpdateWidgetCommand(
                        widgetId = widgetId,
                        tenantId = "tenant-a",
                        name = "Hacked Widget"
                    )
                )
            }.message shouldContain "Tenant isolation violation"
        }

        Then("aggregate state should be unchanged") {
            tenantContext.setCurrentTenantId("tenant-b")
            val widget = queryGateway.query(
                FindWidgetByIdQuery(widgetId),
                Widget::class.java
            ).get()

            widget.name shouldBe "Original Name"  // NOT "Hacked Widget"
        }

        tenantContext.clearCurrentTenant()
    }
}
```

**Validation Checklist**:
- [ ] Test verifies SecurityException thrown
- [ ] Test queries aggregate state after failure
- [ ] Test confirms original state preserved
- [ ] Test verifies no events persisted (event store query)

#### 3.2 Cross-Tenant Attack Scenario Not Comprehensive
**Risk Score**: 3.6/10 (P1)
**Probability**: Medium (40%) - Easy to test happy path, harder to test all attack vectors
**Impact**: Critical (9) - Undetected security vulnerabilities in production

**Required Attack Scenarios**:
1. **Cross-tenant create**: Tenant A creates widget with Tenant B ID in command
2. **Cross-tenant update**: Tenant A updates Tenant B's existing widget
3. **Cross-tenant delete**: Tenant A deletes Tenant B's widget
4. **Missing TenantContext**: Command sent with no TenantContext set
5. **Null tenant in command**: Command with null/empty tenantId field
6. **Context switching**: Multiple commands with different tenants in same test

**Minimum Test Coverage**:
```kotlin
// Scenario 1: Cross-tenant create
Given("TenantContext set to tenant-a") {
    When("CreateWidgetCommand has tenantId=tenant-b") {
        Then("command should be rejected")
    }
}

// Scenario 2: Cross-tenant update
Given("Widget belongs to tenant-b") {
    When("Tenant A tries to update it") {
        Then("command should be rejected")
        Then("aggregate state unchanged")
    }
}

// Scenario 3: Missing TenantContext
Given("TenantContext not set") {
    When("Any command executed") {
        Then("throw IllegalStateException")
    }
}

// Scenario 4: Context switching
Given("Multiple tenants in same test") {
    When("Commands alternate between tenant-a and tenant-b") {
        Then("Each command isolated correctly")
        Then("No context leakage between commands")
    }
}
```

#### 3.3 Unit Tests Use Mocked TenantContext (Anti-Pattern)
**Risk Score**: 3.6/10 (P2)
**Probability**: High (60%) - Developers might mock for convenience
**Impact**: Medium (6) - False test coverage, real bugs undetected

**Anti-Pattern** (PROHIBITED):
```kotlin
// ❌ FORBIDDEN - Mocking TenantContext
val mockContext = mockk<TenantContext>()
every { mockContext.getCurrentTenantId() } returns "tenant-a"

val widget = Widget(command, mockContext)
```

**Required Pattern** (from test-strategy-and-standards-revision-3.md):
```kotlin
// ✅ CORRECT - Real TenantContext with nullable dependencies
val meterRegistry = SimpleMeterRegistry()
val tenantContext = TenantContext(meterRegistry)

tenantContext.setCurrentTenantId("tenant-a")
val widget = Widget(command, tenantContext)
tenantContext.clearCurrentTenant()
```

**Validation**:
- Code review: Search for `mockk<TenantContext>` in test files (should be zero)
- Konsist rule: Verify no TenantContext mocking in unit tests
- Test coverage: All unit tests use real TenantContext instance

## Risk Mitigation Strategies

### Critical Mitigations (P0)

1. **Mandatory TenantContext Validation**
   - EVERY command handler MUST validate TenantContext
   - Use fail-closed pattern: `TenantContext.getCurrentTenantId()` throws if missing
   - Update handlers MUST validate both command AND aggregate tenant

2. **Transaction Rollback Verification**
   - Integration tests MUST verify aggregate state unchanged after security failure
   - Use `either {}` builder with `ensure()` for proper rollback semantics

3. **Comprehensive Security Testing**
   - Implement ALL cross-tenant attack scenarios listed above
   - Test transaction rollback guarantees
   - Test missing/null TenantContext scenarios

### High Priority Mitigations (P1)

4. **Test Isolation Strategy**
   - Always cleanup TenantContext in test Finally blocks
   - Consider Kotest `beforeTest`/`afterTest` hooks
   - Add test: Verify no context leakage between test cases

5. **Error Handling Documentation**
   - Document why CreateWidgetCommand uses exceptions (Axon constraint)
   - Document why UpdateWidgetCommand uses Either pattern (domain preference)
   - Add inline comments explaining tenant validation

6. **Event Sourcing Validation**
   - Add integration test: Aggregate reconstruction after event replay
   - Verify tenantId correctly applied from event payload

### Medium Priority Mitigations (P2)

7. **Performance Monitoring**
   - Add metrics: tenant validation overhead per command
   - Target: <1ms overhead for TenantContext access
   - Alert if p95 latency >200ms

8. **Security Logging**
   - Log tenant isolation violations at WARN level
   - Include: aggregateId, commandTenant, contextTenant
   - DO NOT log sensitive data (JWT tokens)
