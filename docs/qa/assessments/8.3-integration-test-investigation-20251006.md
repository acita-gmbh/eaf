## Root Cause Analysis

### Issue 8.3-BUILD-001: OpenTelemetry ComponentLoader ClassNotFoundException

**Error**: `ClassNotFoundException: io.opentelemetry.common.ComponentLoader`

**Root Cause**: Version conflict between:
- OpenTelemetry SDK 1.54.1 (specified in versions.toml)
- OpenTelemetry SDK 1.49.0 (forced by Spring Boot BOM transitive dependencies)
- OpenTelemetry Instrumentation 2.20.1 (expects ComponentLoader in io.opentelemetry.common)

**Discovery Process**:
1. **Web Search**: Found GitHub issue #11161 documenting ComponentLoader availability problem
2. **Dependency Tree Analysis**: Discovered 1.54.1 -> 1.49.0 downgrades
3. **Ollama Consultation**: Recommended resolutionStrategy.force() approach
4. **Context7**: Spring Boot patterns for excluding auto-configuration

**ComponentLoader Location**:
- In SDK 1.49.0: `io.opentelemetry.sdk.autoconfigure.internal.ComponentLoader` (internal)
- Expected by Instrumentation 2.20.1: `io.opentelemetry.common.ComponentLoader` (moved)
- Result: Instrumentation tries to load class that doesn't exist in older SDK

## Successful Solutions

### ✅ OpenTelemetry Bypass (Solved)

**Final Working Configuration**:
```kotlin
@Configuration
@Import(
    DataSourceAutoConfiguration::class,
    JooqAutoConfiguration::class,
)
@ComponentScan(basePackageClasses = [JooqWidgetProjectionRepository::class])
open class WidgetProjectionTestConfig
```

**Why This Works**:
- Explicitly imports ONLY required auto-configurations
- Avoids classpath scanning that triggers OpenTelemetry loading
- No exclude rules needed (nothing to exclude if not scanned)
- Spring Boot context loads successfully

**Result**: Tests run and Spring Boot context initializes (5s duration vs previous instant crash)

## Research Insights

### Web Search Findings

1. **OpenTelemetry GitHub Issue #11161**:
   - ComponentLoader interface not available in certain version combinations
   - Affects Spring Boot Starter 2.2.0-alpha + older SDKs
   - Solution: Use opentelemetry-bom-alpha instead of opentelemetry-bom

2. **Spring Boot 3.3.x Dependency Issue #43200**:
   - Micrometer Tracing and OpenTelemetry dependencies don't converge
   - Spring Boot BOM forces older OpenTelemetry versions
   - Affects projects mixing actuator + OpenTelemetry

### Ollama (qwen3-coder) Recommendations

**Suggested Approach**:
```kotlin
configurations.all {
    resolutionStrategy {
        force("io.opentelemetry:opentelemetry-api:1.54.1")
        force("io.opentelemetry:opentelemetry-sdk:1.54.1")
        // ... all OpenTelemetry modules
    }
}
```

**Why It Didn't Work**: Spring Boot BOM constraints take precedence over resolutionStrategy.force() in Gradle

### Context7 Spring Boot Patterns

**Test Configuration Best Practices**:
- Use explicit `@Import` for focused integration tests
- Avoid `@SpringBootApplication` in tests (pulls too much)
- Use `@EnableAutoConfiguration` with targeted excludes
- Prefer minimal test configurations over full application

## Lessons Learned

### jOOQ Integration Best Practices

1. **Source Set Configuration**: ALL source sets need jOOQ output directory
2. **Import Pattern**: Use `jooq.tables.references.WIDGET_PROJECTION`
3. **Security**: Always add tenant WHERE clause to UPDATE statements
4. **Case Sensitivity**: Ensure DDL case matches jOOQ generation expectations

### Spring Boot Test Configuration

1. **Minimal is Better**: Explicit @Import > @SpringBootApplication for focused tests
2. **OpenTelemetry**: Version conflicts common with Spring Boot 3.5.x
3. **Testcontainers**: initScript is best for schema initialization
4. **Lifecycle**: beforeSpec hooks challenging with Spring Boot context timing

### Dependency Management

1. **BOM Precedence**: Spring Boot BOM overrides resolutionStrategy.force()
2. **Version Downgrades**: Check dependency tree for unexpected downgrades
3. **Internal Classes**: Be cautious of dependencies using io.*.internal packages
4. **Instrumentation != SDK**: OpenTelemetry instrumentation and SDK have separate versioning
