## Requirement Mappings

### AC1: Database Interceptor Sets Session Variable Per Transaction

**Requirement**: A database interceptor sets the current PostgreSQL session variable (e.g., `SET app.current_tenant = ?`) using the ID from the TenantContext at the start of every transaction.

**Coverage**: FULL

#### Given-When-Then Mappings

**Integration Test**: `TenantSessionInterceptorIntegrationTest::sets session variable on transaction start` (4.3-INT-001)
- **Given**: TenantContext is populated with valid tenant ID
- **When**: A new transaction begins
- **Then**: Session variable `app.current_tenant` is set to the tenant ID from context

**Integration Test**: `TenantSessionInterceptorIntegrationTest::session variable matches context value` (4.3-INT-002)
- **Given**: TenantContext contains specific tenant UUID
- **When**: Interceptor executes at transaction start
- **Then**: `current_setting('app.current_tenant')` returns the exact UUID from context

**Integration Test**: `TenantSessionInterceptorIntegrationTest::throws exception if context missing` (4.3-INT-003)
- **Given**: TenantContext is empty (no tenant set)
- **When**: Transaction attempts to start
- **Then**: Interceptor throws exception (fail-closed behavior) and transaction is not created

**Integration Test**: `TenantSessionInterceptorIntegrationTest::clears session variable on connection release` (4.3-INT-004)
- **Given**: Connection has session variable set from previous transaction
- **When**: Connection is returned to the pool
- **Then**: Session variable is explicitly reset/cleared

**Integration Test**: `TenantSessionInterceptorIntegrationTest::multiple transactions update session variable correctly` (4.3-INT-005)
- **Given**: Same connection is reused for multiple transactions with different tenants
- **When**: Each transaction begins with different TenantContext
- **Then**: Session variable is updated to match current context each time

**Integration Test**: `TenantSessionInterceptorIntegrationTest::concurrent transactions maintain isolated session variables` (4.3-INT-006)
- **Given**: Multiple threads executing transactions concurrently
- **When**: Each thread has different TenantContext value
- **Then**: Each transaction sees only its own tenant's session variable (thread-safe)

**Unit Test**: `SessionVariableSqlGeneratorTest::generates correct SET LOCAL statement` (4.3-UNIT-001)
- **Given**: A tenant UUID value
- **When**: SQL generation method is called
- **Then**: Returns `SET LOCAL app.current_tenant = '<uuid>'` with correct syntax

### AC1 (Subtask 1.3): Session Variable Cleared on Connection Release

**Requirement**: Ensure the interceptor clears or resets the session variable on connection release to avoid tenant leakage across pooled connections.

**Coverage**: FULL

#### Given-When-Then Mappings

**Integration Test**: `ConnectionPoolingIntegrationTest::session variable cleared on connection return` (4.3-INT-004)
- **Given**: Connection with session variable set for Tenant A

**Integration Test**: `ConnectionPoolingIntegrationTest::connection reuse with different tenant` (4.3-INT-005)

**Integration Test**: `ConnectionPoolingIntegrationTest::minimal pool stress test` (4.3-INT-047)

### AC3: RLS Policies Enforce Tenant Isolation on All CRUD Operations

**Requirement**: The RLS policies strictly enforce that all SELECT, INSERT, UPDATE, and DELETE operations can only apply to rows where the `tenant_id` column matches the `app.current_tenant` session variable.

**Coverage**: FULL

#### Given-When-Then Mappings

**Integration Test**: `RlsEnforcementIntegrationTest::SELECT filtered by tenant_id` (4.3-INT-017)
- **Given**: Database with data from Tenant A, B, and C
- **When**: Session variable set to Tenant A and SELECT query executed
- **Then**: Only Tenant A's rows are returned (Tenant B and C rows hidden)

**Integration Test**: `RlsEnforcementIntegrationTest::INSERT validates tenant_id matches session` (4.3-INT-018)
- **Given**: Session variable set to Tenant A
- **When**: Attempting to INSERT row with `tenant_id = Tenant B`
- **Then**: INSERT fails or row is rejected (cannot write to other tenant)

**Integration Test**: `RlsEnforcementIntegrationTest::UPDATE only affects current tenant` (4.3-INT-019)
- **Given**: Session variable set to Tenant A, database has rows for Tenant A and B
- **When**: UPDATE statement executed without explicit tenant filter
- **Then**: Only Tenant A's rows are updated (Tenant B rows untouched)

**Integration Test**: `RlsEnforcementIntegrationTest::DELETE only affects current tenant` (4.3-INT-020)
- **Given**: Session variable set to Tenant A, database has rows for Tenant A and B
- **When**: DELETE statement executed without explicit tenant filter
- **Then**: Only Tenant A's rows are deleted (Tenant B rows preserved)

**Integration Test**: `RlsEnforcementIntegrationTest::query without session variable returns zero rows` (4.3-INT-021)
- **Given**: Database with data from multiple tenants
- **When**: Query executed without session variable set
- **Then**: Zero rows returned (fail-closed behavior)

**Integration Test**: `RlsEnforcementIntegrationTest::query with wrong tenant_id returns zero rows` (4.3-INT-022)
- **Given**: User authenticated for Tenant A, session variable set to Tenant A
- **When**: Attempting to query by manipulating `tenant_id` filter to Tenant B
- **Then**: Zero rows returned (RLS overrides application filter)

**Integration Test**: `RlsEnforcementIntegrationTest::RLS applies to event store and projections` (4.3-INT-023)
- **Given**: RLS policies enabled on both tables
- **When**: Queries executed against `domain_event_entry` and `widget_projection`
- **Then**: Both tables enforce tenant isolation correctly

**Integration Test**: `RlsEnforcementIntegrationTest::RLS with complex WHERE clauses` (4.3-INT-024)
- **Given**: Session variable set to Tenant A
- **When**: Complex query with multiple AND/OR conditions
- **Then**: RLS filter combines correctly with application filters, only Tenant A rows returned

**Integration Test**: `RlsEnforcementIntegrationTest::RLS with JOINs across tables` (4.3-INT-025)
- **Given**: JOIN query between event store and projection tables
- **When**: Session variable set to Tenant A
- **Then**: JOIN result only includes Tenant A rows from both tables

### AC4: Integration Tests Verify RLS Blocks Malicious SQL

**Requirement**: Integration tests verify that even a direct SQL query (simulating an SQL injection vulnerability) attempting to read data without the session variable correctly set returns zero rows.

**Coverage**: FULL

#### Given-When-Then Mappings

**Integration Test**: `SqlInjectionSimulationTest::raw DSLContext query without interceptor` (4.3-INT-027)
- **Given**: Direct jOOQ `DSLContext` bypassing normal application flow
- **When**: Query executed without interceptor setting session variable
- **Then**: Zero rows returned due to RLS policy enforcement

**Integration Test**: `SqlInjectionSimulationTest::direct JDBC query without session variable` (4.3-INT-028)
- **Given**: Raw JDBC connection bypassing all application layers
- **When**: SELECT query executed directly
- **Then**: Zero rows returned (RLS is database-level enforcement)

**Integration Test**: `SqlInjectionSimulationTest::malformed tenant_id in session variable` (4.3-INT-029)
- **Given**: Session variable set to invalid UUID format
- **When**: Query executed with malformed tenant_id
- **Then**: Database error or zero rows returned (fail safely)

**Integration Test**: `SqlInjectionSimulationTest::session variable set to null` (4.3-INT-030)
- **Given**: Session variable explicitly set to null
- **When**: Query executed
- **Then**: Zero rows returned (null tenant = no access)

**Integration Test**: `SqlInjectionSimulationTest::session variable cleared mid-transaction` (4.3-INT-031)
- **Given**: Transaction started with session variable set
- **When**: Session variable cleared during transaction, then query executed
- **Then**: Query fails or returns zero rows (transaction scope validation)

**Integration Test**: `SqlInjectionSimulationTest::connection pool stress test` (4.3-INT-032)
- **Given**: Minimal connection pool (2 connections) with 5 tenants rapidly switching
- **When**: Attempting to exploit connection reuse for cross-tenant access
- **Then**: All queries correctly isolated, no tenant leakage observed
