## High-Priority Risks (Score 6)

### TECH-001: Architectural Dependency on Story 7.4a

**Score: 6 (High)** — Probability: Medium (2) × Impact: High (3)

**Description**: Story 7.4b generates UI resources that import from `@axians/eaf-admin-shell` (Story 7.4a framework). If 7.4a is incomplete, unstable, or has breaking changes, generated code will be broken or non-functional.

**Affected Components**:
- `ReactAdminResourceGenerator.kt` (generates imports: `import { EmptyState, LoadingSkeleton } from '@axians/eaf-admin-shell'`)
- `List.tsx.mustache`, `Create.tsx.mustache`, `Edit.tsx.mustache`, `Show.tsx.mustache` templates (all reference framework components)
- `ResourceExport.ts.mustache` (exports resource registration object for AdminShell)

**Probability: Medium (2)**:
- Story 7.4a is in Draft status (not yet implemented)
- 7.4a has 3 high-risk security items (score 6) that could delay completion
- API surface of 7.4a (ResourceConfig, AdminShell props) is well-defined in story
- Breaking changes possible during 7.4a implementation

**Impact: High (3)**:
- **Generated code won't compile**: Missing imports if 7.4a not built
- **Runtime errors**: If 7.4a components have bugs
- **Rework required**: Breaking changes in 7.4a API require template updates and regeneration
- **Blocked testing**: Can't validate generated UI without working framework

**Mitigation Strategies**:

1. **Sequential implementation** (Task 3.6):
   - Validate `framework/admin-shell` exists and is built before generating resources
   - CLI command fails early if framework not available: `AdminShellNotBuilt` error

2. **API contract validation**:
   - Test generated imports against 7.4a's exported API surface
   - Integration test (7.4b-INTEGRATION-001) verifies generated code compiles with framework

3. **Versioning strategy**:
   - Generated code includes comment: `// Generated for @axians/eaf-admin-shell v0.1.0`
   - CLI warns if framework version mismatch detected

4. **Documentation**:
   - README documents dependency: "Requires framework/admin-shell to be built first"
   - Error messages guide users: "Run 'npm run build:admin-shell' before generating UI resources"

**Testing Requirements**:
- **7.4b-UNIT-006**: Validate framework/admin-shell existence before generation
- **7.4b-INTEGRATION-001**: Generate resource, verify imports resolve, TypeScript compiles

**Residual Risk**: **Low** — With sequential implementation and early validation, risk is mitigated. Framework API is stable by design (publish-ready).

**Owner**: Dev (CLI Lead + Frontend Lead coordination)
**Timeline**: Ensure Story 7.4a core implementation complete before starting 7.4b

### SEC-001: Path Traversal and Injection Attacks

**Score: 6 (High)** — Probability: Medium (2) × Impact: High (3)

**Description**: CLI accepts user input for resource names and paths. Malicious input could exploit path traversal (`../../etc/passwd`), injection (`<script>alert(1)</script>`), or invalid names (`lowercase-name`) to write files outside intended directories or inject malicious code into generated files.

**Affected Components**:
- `UiResourceCommand.kt` (accepts resourceName parameter)
- `ReactAdminResourceGenerator.kt` (constructs file paths, renders templates)
- All Mustache templates (if user input not sanitized before rendering)

**Probability: Medium (2)**:
- CLI is developer-facing tool (internal use, not public API)
- Story 7.3 established 3-layer defense pattern (reduces but doesn't eliminate risk)
- Developers may test with malicious inputs (intentional or accidental)
- Automated scripts could generate invalid inputs

**Impact: High (3)**:
- **CWE-22 (Path Traversal)**: Write files outside `products/{module}/ui-module/` (e.g., overwrite `/etc/passwd`, inject into build scripts)
- **CWE-79 (XSS)**: Inject malicious code into generated TypeScript files (executed in developer's IDE or browser)
- **CWE-94 (Code Injection)**: Inject executable code into templates (compromises developer machine)
- **File system corruption**: Overwrite critical project files

**Mitigation Strategies** (from Story 7.3 ADVISORY-001):

**3-Layer Defense** (Task 3.4):

1. **Layer 1: Input Validation**
   - **PascalCase pattern**: `^[A-Z][A-Za-z0-9]*$` (regex validation)
   - Reject: `lowercase-name`, `123Invalid`, `my-resource`, `user@email.com`
   - Whitelist: Only alphanumeric characters, must start with uppercase letter

2. **Layer 2: Path Sanitization**
   - **Prevent traversal**: Remove `../`, `./`, `~`, absolute paths
   - **Normalize separators**: Replace backslashes with forward slashes
   - Use `Paths.get()` for platform-independent path construction

3. **Layer 3: Canonical Path Verification**
   - **Resolve canonical path**: `File.getCanonicalPath()`
   - **Verify within boundary**: `canonicalPath.startsWith(projectRoot.canonicalPath)`
   - **Reject if outside**: Throw `PathTraversalError` if validation fails

**Code Example** (from Story 7.3):
```kotlin
// Layer 1: Input validation
require(resourceName.matches(Regex("^[A-Z][A-Za-z0-9]*$"))) {
    "Resource name must be PascalCase: $resourceName"
}

// Layer 2: Path sanitization
val sanitized = resourceName.replace(Regex("[./\\\\~]"), "")

// Layer 3: Canonical path verification
val targetDir = File("products/$productModule/ui-module/src/resources/${sanitized.lowercase()}")
val canonicalPath = targetDir.canonicalFile
val projectRoot = File(".").canonicalFile
require(canonicalPath.startsWith(projectRoot)) {
    "Path traversal detected: $canonicalPath"
}
```

**Testing Requirements** (Task 7.1, 7.2):
- **7.4b-UNIT-001** (Security-First TDD): Attack scenarios MUST be implemented FIRST
  - Path traversal: `../../etc/passwd`, `../../../root/.ssh/id_rsa`
  - Injection: `<script>alert(1)</script>`, `'; DROP TABLE users; --`
  - Invalid PascalCase: `lowercase-name`, `123Invalid`, `my-resource`

**Residual Risk**: **Low** — 3-layer defense (from Story 7.3) provides comprehensive protection. Developer-facing CLI reduces attack surface compared to public APIs.

**Owner**: Dev (CLI Lead)
**Timeline**: Implement security test FIRST (Task 7.1) before any file generation logic

### TECH-003: Kotlin DTO Parsing Failures

**Score: 4 (Medium)** — Probability: Medium (2) × Impact: Medium (2)

**Description**: Hybrid regex approach (Task 5.2) for parsing Kotlin DTOs may fail on complex types (nested DTOs, custom annotations), resulting in missing fields or incorrect type mappings.

**Affected Components**: `ReactAdminResourceGenerator.kt` (DTO parsing logic)

**Mitigation**:
- **Fallback to manual --fields**: If parsing fails, log clear error with fallback message
- **Simple regex sufficient for 90% of DTOs**: Handles flat DTOs with primitive/standard types (String, Int, BigDecimal, List<T>)
- **Future enhancement**: AST parser (kotlin-compiler-embeddable) for complex types

**Testing**: 7.4b-UNIT-004 (type mapping accuracy), 7.4b-UNIT-005 (DTO parsing success/failure)
