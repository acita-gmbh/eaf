### 1.2 Defense-in-Depth Validation (Critical)

**Requirement**: Layer 2 (Service) must enforce tenant isolation independently of Layer 1 (Request Filter), preventing bypass attacks.

**Assessment**: ✅ **EXCELLENT** (95/100)

**Architecture Validation**:

Layer 1 (Request Filter - Story 4.1):
- TenantContextFilter extracts tenant from JWT
- Populates ThreadLocal TenantContext
- Order: After BearerTokenAuthenticationFilter

Layer 2 (Service - Story 4.2):
- Command handlers validate tenant independently
- NO dependency on HTTP request context
- Operates on TenantContext state only

Layer 3 (Database RLS - Future Story 4.3):
- PostgreSQL Row-Level Security
- Final isolation guarantee

**Independence Verification**:
```kotlin
// Layer 2 operates independently of Layer 1
@CommandHandler
constructor(command: CreateWidgetCommand) {
    // Does NOT access HTTP request, JWT, or SecurityContext
    // Only accesses TenantContext (could be set by ANY caller)
    val currentTenant = TenantContext.getCurrentTenantId()

    require(command.tenantId == currentTenant) {
        "Tenant isolation violation"
    }

    apply(WidgetCreatedEvent(...))
}
```

**Attack Scenario Validation**:

**Scenario**: Attacker bypasses TenantContextFilter (Layer 1)
- **Layer 1 Bypassed**: TenantContext not set
- **Layer 2 Response**: `getCurrentTenantId()` throws IllegalStateException
- **Result**: Request rejected, no data access ✅

**Scenario**: Attacker manipulates JWT with valid signature but wrong tenant
- **Layer 1**: TenantContextFilter sets TenantContext to attacker's tenant
- **Layer 2**: Command handler validates `command.tenantId == TenantContext`
- **Result**: Mismatch detected, SecurityException thrown ✅

**Scenario**: Attacker sends command with fabricated tenantId
- **Layer 1**: TenantContextFilter sets TenantContext from JWT
- **Layer 2**: Command handler validates command.tenantId vs TenantContext
- **Result**: Mismatch detected, SecurityException thrown ✅

**Test Requirements**:
- [ ] Integration test: Bypass TenantContextFilter, send command → rejection
- [ ] Integration test: Valid JWT for Tenant A, command for Tenant B → rejection
- [ ] Integration test: Missing TenantContext, any command → IllegalStateException

**Score Justification**: -5 points because integration test for bypassing Layer 1 is complex to implement. Need to verify Layer 2 operates independently without HTTP context.

### 1.4 Transaction Rollback Guarantees (Critical)

**Requirement**: Security violations must trigger immediate transaction rollback with no partial state persistence.

**Assessment**: ✅ **EXCELLENT** (100/100)

**Axon Framework Guarantees**:
1. CommandHandler exceptions automatically rollback event store transaction
2. No events persisted if handler throws exception
3. Aggregate state not updated if command fails

**Pattern Validation**:

**Constructor Handlers** (Exception-based):
```kotlin
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId()

    require(command.tenantId == currentTenant) {
        "Tenant isolation violation"
    }
    // ↑ Exception thrown → Axon rolls back transaction

    apply(WidgetCreatedEvent(...))
    // ↑ Event never applied, aggregate never created
}
```

**Method Handlers** (Either-based with ensure):
```kotlin
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = TenantContext.getCurrentTenantId()

    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(...)
    }
    // ↑ ensure() throws internally → Axon rolls back transaction

    apply(WidgetUpdatedEvent(...))
    // ↑ Event never applied, aggregate state unchanged
}
```

**CRITICAL: Avoid Anti-Pattern**:
```kotlin
// ❌ WRONG - Either.left doesn't throw exception
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> {
    val currentTenant = TenantContext.getCurrentTenantId()

    if (this.tenantId != currentTenant) {
        return Either.left(WidgetError.TenantIsolationViolation(...))
        // ⚠️ No exception thrown → Transaction NOT rolled back
    }

    apply(WidgetUpdatedEvent(...))
}
```

**Mitigation**: Use Arrow's `either {}` builder with `ensure()`:
- `ensure()` throws exception internally if condition fails
- Axon catches exception and rolls back transaction
- Either<Error, Success> returned to caller

**Test Requirements**:
- [ ] Integration test: Verify aggregate state unchanged after security failure
- [ ] Integration test: Query event store, verify no events persisted
- [ ] Integration test: Attempt second command on same aggregate, verify original state

**Score Justification**: Full marks. Axon's transaction rollback is automatic and reliable. Using `either {}` with `ensure()` preserves this guarantee while providing Either-based API.

## 3. Reliability (Weight: 20%, Score: 80/100)

### 3.1 Error Handling Consistency (Critical)

**Requirement**: Tenant validation errors must be handled consistently across all command handlers.

**Assessment**: ⚠️ **MODERATE** (70/100)

**Challenge**: Story 4.2 requires TWO different error handling patterns:

1. **Constructor Handlers** (CreateWidgetCommand):
   - MUST use exception-based error handling
   - Axon constraint: Constructors cannot return Either

2. **Method Handlers** (UpdateWidgetCommand):
   - SHOULD use Either<WidgetError, Unit> pattern
   - Domain preference: Type-safe error handling

**Consistency Concerns**:

**Pattern 1**: Exception-based (CreateWidgetCommand)
```kotlin
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId()

    require(command.tenantId == currentTenant) {
        "Tenant isolation violation"  // ← String message
    }

    apply(WidgetCreatedEvent(...))
}
```

**Pattern 2**: Either-based (UpdateWidgetCommand)
```kotlin
@CommandHandler
fun handle(command: UpdateWidgetCommand): Either<WidgetError, Unit> = either {
    val currentTenant = TenantContext.getCurrentTenantId()

    ensure(this.tenantId == currentTenant) {
        WidgetError.TenantIsolationViolation(  // ← Structured error
            requestedTenant = this.tenantId,
            actualTenant = currentTenant
        )
    }

    apply(WidgetUpdatedEvent(...))
}
```

**Inconsistency Issues**:
1. CreateWidget throws String-based exception
2. UpdateWidget returns structured WidgetError
3. Callers must handle two different error types
4. Error message format inconsistent

**Mitigation Strategy**:
```kotlin
// Define custom exception matching WidgetError structure
class TenantIsolationException(
    val requestedTenant: String,
    val actualTenant: String
) : SecurityException("Tenant isolation violation: requested=$requestedTenant, actual=$actualTenant")

// Use in constructor handler
@CommandHandler
constructor(command: CreateWidgetCommand) {
    val currentTenant = TenantContext.getCurrentTenantId()

    if (command.tenantId != currentTenant) {
        throw TenantIsolationException(
            requestedTenant = command.tenantId,
            actualTenant = currentTenant
        )
    }

    apply(WidgetCreatedEvent(...))
}
```

**Test Requirements**:
- [ ] Unit test: Verify CreateWidget throws TenantIsolationException
- [ ] Unit test: Verify UpdateWidget returns Either.left(TenantIsolationViolation)
- [ ] Integration test: Verify both errors mapped to 403 Forbidden by global advice

**Score Justification**: -30 points due to dual error handling patterns. Mitigations possible but add complexity.

**Axon Transaction Lifecycle**:
1. CommandGateway receives command
2. Axon starts transaction
3. @CommandHandler invoked (INSIDE transaction)
4. Tenant validation executes (INSIDE transaction)
5. Events applied (INSIDE transaction)
6. Transaction commits (if no exceptions)

**Validation Point**:
```kotlin
@CommandHandler
constructor(command: CreateWidgetCommand) {

        "Tenant isolation violation"

    // ↑ Exception thrown → Transaction rolled back

    apply(WidgetCreatedEvent(...))

}
```

**Guarantee**:
- Tenant validation ALWAYS inside transaction
- Exception ALWAYS triggers rollback
- No events persisted if validation fails

**Test Requirements**:

- [ ] Integration test: Verify aggregate not created after validation failure

**Score Justification**: Full marks. Axon guarantees tenant validation within transaction boundary.

        When("widget is created") {
            val widget = Widget(command, tenantContext)

            Then("widget should have correct tenant") {

            }
        }
        // Cleanup automatic via afterTest hook
