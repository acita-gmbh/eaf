## High Risks (Score 6) - Monitoring Required

### **TECH-001: Delayed Mutation Coverage Feedback**

**Score: 6 (High Risk - Orange)**

**Probability:** High (3)
- **Reasoning:** This WILL happen by design - mutation testing moves from immediate PR feedback to next-day nightly feedback
- **Likelihood:** 100% - This is the intended behavior change

**Impact:** Medium (2)
- **Potential Consequences:**
  - Mutation coverage regressions detected 24 hours later instead of immediately
  - Developers may introduce poorly-tested code that passes PR but fails nightly
  - Team must monitor nightly results instead of relying on PR gates
- **Business Impact:** Minor - Quality still maintained, just detected on different timeline
- **User Impact:** None - Internal development process change

**Affected Components:**
- PR pipeline workflow (`.github/workflows/ci.yml`)
- Nightly pipeline workflow (`.github/workflows/nightly.yml`)
- Developer workflow and expectations
- Quality gate process

**Detection Method:** Post-implementation monitoring of nightly runs and developer feedback

**Mitigation Strategy:**

**Preventive:**
- Developers can still run `./gradlew pitest` locally before PR (documented in Task 7)
- Documentation emphasizes running pitest for risky changes (Task 7, Subtask 1)
- Clear communication about the trade-off (57% faster PR feedback vs 24h mutation detection)

**Detective:**
- Nightly failures create immediate GitHub Issues (AC6)
- Team notified via Slack/email for critical failures
- Dashboard showing nightly health trends (AC7)

**Corrective:**
- Critical modules can keep pitest in PR if needed (configurable approach mentioned)
- Rollback capability: Re-add pitest to PR workflow if 24h delay proves problematic

**Testing Focus:**
- Validate notification system works reliably (AC6 - manual failure trigger test)
- Measure developer satisfaction with new feedback model (Success Metric 4)
- Monitor first week of nightly runs closely (Rollout Strategy Phase 2)

**Residual Risk:** Low - Well-mitigated with clear process and monitoring

**Owner:** Development team + Process steward

**Timeline:** Ongoing monitoring post-implementation

## Detailed Risk Analysis

### TECH-001: Delayed Mutation Coverage Feedback (Score 6)

**Full Description:**
Story 8.7 moves mutation testing from PR pipeline (immediate feedback) to nightly pipeline (next-day feedback). This is an intentional architectural decision to improve developer experience (57% faster PR feedback) at the cost of delayed mutation regression detection.

**Root Cause:** CI resource optimization - mutation testing takes 3-5 minutes and runs 50+ times/day in current PR model.

**Affected Workflows:**
- Developer: Commits code → Waits for PR feedback → Merges → Mutation results available next morning
- Current: Commits code → Waits 6-8 min including mutation → Merges with mutation validation

**Trade-off Analysis:**
- **Benefit:** 4 minutes/commit × 50 commits/day = 200 minutes/day saved team-wide
- **Cost:** Mutation regressions detected within 24 hours vs immediately
- **Judgment:** Benefits outweigh costs for typical development workflow

**Mitigation Effectiveness Assessment:**
- ✅ **Strong:** Documentation and developer education (AC9 - 3 files updated)
- ✅ **Strong:** Immediate nightly failure notifications (AC6)
- ✅ **Adequate:** Local pitest execution option (AC10 - still works)
- ⚠️ **Potential Gap:** No measurement of how often developers actually run pitest locally

**Recommended Monitoring:**
1. Track "pitest run locally before PR" frequency (add to Success Metrics)
2. Measure mutation regression discovery lag (time from commit to detection)
3. Survey developers monthly: "Is 24h mutation feedback acceptable?"
4. Track critical mutation regressions that would have been caught in PR

**Escalation Trigger:** If 3+ critical mutation regressions occur in 1 month that PR pitest would have caught.

- JWT format parsing vulnerabilities (JwtFormat target)
- Token extraction edge cases (TokenExtractor target)
- Role normalization injection risks (RoleNormalization target)

**Mitigation Effectiveness:**
- ✅ **Excellent:** CRITICAL severity labeling for fuzz findings (AC6)
- ✅ **Excellent:** Security team auto-tagging in GitHub Issues
- ✅ **Strong:** Crash corpus saved for reproduction (AC3)
- ✅ **Adequate:** Slack/email immediate notification

**Security Response Process (Recommended):**
1. Fuzz finding triggers CRITICAL issue + immediate notification
2. Security team triages within 4 hours
3. Severity assessment: CVSS scoring
4. Fix timeline based on severity (Critical: 24h, High: 7 days)
5. Regression test added to prevent recurrence
