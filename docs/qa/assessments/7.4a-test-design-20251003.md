## Test Strategy Overview

- **Total Test Scenarios**: 16
- **Unit Tests**: 13 (81%)
- **Integration Tests**: 3 (19%)
- **E2E Tests**: 0 (0% - not applicable for framework library)
- **Priority Distribution**:
  - **P0 (Critical Security)**: 6 tests (38%)
  - **P1 (Core Infrastructure)**: 5 tests (31%)
  - **P2 (Theming/Components/Build)**: 5 tests (31%)

**Test Philosophy**: "Shift-left" approach prioritizing unit tests for fast feedback. Frontend framework libraries are best validated through unit and integration tests. E2E tests will occur in consuming product modules (Story 7.4b, Epic 8).

**Risk Alignment**: Test design directly addresses highest risks from risk profile (SEC-001, SEC-002, SEC-003 all score 6). Security tests account for 38% of test suite.

|---------|-------|----------|------------------|---------------|-----------------|

| 7.4a-UNIT-P0-006 | Unit | P0 | Token refresh flow edge cases (expiration during refresh, network failure) | Tests automatic token refresh logic with error handling | SEC-003, SEC-001 |

**Given-When-Then Descriptions**:

**7.4a-UNIT-P0-001: XSS Payload Sanitization**
- **Given**: A malicious XSS payload string (`<script>alert('XSS')</script>`, `<img src=x onerror=alert(1)>`)
- **When**: `dataProvider.create` or `update` method is called with data containing this payload, and `DOMPurify` is applied
- **Then**: The payload is sanitized, resulting string does not contain executable script tags or attributes, preventing XSS execution

**7.4a-UNIT-P0-002: CSRF Protection**
- **Given**: An authenticated user session with JWT tokens in `localStorage`
- **When**: `dataProvider` makes an API request
- **Then**: JWT is transmitted via `Authorization: Bearer` header, no session-identifying cookies are sent or accepted, confirming CSRF resistance

**7.4a-UNIT-P0-003: Tenant Context Propagation**
- **Given**: A valid JWT containing `tenant_id` claim
- **When**: `dataProvider` processes an API request
- **Then**: `tenant_id` is correctly extracted from JWT, and `X-Tenant-ID` header is injected
- **Given**: A JWT missing `tenant_id` claim
- **When**: `dataProvider` attempts to process an API request
- **Then**: Request is rejected with error, demonstrating fail-closed behavior

**7.4a-UNIT-P0-004: Keycloak Login Flow**
- **Given**: Valid Keycloak credentials
- **When**: `authProvider.login()` method is called
- **Then**: Keycloak successfully authenticates, `access_token` and `refresh_token` are stored in `localStorage`
- **Given**: Invalid credentials or network error during login
- **When**: `authProvider.login()` method is called
- **Then**: Login fails, `localStorage` remains clear, promise rejects with appropriate error

**7.4a-UNIT-P0-005: Token Expiration Detection**
- **Given**: A valid, unexpired JWT stored in `localStorage`
- **When**: `authProvider.checkAuth()` method is called
- **Then**: Method resolves, indicating authenticated session
- **Given**: Missing JWT or expired refresh token
- **When**: `authProvider.checkAuth()` method is called
- **Then**: Method rejects, triggering logout or redirect to login page

**7.4a-UNIT-P0-006: Token Refresh Edge Cases**
- **Given**: `access_token` nearing expiration (<5 min remaining) and valid `refresh_token`
- **When**: `authProvider.checkAuth()` method is called
- **Then**: `refreshAccessToken` function invoked, new `access_token` obtained and stored, method resolves
- **Given**: Valid `access_token` but expired `refresh_token` or network error during refresh
- **When**: `authProvider.checkAuth()` attempts to refresh token
- **Then**: Method rejects, triggering full re-login flow or logout

- **Given**: API response with malformed/non-RFC 7807 error structure

**7.4a-UNIT-P1-005: Logout Flow**
- **Given**: Authenticated user with tokens in `localStorage`
- **When**: `authProvider.logout()` method called
- **Then**: All authentication-related items cleared from `localStorage`, user redirected to Keycloak logout endpoint

## Test Level Distribution Rationale

### Unit Tests: 13 (81%)

**Justification**: Frontend framework libraries benefit from extensive unit testing because:
1. **Pure logic testability**: dataProvider/authProvider methods are stateless functions (easy to test in isolation)
2. **Fast feedback**: Unit tests run in milliseconds (Vitest), enabling TDD workflow
3. **Security validation**: XSS sanitization, JWT handling, tenant extraction are pure logic (no external dependencies)
4. **Component isolation**: React components can be tested with @testing-library/react without full app context

**Examples**:
- `dataProvider` JWT injection, tenant extraction, error mapping (pure functions)
- `authProvider` login/logout logic, token validation (stateless)
- `AdminShell` resource registration (component rendering)

### Integration Tests: 3 (19%)

**Justification**: Integration tests validate:
1. **Build pipeline**: Vite library mode, TypeScript declaration generation (full build process)
2. **Consumption contract**: Simulating product module import and usage (BUS-001, BUS-002)
3. **Full stack validation**: AdminShell + providers + theme working together

**Examples**:
- `npm run build` → verify dist/ outputs (integration of Vite + TypeScript + rollup)
- Import library in test project → render AdminShell (integration of all framework components)

### E2E Tests: 0 (0%)

**Justification**: E2E tests not applicable for framework library because:
1. **No deployable application**: Framework is consumed by product modules, not deployed standalone
2. **User journeys in products**: E2E tests belong in consuming apps (Epic 8 licensing-server frontend)
3. **Browser automation overhead**: E2E tests slow, brittle, and unnecessary for infrastructure libraries

**Where E2E occurs**: Story 7.4b (generator testing), Story 8.2/8.3 (licensing-server frontend integration)

## Recommended Execution Order

### Phase 1: Security Tests First (P0)
**Run first for fail-fast on critical vulnerabilities**

1. **7.4a-UNIT-P0-001**: XSS sanitization (SEC-001, SEC-004)
2. **7.4a-UNIT-P0-002**: CSRF protection (SEC-002, SEC-005)
3. **7.4a-UNIT-P0-003**: Tenant context propagation (SEC-002)
4. **7.4a-UNIT-P0-004**: Keycloak login flow (SEC-003)
5. **7.4a-UNIT-P0-005**: Token expiration (SEC-003)
6. **7.4a-UNIT-P0-006**: Token refresh edge cases (SEC-003)

**Rationale**: Security risks are highest (score 6). If these tests fail, implementation must be corrected before proceeding. Estimated runtime: ~10 seconds (Vitest unit tests).

### Phase 2: Core Infrastructure Tests (P1)
**Validate core functionality after security is proven**

7. **7.4a-UNIT-P1-001**: AdminShell resource registration (TECH-001, BUS-001)
8. **7.4a-UNIT-P1-002**: JWT token injection (SEC-001)
9. **7.4a-UNIT-P1-003**: RFC 7807 error mapping (TECH-002)
10. **7.4a-UNIT-P1-004**: Role extraction (SEC-003)
11. **7.4a-UNIT-P1-005**: Logout flow (SEC-003)

**Rationale**: Core framework APIs must work correctly for product modules to consume. Estimated runtime: ~8 seconds.

### Phase 3: Theming & Components (P2)
**Validate secondary features and UX polish**

12. **7.4a-UNIT-P2-001**: Theme application (TECH-001)
13. **7.4a-UNIT-P2-002**: Shared components (BUS-002)

**Rationale**: Important for UX but not blocking for MVP. Can be deferred if time-constrained. Estimated runtime: ~5 seconds.

### Phase 4: Integration & Build Validation (P2)
**Final validation before npm publish**

14. **7.4a-INTEGRATION-P2-003**: Build library, verify exports (TECH-003, TECH-004)
15. **7.4a-INTEGRATION-P2-004**: Import library in test project (BUS-001, BUS-002)

**Rationale**: Validates end-to-end build and consumption. Run after all unit tests pass. Estimated runtime: ~15 seconds (npm build + test project import).

**Total Estimated Runtime**: ~38 seconds for full test suite

  "iat": 1735686000

**Expired JWT**:

{
  "sub": "user-uuid",
  "tenant_id": "tenant-a-uuid",

### Mock Requirements

```typescript
beforeEach(() => {
  global.localStorage = {
    getItem: vi.fn(),
    setItem: vi.fn(),
    removeItem: vi.fn(),
    clear: vi.fn()
  };
});
```

**Fetch Mock** (for dataProvider/authProvider network calls):
```typescript
global.fetch = vi.fn(() =>
  Promise.resolve({
    ok: true,
    json: () => Promise.resolve({ access_token: '...', refresh_token: '...' })
  })
);
```

**DOMPurify Mock** (for sanitization tests):
```typescript
import DOMPurify from 'dompurify';
vi.mock('dompurify');
DOMPurify.sanitize.mockReturnValue('sanitized output');
```

## Key Principles Applied

1. **Shift-left**: 81% unit tests enable fast feedback loops (TDD workflow)
2. **Risk-based**: 38% P0 security tests address highest-risk areas (SEC-001, SEC-002, SEC-003)
3. **Efficient coverage**: Test once at the right level (no duplicate integration tests for unit-testable logic)
4. **Maintainability**: Pure functions (dataProvider, authProvider) are highly testable and maintainable
5. **Fast feedback**: Entire test suite runs in ~38 seconds (enables frequent execution)
