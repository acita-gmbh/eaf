## Risk Register

### TECH-001: Flowable Auto-Migration Failure

**Category**: Technical
**Probability**: Low (1) - Flowable 7.1.0 mature, Spring Boot auto-config well-tested
**Impact**: Medium (2) - Would block story completion, require manual schema intervention
**Risk Score**: 2 (LOW RISK)

**Description**: Flowable auto-migration might fail to create schema on first startup due to PostgreSQL permissions, connection issues, or schema naming conflicts.

**Affected Components**:
- `framework/workflow` module
- PostgreSQL `flowable` schema
- FlowableConfiguration auto-config

**Detection Method**: Integration test validates schema creation (Subtask 5.3)

**Mitigation Strategy**: PREVENTIVE
- **Actions**:
  1. Integration test validates 15+ ACT_* tables created (Subtask 5.3)
  2. Startup validation confirms ProcessEngine.getRepositoryService() available (Subtask 4.3)
  3. SQL query verifies schema tables exist before proceeding
  4. Testcontainers provides realistic PostgreSQL environment
- **Testing Requirements**:
  - Integration test with real PostgreSQL via Testcontainers
  - SQL catalog query validation
  - Startup log validation for migration success
- **Residual Risk**: Minimal - Integration tests catch failures early in development
- **Owner**: Dev Agent
- **Timeline**: Before story completion (validated in Task 5)

---

### TECH-002: PostgreSQL Schema Naming Conflict

**Category**: Technical
**Probability**: Low (1) - Dedicated `flowable` schema name unlikely to conflict
**Impact**: Low (1) - Easy to resolve by renaming schema in configuration
**Risk Score**: 1 (MINIMAL RISK)

**Description**: The `flowable` schema name might conflict with existing database objects, preventing schema creation.

**Affected Components**:
- PostgreSQL `flowable` schema
- application.yml configuration

**Detection Method**: Startup failure or integration test failure

**Mitigation Strategy**: PREVENTIVE
- **Actions**:
  1. Use dedicated schema name (`flowable`) unlikely to conflict with application schemas
  2. Integration test validates schema accessible
  3. Configuration externalized to application.yml (easy to change if needed)
- **Testing Requirements**:
  - Integration test schema query (Subtask 5.3)
- **Residual Risk**: Negligible - Schema name can be changed in configuration if conflict occurs
- **Owner**: Dev Agent
- **Timeline**: Immediate (configuration change if needed)

### OPS-001: Missing Flowable-Specific Operational Monitoring

**Category**: Operational
**Probability**: Medium (2) - Story doesn't specify Flowable engine metrics/monitoring
**Impact**: Low (1) - Epic 5 observability provides basic monitoring, Flowable metrics can be added later
**Risk Score**: 2 (LOW RISK)

**Description**: No Flowable-specific monitoring (process execution metrics, job failures, task durations) specified in this story. Operational visibility into workflow engine health may be limited.

**Affected Components**:
- Flowable ProcessEngine
- Observability infrastructure (Epic 5)

**Detection Method**: Code review - no Flowable metrics integration specified

**Mitigation Strategy**: DETECTIVE
- **Actions**:
  1. Epic 5 observability infrastructure (Prometheus, logs, tracing) already in place
  2. Flowable Spring Boot Actuator endpoints provide basic engine health
  3. Future story can add custom Flowable metrics if operational needs emerge
  4. Document as technical debt for Epic 6+ stories
- **Testing Requirements**:
  - Verify Flowable actuator endpoints available (if applicable)
  - Integration test confirms engine operational
- **Residual Risk**: Low - Basic observability sufficient for MVP, can enhance based on operational experience
- **Owner**: Product Owner (prioritize future story if needed)
- **Timeline**: Post-MVP if operational monitoring gaps identified
