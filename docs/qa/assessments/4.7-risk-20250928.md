## Detailed Risk Register

### Medium Priority Risks (Score: 4)

#### API-001: WidgetController Endpoints Missing or Not Scanned
**Score: 4 (Medium)**
**Category**: Technical/Integration
**Probability**: Medium (2) - WidgetController may not exist or be properly scanned
**Impact**: Medium (2) - Blocks REST API tests but doesn't affect domain tests

**Affected Components**:
- WidgetApiIntegrationTest (7 tests)
- WidgetWalkingSkeletonIntegrationTest (2 tests)
- Spring Boot component scanning configuration
- WidgetDemoApplication component scan paths

**Rationale**:
- Tests expect /widgets endpoints (POST, GET, PUT, DELETE)
- WidgetDemoApplication may not scan widget packages correctly
- Framework auto-configuration may not include web controllers
- Component scanning configuration may be incomplete

**Mitigation Strategy** (Preventive + Corrective):
1. **Verify WidgetController Exists**: Check if controller class exists in widget-demo
2. **Check Component Scanning**: Verify WidgetDemoApplication scans correct packages
3. **Test Endpoint Discovery**: Use Spring Boot actuator to list mapped endpoints
4. **Create if Missing**: Add WidgetController if not present
5. **Incremental Testing**: Test each endpoint individually after creation

**Testing Requirements**:
- Verify /widgets endpoints respond with 2xx or expected error codes
- Test component scanning includes widget packages
- Validate Spring Boot auto-configuration includes web layer

**Residual Risk**: Low - Standard Spring Boot controller pattern, well-documented

**Owner**: Dev Agent
**Timeline**: Task 1 (REST API fixes)

**Expected Resolution**:
- If WidgetController exists: Configure scanning (30 min)
- If WidgetController missing: Create standard REST controller (1 hour)

#### AUTH-001: JWT Authentication Configuration Complex
**Score: 4 (Medium)**
**Category**: Security/Configuration
**Probability**: Medium (2) - Keycloak TestContainer + JWT validation complex
**Impact**: Medium (2) - Blocks authentication tests but domain tests unaffected

**Affected Components**:
- KeycloakTestTokenProvider token generation
- Keycloak TestContainer realm configuration
- 10-layer JWT validation system (from Epic 3)
- WidgetApiIntegrationTest authentication tests (4 tests)

**Rationale**:
- Keycloak TestContainer requires realm setup
- JWT tokens must include correct claims (tenant_id, roles)
- 10-layer JWT validation system may not accept test tokens
- TestContainer Keycloak may not be properly configured

**Mitigation Strategy** (Detective + Corrective):
1. **Test Token Generation**: Verify KeycloakTestTokenProvider.getAdminToken() works
2. **Validate Token Claims**: Check token includes tenant_id, iss, exp, etc.
3. **Test JWT Validation**: Verify 10-layer validation accepts test tokens
4. **Realm Configuration**: Check Keycloak TestContainer realm setup
5. **Fallback**: Use mock JWT tokens if Keycloak configuration too complex

**Testing Requirements**:
- Generate test token and inspect claims
- Test token against 10-layer JWT validation
- Verify Keycloak TestContainer starts with correct realm
- Test authentication endpoints return 401 for invalid tokens

**Residual Risk**: Medium - Keycloak configuration can be complex, but fallback options exist

**Owner**: Dev Agent
**Timeline**: Task 3 (JWT authentication)

**Expected Resolution**:
- Simple token issues: 30 min
- Keycloak realm configuration: 1-2 hours
- Fallback to mock tokens: 30 min

#### DOMAIN-001: TenantContext IllegalStateException in Tests
**Score: 2 (Low)**
**Category**: Technical/Domain Logic
**Probability**: Low (1) - Pattern works in framework modules
**Impact**: Medium (2) - Affects domain-level tests

**Affected Components**:
- WidgetDomainIntegrationTest (4 tests)
- TenantContext ThreadLocal management
- Widget aggregate command handlers
- AggregateTestFixture integration

**Rationale**:
- TenantContext.getCurrentTenantId() throws IllegalStateException when no tenant set
- Widget command handlers require tenant context
- AggregateTestFixture may not properly set ThreadLocal
- Test setup may not call tenantContext.setCurrentTenantId() correctly

**Mitigation Strategy** (Detective + Corrective):
1. **Debug IllegalStateException**: Check exact error message and stack trace
2. **Verify TenantContext Setup**: Ensure setCurrentTenantId() called before commands
3. **Check ThreadLocal Scope**: Verify ThreadLocal works in test environment
4. **Pattern Validation**: Compare with working TenantBoundaryValidationIntegrationTest
5. **Timing Fix**: Ensure tenant set before aggregate construction

**Testing Requirements**:
- Verify TenantContext.getCurrentTenantId() returns expected value
- Test ThreadLocal isolation between test methods
- Validate tenant context cleanup after each test

**Residual Risk**: Very Low - Pattern proven in TenantBoundaryValidationIntegrationTest

**Owner**: Dev Agent
**Timeline**: Task 4 (Domain logic)
