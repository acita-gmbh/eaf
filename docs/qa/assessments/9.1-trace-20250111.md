### AC 17: OIDC Authentication Flow Working (Critical - Security)

**Validation Test:** 8.8-E2E-017
**Coverage:** **FULL**
**Priority:** P0

**Given:** User navigates to React-Admin portal (http://localhost:5173)
**When:** User is not authenticated
**Then:** Complete OIDC flow executes:
1. Redirect to Keycloak login page
2. User enters credentials and submits
3. Token acquisition after successful login
4. Token stored in localStorage or session
5. Token refresh mechanism works before expiration
6. User can logout successfully

**Validation Method:**
- Task 4: 7 subtasks validate each step
- Subtask 4.3: Test login redirect
- Subtask 4.4: Test token acquisition
- Subtask 4.5: Test token refresh
- Subtask 4.6: Test logout flow
- Subtask 4.7: Verify Authorization header in network tab

**Test Priority:** P0 (Blocking - authentication is prerequisite for all other functionality)

**Risk Mitigated:** SEC-002 (Keycloak auth issues - Score 3), TECH-004 (Frontend-backend integration - Score 6)

**NFR Validation:** Security NFR (Keycloak OIDC requirement)

### AC 27: Tenant Isolation Enforced (Critical - Security)

**Validation Test:** 8.8-E2E-027
**Coverage:** **FULL**
**Priority:** P0

**Given:** Two tenants exist with separate widget data
  - Tenant A: 5 widgets ("Widget A-1" through "Widget A-5")
  - Tenant B: 3 widgets ("Widget B-1" through "Widget B-3")

**When:** User authenticates as Tenant A

**Then:** Strict tenant isolation enforced
- Widget List displays ONLY Tenant A's 5 widgets
- Tenant B's 3 widgets are NOT visible
- Tenant B widgets NOT in search results
- Tenant B widgets NOT accessible via direct URL
- Creating new widget automatically associates with Tenant A
- Cannot view/edit Tenant B widgets (403 Forbidden or filtered)

**Validation Method:**
- Subtask 5.11: Verify tenant isolation (multi-tenant test)
- Create test data for two separate tenants
- Login as each tenant and verify isolation
- Attempt cross-tenant access (should fail)
- Network tab inspection: Verify tenant context in JWT

**Test Priority:** P0 (Security-critical - tenant data leakage prevention)

**Risk Mitigated:** Tenant isolation breach (security vulnerability)

**NFR Validation:** Security NFR (multi-tenant isolation requirement)

**Related Backend:** Epic 4 3-layer tenant isolation (request filter → service validation → database RLS)

### AC 33: Keyboard Navigation Functional for All CRUD Operations (Critical - Accessibility)

**Validation Test:** 8.8-ACC-033
**Coverage:** **FULL**
**Priority:** P0

**Given:** User with keyboard only (no mouse available)
**When:** User attempts to complete all Widget CRUD operations
**Then:** All functionality accessible via keyboard:
- **Tab:** Moves focus through interactive elements in logical order
- **Enter/Space:** Activates buttons and links
- **Escape:** Closes dialogs and modals
- **No keyboard traps:** User can navigate away from any element
- **Logical tab order:** Follows visual layout top-to-bottom, left-to-right

**Specific Keyboard Flows to Validate:**
1. Navigate to Widget List → Press Tab to focus first widget → Press Enter to view details
2. Navigate to Create Widget → Tab through form fields → Press Enter to submit
3. Navigate to Edit Widget → Tab to field → Type changes → Tab to Save → Press Enter
4. Open dialog → Press Escape → Dialog closes

**Validation Method:**
- Task 7, Subtasks 7.1-7.4: Comprehensive keyboard navigation testing
- Test through Widget List page (Subtask 7.1)
- Test through Create Widget form (Subtask 7.2)
- Test through Edit Widget form (Subtask 7.3)
- Verify Tab, Enter, Space, Escape keys work correctly (Subtask 7.4)

**Test Priority:** P0 (PRD requirement - WCAG AA compliance mandatory)

**WCAG Standard:** 2.1.1 Keyboard (Level A - required for AA compliance)

**Risk Mitigated:** SEC-003 (WCAG AA compliance - Score 2)

### AC 34: Screen Reader Compatibility Verified (Critical - Accessibility)

**Validation Test:** 8.8-ACC-034
**Coverage:** **FULL**
**Priority:** P0

**Given:** User with screen reader enabled (NVDA, JAWS, or VoiceOver)
**When:** User navigates through Widget CRUD operations
**Then:** Screen reader compatibility verified:
- **ARIA labels:** All form fields have labels announced clearly
- **ARIA announcements:** Success/error states announced via ARIA live regions
- **Semantic HTML:** Logical structure with nav, main, form, button elements
- **Navigation landmarks:** Clear regions for screen reader navigation

**Specific Screen Reader Behaviors to Validate:**
1. Form fields announced with label and type (e.g., "Widget Name, edit text")
2. Buttons announced with purpose (e.g., "Create Widget, button")
3. Success notification announced (e.g., "Widget created successfully")
4. Error messages announced (e.g., "Name is required, error")
5. Widget List announces item count (e.g., "Showing 10 of 25 widgets")

**Validation Method:**
- Subtask 7.8: Test with screen reader (NVDA, JAWS, or VoiceOver)
- Test on Windows with NVDA (free, open-source)
- Test on Mac with VoiceOver (built-in)
- Optional: Test with JAWS (Windows, commercial)

**Test Priority:** P0 (PRD requirement - WCAG AA compliance mandatory)

**WCAG Standard:** 4.1.2 Name, Role, Value (Level A - required for AA compliance)

**Implementation Requirements:**
- Subtask 7.5: Add ARIA labels to all form fields
- Subtask 7.6: Add ARIA announcements for create/update/delete success
- Subtask 7.7: Verify semantic HTML usage

**Risk Mitigated:** SEC-003 (WCAG AA compliance - Score 2)

### AC 35: Color Contrast Meets WCAG AA Standards (Critical - Accessibility)

**Validation Test:** 8.8-ACC-035
**Coverage:** **FULL**
**Priority:** P0

**Given:** Application rendered in browser with default Material UI theme
**When:** Checking contrast ratios for all text and UI components
**Then:** WCAG AA contrast standards met:
- **Normal text (<18pt):** ≥4.5:1 contrast ratio
- **Large text (≥18pt):** ≥3:1 contrast ratio
- **UI components:** ≥3:1 contrast ratio (buttons, form borders, icons)

**Specific Elements to Validate:**
1. Widget List: Text on background, column headers
2. Forms: Labels, input text, placeholder text
3. Buttons: Primary button text, secondary button text
4. Error messages: Red error text on background
5. Success notifications: Green notification text on background

**Validation Method:**
- Subtask 7.9: Check color contrast using browser DevTools or axe extension
- Chrome DevTools: Inspect element → Styles → Contrast ratio shown
- axe DevTools: Run accessibility scan → Review contrast findings
- WebAIM Contrast Checker: Manual verification of color combinations

**Test Priority:** P0 (PRD requirement - WCAG AA compliance mandatory)

**WCAG Standard:** 1.4.3 Contrast (Minimum) - Level AA

**Material UI Baseline:** Material UI default theme typically meets WCAG AA contrast, but custom theme colors must be validated

**Risk Mitigated:** SEC-003 (WCAG AA compliance - Score 2)
