# Test Design: Story 7.2 - Create "New Module" Generator

**Date**: 2025-10-03
**Designer**: Quinn (Test Architect)
**Story**: 7.2 - Create "New Module" Generator
**Epic**: 7 - Scaffolding CLI v1

## Test Strategy Overview

- **Total test scenarios**: 14
- **Unit tests**: 9 (64%) - Business logic, validation, template rendering
- **Integration tests**: 3 (22%) - End-to-end generation, build verification
- **Manual tests**: 2 (14%) - IDE verification, visual validation
- **Priority distribution**: P0: 12, P1: 1, P2: 1

**Test Pyramid Adherence**: ✅ Excellent
- Follows recommended 70/20/10 split (64/22/14 is very close)
- Shift-left emphasis with majority unit tests
- Integration tests focus on critical builds
- Manual tests only for visual verification

## Test Level Justification

### Why Unit Tests (9 tests, 64%)

**Appropriate for**:
- ✅ Input validation logic (7.2-UNIT-002, 006) - Pure functions, no I/O
- ✅ Template rendering (7.2-UNIT-004) - Mustache is deterministic
- ✅ File structure validation (7.2-UNIT-007, 008) - Directory paths are strings
- ✅ Error handling (7.2-UNIT-009) - Arrow Either pattern is pure
- ✅ settings.gradle.kts parsing (7.2-UNIT-005) - String manipulation

**Justification**:
- Fast feedback (milliseconds vs seconds)
- No external dependencies (file system can be mocked)
- Easy to write and maintain
- 100% reproducible
- Run in any environment

### Why Integration Tests (3 tests, 22%)

**Required for**:
- ✅ **Gradle build verification** (7.2-INT-001) - Can't mock Gradle runtime
- ✅ **Quality gates validation** (7.2-INT-002) - Requires real ktlint/detekt/konsist
- ✅ **Spring Modulith recognition** (7.2-INT-003) - Requires Spring container

**Justification**:
- These validations **cannot be done at unit level**
- Plugin order errors only manifest in real builds
- Quality gate tools require actual source files
- Spring Modulith annotation processing requires runtime

### Why Manual Tests (2 tests, 14%)

**Appropriate for**:
- ✅ IDE visual verification (7.2-MANUAL-001) - Autocomplete, syntax highlighting
- ✅ Formatting preservation (7.2-MANUAL-002) - Human judgment of "looks right"

**Justification**:
- Visual validation best done by humans
- Low priority (P2) - nice-to-have confirmation
- Automation cost > value

## Priority Assignment Rationale

**All functional validations are P0** because:
- CLI tool correctness is critical for developer productivity
- Invalid generated code wastes developer time
- Security vulnerabilities (SEC-002) are unacceptable
- File corruption (TECH-004) breaks builds

**P0 includes**:
- All 9 unit tests (validation, rendering, error handling)
- All 3 integration tests (build success, quality gates, Spring recognition)

### P1 Tests (1 test) - Important But Not Blocking

**None** - All identified tests are either P0 (critical) or P2 (visual confirmation)

### P2 Tests (2 tests) - Nice-to-Have Validation

**Manual tests only**:
- 7.2-MANUAL-001: IDE verification (developer will notice issues anyway)
- 7.2-MANUAL-002: Syntax highlighting (cosmetic validation)

## Maintainability Considerations

### Test Data Management

**Approach**: **Inline test data** (no external files needed)

**Rationale**:
- Module names are simple strings
- Templates are small (< 100 lines each)
- No complex domain models
- Inline data improves readability

**Example**:
```kotlin
test("valid module name creates all files") {
    val moduleName = "test-module"
    // Test continues inline...
}
```

### Test Independence

**All tests are independent**:
- No shared state between tests
- Each test creates its own temporary directory
- No test execution order dependencies
- Can run in parallel (future enhancement)

### Test Clarity

**Good practices followed**:
- Descriptive test names (e.g., "verify malicious module names rejected")
- Given-When-Then structure in test bodies
- Clear assertions (e.g., `result.shouldBeLeft()`)
- Comments explaining WHY not just WHAT
