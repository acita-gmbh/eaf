# Test Design Matrix - Story 2.1: Define Widget Domain Aggregate

## Test Strategy Overview

This test design establishes the foundational testing patterns for CQRS/Event Sourcing using Constitutional TDD with the Nullable Design Pattern.

## Test Scenarios by Acceptance Criteria

### AC1: Spring Modulith Module Creation
**Priority: P0** | **Level: Integration**

- **Scenario 1.1**: Module boundary verification
  - **Given**: Widget module is created with @ApplicationModule
  - **When**: Konsist tests execute
  - **Then**: Spring Modulith boundaries are enforced without violations

### AC2: Widget Aggregate with Axon Annotations
**Priority: P0** | **Level: Unit**

- **Scenario 2.1**: Aggregate creation and annotation validation
  - **Given**: Widget class with @Aggregate and @AggregateIdentifier
  - **When**: Aggregate is instantiated
  - **Then**: Axon framework recognizes and manages the aggregate

### AC3: CQRS Command/Event/Handler Implementation
**Priority: P0** | **Level: Unit + Integration**

- **Scenario 3.1**: Command handling (Unit - Nullable)
  - **Given**: CreateWidgetCommand with valid data
  - **When**: Command handler processes the command
  - **Then**: WidgetCreatedEvent is applied to aggregate

- **Scenario 3.2**: Event sourcing persistence (Integration - Testcontainers)
  - **Given**: CreateWidgetCommand dispatched via command bus
  - **When**: Event is persisted to event store
  - **Then**: Event appears in Axon domain_event_entry table

### AC4: Constitutional TDD with Nullable Design Pattern
**Priority: P0** | **Level: Unit**

- **Scenario 4.1**: Fast domain testing with Nullable Pattern
  - **Given**: Nullable infrastructure substitutes
  - **When**: Domain logic tests execute
  - **Then**: Tests complete in <5ms average with full business logic validation

### Edge Cases & Error Scenarios
**Priority: P1** | **Level: Unit**

- **Scenario 5.1**: Invalid command validation
  - **Given**: CreateWidgetCommand with invalid data
  - **When**: Command validation occurs
  - **Then**: DomainError returned via Arrow Either pattern

- **Scenario 5.2**: Tenant context validation
  - **Given**: Command without proper tenant context
  - **When**: Multi-tenancy validation occurs
  - **Then**: Command rejected with tenant isolation error

## Test Coverage Expectations

| Component | Unit Tests | Integration Tests | Coverage Target |
|-----------|------------|-------------------|-----------------|
| Widget Aggregate | 80% line, 80% mutation | Event persistence validation | 90%+ |
| CreateWidgetCommand | Validation scenarios | Command bus dispatch | 80%+ |
| WidgetCreatedEvent | Serialization/deserialization | Event store retrieval | 80%+ |

## Performance Benchmarks

- **Nullable Pattern Tests**: Target 61.6% improvement over integration baseline
- **Integration Tests**: <200ms per test with Testcontainers
- **Full Test Suite**: <30s total execution time

## Test Data Strategy

- **Domain Tests**: Use deterministic test data builders
- **Integration Tests**: Use Testcontainers with controlled data sets
- **No Shared State**: Each test creates its own isolated aggregate instances
