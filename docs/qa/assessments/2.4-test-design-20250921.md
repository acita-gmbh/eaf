## Detailed Test Scenario Specifications

### Priority 0 (P0) Test Scenarios - Critical Epic 2 Validation

#### 2.4-UNIT-001: Query Structure Validation
- **Purpose**: Validate FindWidgetByIdQuery requires widgetId and tenantId
- **Test Type**: Unit test for query structure validation
- **Expected Outcome**: Query validation prevents missing required fields
- **Risk Mitigation**: Reduces QueryGateway integration complexity (TECH-001)

#### 2.4-INT-001: QueryGateway Integration
- **Purpose**: Validate WidgetQueryHandler processes queries via QueryGateway
- **Test Type**: Integration test with QueryGateway and WidgetProjectionRepository
- **Expected Outcome**: Query dispatch and response handling works correctly
- **Risk Mitigation**: Directly addresses QueryGateway integration risk (TECH-001)

#### 2.4-INT-002: Pagination Query Processing
- **Purpose**: Validate FindWidgetsQuery handler with pagination and filtering
- **Test Type**: Integration test with database and pagination validation
- **Expected Outcome**: Pagination parameters processed correctly with tenant isolation
- **Risk Mitigation**: Addresses query complexity and performance concerns

#### 2.4-INT-004: REST API Query Endpoint
- **Purpose**: Validate GET /widgets/{id} endpoint dispatches queries correctly
- **Test Type**: Integration test with REST controller and QueryGateway
- **Expected Outcome**: API endpoint integrates with query processing seamlessly
- **Risk Mitigation**: Reduces REST API extension complexity (TECH-002)

#### 2.4-INT-005: REST API Pagination Endpoint
- **Purpose**: Validate GET /widgets endpoint with pagination parameters
- **Test Type**: Integration test with API controller and query processing
- **Expected Outcome**: Pagination and filtering work through complete API flow
- **Risk Mitigation**: Validates API pagination implementation

#### 2.4-UNIT-005: Read Model Isolation
- **Purpose**: Validate query handlers access only WidgetProjectionRepository
- **Test Type**: Unit test for architectural boundary enforcement
- **Expected Outcome**: No direct event store access in query handlers
- **Risk Mitigation**: Ensures CQRS read-side isolation (architectural requirement)

#### 2.4-INT-008: CQRS Read-Side Isolation
- **Purpose**: Validate query operations completely isolated from event store
- **Test Type**: Integration test verifying read model boundaries
- **Expected Outcome**: Query processing never accesses domain_event_entry tables
- **Risk Mitigation**: Validates data consistency and CQRS architecture (DATA-001)

#### 2.4-E2E-001: Complete Walking Skeleton Flow
- **Purpose**: Validate POST command → projection → GET query → data consistency
- **Test Type**: End-to-end integration test with Testcontainers
- **Expected Outcome**: Complete CQRS flow works within <500ms performance target
- **Risk Mitigation**: Critical Epic 2 validation addressing PERF-001 and BUS-001

### Priority 1 (P1) Test Scenarios - Important Validation

#### 2.4-UNIT-002: Pagination Parameter Validation
- **Purpose**: Validate FindWidgetsQuery pagination parameter handling
- **Test Type**: Unit test for query parameter processing
- **Expected Outcome**: Pagination defaults and validation work correctly

#### 2.4-INT-003: Multi-Tenant Query Isolation
- **Purpose**: Validate tenant isolation throughout query processing chain
- **Test Type**: Integration test with multi-tenant data scenarios
- **Expected Outcome**: Queries return only tenant-specific data

#### 2.4-UNIT-003: DTO Security Validation
- **Purpose**: Validate WidgetResponse excludes sensitive fields
- **Test Type**: Unit test for response DTO field validation
- **Expected Outcome**: tenantId and updatedAt excluded from API responses

#### 2.4-UNIT-004: Data Transformation Logic
- **Purpose**: Validate WidgetProjection to WidgetResponse conversion
- **Test Type**: Unit test for data mapping and JSON metadata handling
- **Expected Outcome**: Accurate data transformation with metadata deserialization

#### 2.4-INT-006: JWT Authentication Integration
- **Purpose**: Validate API endpoints require and process JWT authentication
- **Test Type**: Integration test with authentication and authorization
- **Expected Outcome**: Proper JWT validation and tenant extraction

#### 2.4-E2E-002: Multi-Tenant Walking Skeleton
- **Purpose**: Validate complete flow with multiple tenants
- **Test Type**: End-to-end test with tenant isolation validation
- **Expected Outcome**: Walking Skeleton maintains tenant boundaries throughout

### Priority 2 (P2) Test Scenarios - Additional Coverage
