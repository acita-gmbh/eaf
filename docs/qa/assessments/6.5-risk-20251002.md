## Detailed Risk Register
- **TECH-001 (Score 6, High)**
  - *Drivers:* BPMN boundary service task must hydrate `operation="sendCommand"`, `commandType`, payload map, and `compensationEnabled`. Null/incorrect variables will halt process in incident queue.
  - *Mitigations:*
    - Extend BPMN doc block with explicit variable checklist.
    - Integration test to assert execution leaves compensation branch successfully.
    - Code review emphasis on `DispatchAxonCommandTask` switch extension.
  - *Testing Focus:* End-to-end failure scenario via Ansible EE container.

- **SEC-001 (Score 3, Low but Security-critical):**
  - *Drivers:* Compensation commands must honor TenantContext to avoid cross-tenant rollback or privilege escalation.
  - *Mitigations:*
    - Reuse fail-closed tenant validation guard in DispatchAxonCommandTask.
    - Add negative unit test (tenant mismatch -> BpmnError TENANT_ISOLATION_VIOLATION).
    - Verify BPMN branch propagates tenantId from failure path before dispatch.
  - *Monitoring:* Alert on `TENANT_ISOLATION_VIOLATION` counter spikes.

- **DATA-001 (Score 4, Medium):**
  - *Drivers:* Cancellation event must set consistent metadata (reason/operator) and be idempotent for retries. Missing projection handling could leave widget stuck ACTIVE.
  - *Mitigations:*
    - Update WidgetProjection handler & tests for cancellation event.
    - Define cancellation reason taxonomy in Dev Notes for traceability.
    - Mutation tests covering duplicate events.

- **OPS-001 (Score 4, Medium):**
  - *Drivers:* Need to instrument Flowable metrics (BPMN error counter, compensation tally) and Grafana dashboards to detect rising failure rate.
  - *Mitigations:*
    - Extend FlowableMetrics with `compensation.triggered` counter or tag existing BPMN error counter.
    - Add Dev task documenting alert thresholds (e.g., >5 cancels/hour).
    - Ensure RunAnsiblePlaybookTask logs include compensation correlation IDs.

- **PERF-001 (Score 2, Low):**
  - *Drivers:* Compensation path adds second Axon dispatch per failure. Under bulk failure storms, Redis-backed rate limiter (100 events/s/tenant) may throttle legitimate retries.
  - *Mitigations:*
    - Load-test compensation scenario in axonIntegrationTest (optional).
    - Document runbook guidance to monitor Redis rate limiter metrics during incidents.

## Risk-Based Testing Strategy
1. **Priority 1 – Critical/High (TECH-001):**
   - E2E integration test that forces RunAnsiblePlaybookTask failure and asserts workflow completes compensation path, leaving process in terminal state and command dispatched once.
   - BPMN validation test ensuring compensation service task variables populate correctly (unit or BPMN model test).
2. **Priority 2 – Medium (DATA-001, OPS-001):**
   - Unit tests for Widget aggregate & projection verifying cancellation idempotency and metadata propagation.
   - Contract test verifying Flowable metrics increment on compensation branch.
   - Documentation check: Grafana panel / alert definitions.
3. **Priority 3 – Low (SEC-001, PERF-001):**
   - Negative test for tenant mismatch -> BpmnError TENANT_ISOLATION_VIOLATION.
   - Stress scenario (optional) verifying Redis rate limiter not breached by compensation flood (can be deferred, monitor in staging).

## Deployment & Monitoring Recommendations
- Enable feature flag or staged rollout for compensation branch until integration tests pass in staging.
- Add temporary log sampling for compensation path (tenant, commandType, reason).
- Ensure operational runbooks reference cancellation event handling and rollback thresholds.
