## Detailed Risk Register

### Medium Priority Risks (Score: 4)

#### TECH-001: Solution Pattern Incompatibility
**Score: 4 (Medium)**
**Probability**: Medium (2) - The OpenTelemetry issue was the root cause for most, but not necessarily all, test failures.
**Impact**: Medium (2) - Some tests may require additional, specific fixes, extending the 1-day timebox.

**Description**:
The minimal test configuration pattern from Story 8.3 (excluding observability auto-configurations) is the primary solution. However, the two observability-specific tests (`LoggingContextIntegrationTest`, `StructuredLoggingIntegrationTest`) might fail precisely *because* observability is excluded. Other tests might have unrelated issues.

**Mitigation Strategy**:
1.  **Apply Pattern First**: Apply the known workaround to all 7 tests as the first step.
2.  **Isolate Failures**: If a test still fails, investigate its specific logic, assuming the architectural blocker is resolved.
3.  **Timebox Enforcement**: The 1-day-per-test timebox remains the critical mitigation. If a test is still unfixable, it will be escalated.
4.  **Observability Test Strategy**: For the two observability tests, a different approach may be needed, such as creating a custom test slice that *only* includes logging and tracing configurations.

**Residual Risk**: Low - The primary blocker is solved; any remaining issues are likely isolated and manageable.

---

#### BUS-001: Business Logic Drift
**Score: 4 (Medium)**
**Probability**: Medium (2) - Tests were disabled for 6+ months; underlying business logic, API endpoints, or database schemas may have changed.
**Impact**: Medium (2) - Tests will fail at runtime even if the architectural pattern is correct, requiring debugging and updates.

**Description**:
After applying the architectural fix, tests may fail due to:
-   `404 Not Found`: REST API endpoints have been moved or renamed.
-   `AssertionError`: Business logic has changed, leading to different outcomes.
-   `SchemaManagementException`: Database schema has evolved (e.g., new columns, constraints).

**Mitigation Strategy**:
1.  **Expect Failures**: The implementation plan should anticipate these as "normal" failures, not blockers.
2.  **Incremental Fixes**: Address failures one at a time (e.g., fix API endpoints, then update assertions).
3.  **Reference Current Implementation**: Use the current `WidgetController`, `Widget` aggregate, and `widget_projection` schema as the source of truth.
4.  **Update, Don't Revert**: Update the tests to match the current business logic. Do not change the business logic to match the old tests.

**Residual Risk**: Low - These are standard test maintenance activities, not unknown architectural problems.

### Low Priority Risks (Score: 2)

#### PERF-001: Test Execution Exceeds 5-Minute Target
**Score: 2 (Low)**
**Probability**: Low (1) - Story 4.6 established a baseline of 0.228s for 5 tests. Adding 7 more complex tests is unlikely to exceed the 300-second budget.
**Impact**: Medium (2) - Affects CI pipeline efficiency.

**Mitigation**:
1.  **Performance Budget**: The story has a clear <5 minute target (AC 12).
2.  **Measure and Profile**: Task 6.2-6.4 includes measurement and optimization.
3.  **Parallel Execution**: Kotest supports parallel execution if needed.

**Residual Risk**: Minimal - The performance buffer is large.

---

#### REG-001: Regressions in Framework or Baseline Tests
**Score: 2 (Low)**
**Probability**: Low (1) - Changes are isolated to test files; no production code is modified.
**Impact**: Medium (2) - A regression would indicate a deeper issue.

**Mitigation**:
1.  **Regression Suite**: Task 7 includes running all integration tests to catch regressions.
2.  **Isolation**: The `@TestConfiguration` pattern isolates changes, reducing the chance of side effects.
3.  **No Shared Config Changes**: The fix does not involve modifying shared convention plugins.

**Residual Risk**: Minimal.
