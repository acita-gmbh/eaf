## Acceptance Criteria Mapping

### AC 1: Auto-Assign Tenant on Create

**Acceptance Criterion**:
> All new aggregates (like `CreateWidgetCommand`) must be automatically assigned the `tenant_id` from the `TenantContext` when created.

**Implementation Requirements**:
1. CreateWidgetCommand handler extracts tenantId from TenantContext
2. Command handler validates command.tenantId matches TenantContext tenant
3. WidgetCreatedEvent receives validated tenantId
4. Aggregate state stores tenantId from event

---

#### AC1-Trace-001: Given-When-Then - Matching Tenant Success

**Given**:
- TenantContext set to "tenant-a" (from JWT via Layer 1 filter)
- CreateWidgetCommand with command.tenantId = "tenant-a"

**When**:
- Widget constructor command handler invoked
- Handler calls `TenantContext.getCurrentTenantId()` (returns "tenant-a")
- Handler validates: `require(command.tenantId == currentTenant)`

**Then**:
- Validation passes (both equal "tenant-a")
- WidgetCreatedEvent applied with tenantId="tenant-a"
- Widget aggregate created with this.tenantId="tenant-a"

**Test Coverage**:
- **UT-4.2-001**: CreateWidget - Matching tenant success path
- **IT-4.2-001**: Cross-tenant create attack (negative validation)

**Validation Checkpoints**:
- [ ] TenantContext.getCurrentTenantId() called in constructor
- [ ] command.tenantId validated against currentTenant
- [ ] WidgetCreatedEvent contains validated tenantId
- [ ] Widget aggregate state has correct tenantId

### AC 2: Load Verification - Existing Aggregate Tenant Validation

**Acceptance Criterion**:
> When loading an existing aggregate (e.g., for an Update command), the framework must verify that the aggregate's stored tenant ID matches the `TenantContext` tenant ID.

**Implementation Requirements**:
1. UpdateWidgetCommand handler loads existing Widget aggregate
2. Handler validates aggregate.tenantId matches TenantContext.getCurrentTenantId()
3. Handler also validates command.tenantId matches TenantContext
4. Both validations required (command AND aggregate)

---

#### AC2-Trace-001: Given-When-Then - Matching Aggregate Tenant Success

**Given**:
- Existing Widget aggregate with this.tenantId = "tenant-a"
- TenantContext set to "tenant-a"
- UpdateWidgetCommand with command.tenantId = "tenant-a"

**When**:
- Widget.handle(UpdateWidgetCommand) invoked
- Handler calls `TenantContext.getCurrentTenantId()` (returns "tenant-a")
- Handler validates: `ensure(this.tenantId == currentTenant)` (passes)
- Handler validates: `ensure(command.tenantId == currentTenant)` (passes)

**Then**:
- Both validations pass
- WidgetUpdatedEvent applied
- Widget aggregate state updated
- Either.right(Unit) returned

**Test Coverage**:
- **UT-4.2-008**: UpdateWidget - Matching tenant success path
- **IT-4.2-002**: Cross-tenant update attack (negative validation)

**Validation Checkpoints**:

### AC 3: Mismatch Handling - Security Exception and Rollback

**Acceptance Criterion**:
> If the tenant IDs do not match, a Security Exception is thrown, the transaction is rolled back, and the operation fails (e.g., 403 Forbidden or 404 Not Found).

**Implementation Requirements**:
1. Throw IllegalArgumentException for CreateWidget (constructor constraint)
2. Return Either.left(WidgetError.TenantIsolationViolation) for UpdateWidget
3. Transaction automatically rolled back by Axon on exception
4. No events persisted in event store
5. Aggregate state unchanged

---

#### AC3-Trace-001: Given-When-Then - Exception Thrown on Mismatch

**Given**:
- TenantContext set to "tenant-a"
- Command (Create or Update) with tenantId = "tenant-b"

**When**:
- Command handler validates tenant
- Mismatch detected: command.tenantId ("tenant-b") != currentTenant ("tenant-a")

**Then**:
- **CreateWidget**: IllegalArgumentException thrown with message "Tenant isolation violation: command=tenant-b, context=tenant-a"
- **UpdateWidget**: Either.left(WidgetError.TenantIsolationViolation(requestedTenant="tenant-b", actualTenant="tenant-a"))

**Test Coverage**:
- **UT-4.2-002**: CreateWidget - SecurityException thrown
- **UT-4.2-009**: UpdateWidget - Either.left returned
- **IT-4.2-001**: Cross-tenant create attack
- **IT-4.2-002**: Cross-tenant update attack

**Validation Checkpoints**:
- [ ] Exception thrown for CreateWidget
- [ ] Either.left returned for UpdateWidget
- [ ] Error message includes both tenant IDs
- [ ] Structured error type (TenantIsolationViolation)

---
