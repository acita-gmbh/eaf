## Critical Risks Requiring Immediate Attention

### 1. SEC-001: Tenant Context Leakage in Thread Pools

**Score: 9 (Critical)**

**Probability**: High (3) - Thread pools inherently reuse worker threads across event processing operations, creating a persistent environment where ThreadLocal variables can leak between executions.

**Impact**: High (3) - Cross-tenant data exposure represents a complete failure of multi-tenancy isolation (CWE-362: Concurrent Execution using Shared Resource with Improper Synchronization). This could result in:
- Tenant A's data written to Tenant B's tables
- Compliance violations (GDPR, SOC 2)
- Complete loss of data integrity guarantees

**Mitigation**:
- **Mandatory**: Implement `finally` block in `TenantEventMessageInterceptor` that calls `TenantContext.clearCurrentTenant()` after every event handler execution
- Add Kotest integration test simulating sequential events from different tenants on same worker thread
- Validate cleanup occurs even when handler throws exception
- Consider additional Redis-based validation for tenant session consistency

**Testing Focus**:
- Integration test: Process Event A (tenant-a) → Event B (tenant-b) on same thread
- Assert TenantContext is tenant-b, not tenant-a
- Negative test: Deliberately omit cleanup, assert failure detection

**Code Reference**:
- `framework/cqrs/src/main/kotlin/com/axians/eaf/framework/cqrs/interceptors/TenantEventMessageInterceptor.kt`
- Story 4.4 Subtask 1.3

## High Risks

### 2. SEC-002: Fail-Open Risk on Missing Metadata

**Score: 6 (High)**

**Probability**: Medium (2) - Requires a bug in `TenantCorrelationDataProvider` or command interceptor to produce events without `tenantId` metadata.

**Impact**: High (3) - Fail-open behavior would allow unauthorized tenant access, bypassing all three layers of tenant isolation (Layers 1, 2, and 3).

**Mitigation**:
- Implement explicit null-check in `TenantEventMessageInterceptor` with fail-closed exception
- Use generic error message per CWE-209 compliance: "Access denied: required context missing"
- Add architectural test validating fail-closed pattern across all tenant boundaries
- Document metadata contract in KDoc with @throws annotation

**Testing Focus**:
- Negative test: Dispatch event with manually removed `tenantId` metadata
- Assert `SecurityException` thrown with generic message
- Verify event processing stops (no database writes attempted)

**Code Reference**:
- Story 4.4 Subtask 1.1
- `docs/architecture/security.md#session-variable-management`

## Medium Risks

### 3. TECH-001: Async Context Propagation Complexity

**Score: 4 (Medium)**

**Probability**: Medium (2) - Bridging Micrometer Context Propagation with Axon's Unit of Work requires non-trivial framework integration knowledge.

**Impact**: Medium (2) - Implementation complexity could lead to:
- Development delays (2-4 additional days)
- Subtle bugs in context propagation
- Difficulty debugging async failures

**Mitigation**:

- Create prototype integration before full implementation
- Add comprehensive KDoc explaining Micrometer-Axon bridge
- Document configuration in `AxonConfiguration` with inline comments

**Testing Focus**:
- Integration test: Command → Event → Async Projection Handler
- Assert `TenantContext.getCurrentTenantId()` succeeds in @EventHandler
- Log context propagation at each boundary for debugging

## Low Risks

### 4. DATA-001: RLS Bypass Through Raw SQL

**Score: 3 (Low)**

**Probability**: Low (1) - Architecture explicitly prevents raw SQL in projection handlers through repository abstraction.

**Impact**: High (3) - If raw SQL were used, it could bypass `app.current_tenant` session variable, breaking RLS enforcement.

**Mitigation**:
- Enforce architectural rule: All database access through Spring Data JPA repositories
- Add Konsist test validating no SQL injection patterns in projection handlers
- Include code review checklist item for RLS compliance

**Testing Focus**:
- Konsist test: Scan projection package for raw SQL strings
- Manual code review of all projection handlers
- Document architectural constraint in `component-specifications.md`

**Code Reference**:
- Story 4.4 Subtask 3.3
- `docs/architecture/security.md#session-variable-management`

### 5. PERF-001: Interceptor Overhead on Event Processing

**Score: 2 (Low)**

**Probability**: Medium (2) - Every event processed incurs TenantContext setup/teardown cost.

**Impact**: Low (1) - Expected overhead <5ms per event is acceptable for EAF's performance targets (p95 <200ms total).

**Mitigation**:
- Benchmark interceptor with Micrometer `Timer` around `handle()` method
- Target <2ms overhead per event
- Cache tenant validation results in Redis (TTL 5 minutes)
- Monitor p95 latency with Grafana dashboard

**Testing Focus**:
- Performance test: Process 1000 events sequentially
- Measure p95 latency, assert <5ms overhead
- Compare baseline (no interceptor) vs. instrumented
