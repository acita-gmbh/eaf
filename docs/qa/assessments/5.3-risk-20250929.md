## High Risks (Score 6)

### TECH-001: Trace Context Loss in Async Event Processing

**Score: 6 (High Risk)**
**Probability**: Medium (2) - Async context propagation is notoriously complex
**Impact**: High (3) - Lost trace context breaks distributed tracing value proposition

**Description**:
OpenTelemetry trace context (trace_id, span_id) may not propagate correctly through Axon's asynchronous event handlers, especially across thread boundaries or coroutine contexts. This would result in fragmented traces that cannot be correlated end-to-end.

**Affected Components**:
- `TracingEventInterceptor`
- `TenantEventMessageInterceptor` (Story 4.4 integration point)
- Axon event processors (async processing groups)

**Root Cause Analysis**:
- ThreadLocal context doesn't automatically transfer to new threads
- Axon event metadata requires manual trace context injection/extraction
- Micrometer Context Propagation must bridge OpenTelemetry and Axon

**Mitigation Strategy** (Preventive):

**Actions**:
1. Leverage existing Micrometer Context Propagation infrastructure from Story 4.4
2. Implement TracingEventInterceptor with explicit context restoration logic (code example provided in story)
3. Register tracing interceptors BEFORE tenant validation interceptors (order matters)
4. Add comprehensive integration test (Subtask 5.2) validating trace flow: command→event→projection

**Testing Requirements**:
- **P0 Integration Test**: `AxonTracePropagationIntegrationTest` dispatches command, publishes event, verifies trace_id preservation across async boundary
- **Multi-tenant scenario**: Verify trace context and tenant context both propagate correctly
- **Edge case**: Test trace propagation when no parent span exists (should create new root span)

**Residual Risk**: Low - Micrometer Context Propagation is proven technology; pattern documented in Story 4.4

**Owner**: dev
**Timeline**: Must validate before AC2 sign-off

### SEC-001: Tenant Context Leakage in Span Attributes

**Score: 6 (High Risk)**
**Probability**: Medium (2) - Easy to accidentally tag wrong tenant in infrastructure code
**Impact**: High (3) - Cross-tenant information disclosure via trace metadata

**Description**:
TenantAwareTracingFilter or TracingCommandInterceptor could inadvertently tag spans with incorrect tenant_id, exposing tenant-specific trace data to wrong tenants in observability backend (Jaeger UI).

**Affected Components**:
- `TenantAwareTracingFilter`
- `TracingCommandInterceptor`
- Jaeger/Tempo backend (tenant_id filtering)

**Root Cause Analysis**:
- Infrastructure observability uses graceful fallback (`?: "system"`)
- Risk of using wrong TenantContext API (current() vs getCurrentTenantId())
- Span attributes are immutable after creation - hard to fix post-tagging

**Mitigation Strategy** (Preventive):

**Actions**:
1. Follow established tenant tagging pattern from Story 5.2 CustomMetrics (graceful fallback for infrastructure observability)
2. Use `tenantContext.current() ?: "system"` for span attributes (never fail-closed in observability)
3. Implement Subtask 5.4 validation test: multi-tenant scenario asserts correct tenant_id in spans
4. Code review checklist: Verify no hardcoded tenant values, no cross-tenant span correlation

**Testing Requirements**:
- **P0 Security Test**: Create spans for tenant-a and tenant-b in sequence, verify tenant_id attributes don't cross-contaminate
- **Negative test**: Verify "system" tenant used when TenantContext is empty (background jobs)
- **Integration test**: Query Jaeger API, filter by tenant_id=tenant-a, assert no tenant-b trace data appears

**Residual Risk**: Low - Pattern proven in Story 5.2, explicit test coverage planned

**Owner**: dev + qa
**Timeline**: Must validate before production deployment

## Medium Risks (Score 4)

### TECH-002: OTLP Exporter Connection Failures

**Score: 4 (Medium Risk)**
**Probability**: Medium (2) - Collector unavailability is common in distributed systems
**Impact**: Medium (2) - Traces lost but application continues functioning

**Description**:
If OTLP collector (Jaeger) is unavailable, span data accumulates in memory until buffer limits reached, potentially causing memory pressure or dropped traces.

**Mitigation**:
- Configure batch span processor with limits: `otel.bsp.max.queue.size=2048`, `otel.bsp.max.export.batch.size=512` (documented in story)
- Integration test verifies graceful degradation when collector unavailable
- Troubleshooting section documents "Traces not appearing" diagnostic steps

**Testing**: Subtask 5.1 includes OTLP exporter configuration validation

**Residual Risk**: Very Low - Documented mitigation with reasonable defaults
