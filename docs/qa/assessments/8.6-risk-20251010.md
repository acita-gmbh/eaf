## Critical Risks Requiring Immediate Attention

### 1. TECH-002: CI Pipeline Timeout (45-minute limit)

**Score: 6 (HIGH RISK)**

**Probability**: Medium (2) - Mutation testing generates hundreds of mutants for security module
- framework/security has 10-layer JWT validation (complex conditional logic)
- framework/cqrs has tenant event interceptors (multiple code paths)
- STRONGER mutators = more mutations to test
- Multi-threading helps but not guaranteed to finish in 45 minutes

**Impact**: High (3) - Pipeline failure blocks development
- PR merges blocked if mutation-testing CI job fails
- Developer frustration with slow feedback loops
- Potential to disable mutation testing entirely if unreliable

**Mitigation**:
1. **Phase 1: Baseline execution with timeout monitoring**
   - Run `./gradlew :framework:security:pitest` locally first
   - Measure actual execution time before enabling in CI
   - Document baseline metrics in Task 2 subtask 2.3

2. **Pre-emptive timeout handling**:
   ```kotlin
   // QualityGatesConventionPlugin.kt adjustment if needed
   pitest {
       threads = Runtime.getRuntime().availableProcessors()
       timeoutConstant = 5000 // Increase from default if needed
       excludedMethods = ["toString", "hashCode", "equals"] // Skip utility methods
   }
   ```

3. **Incremental rollout strategy**:
   - AC16: Enable framework/security only initially
   - Validate CI execution time < 30 minutes
   - Then enable framework/cqrs
   - Keep framework/core optional if low business logic

4. **Fallback plan**:
   - Make mutation testing optional in CI (`continue-on-error: true`)
   - Require local pitest execution before PR approval
   - Consider scheduled nightly mutation runs instead of per-PR

**Testing Focus**:
- Task 5 subtask 5.5: Validate 45-minute timeout is sufficient
- Monitor CI execution time in first 3 PR runs
- Set alert threshold at 40 minutes (safety margin)

**Residual Risk**: LOW - With multi-threading and scoped modules, should execute in 20-30 minutes

#### TECH-003: Equivalent Mutants (Unfixable Survivors)

**Score: 4 (MEDIUM RISK)**

**Probability**: Medium (2) - Industry standard ~5-10% equivalent mutants
- Some mutations are semantically equivalent to original code
- Examples: `i++` vs `++i` in certain contexts, redundant null checks
- STRONGER mutators increase equivalent mutant likelihood

**Impact**: Medium (2) - Reduces mutation coverage metrics
- Could prevent reaching 80% threshold
- Requires manual review and documentation
- Potential for debate: "Is this mutant really equivalent?"

**Mitigation**:
1. **Documentation protocol** (AC15 & Task 3 subtask 3.6):
   - Document all accepted surviving mutants with rationale
   - Example: "Mutant survives because [technical reason], accepted because [business justification]"
   - Store in `framework/security/docs/mutation-coverage.md`

2. **Threshold adjustment**:
   - If >5% equivalent mutants, consider 75% threshold
   - Requires architectural review approval

3. **Pitest exclusions**:
   ```kotlin
   pitest {
       excludedMethods = ["equals", "hashCode", "toString"]
       avoidCallsTo = ["kotlin.jvm.internal", "java.util.Objects"]
   }
   ```

**Testing Focus**:
- AC9: Identify surviving mutants by category
- AC10: Assess effort to reach 80% (includes equivalent mutant analysis)
- AC15: Document accepted surviving mutants

**Residual Risk**: LOW-MEDIUM - Manageable with documentation

#### PERF-001: Local Development Workflow Impact

**Score: 4 (MEDIUM RISK)**

**Probability**: Medium (2) - Pitest adds time to `./gradlew check`
- Mutation testing takes 5-10x longer than unit tests
- Developers run `./gradlew check` before committing
- Estimated 3-5 minutes per module

**Impact**: Medium (2) - Developer productivity
- Slower feedback loops
- Developers may skip local quality gates
- Could reduce TDD adoption

**Mitigation**:
1. **Fast feedback workflow** (documented in AC23):
   - Regular development: `./gradlew jvmKotest` (fast tests only)
   - Pre-commit: `./gradlew check` (includes pitest)
   - Optional: Skip pitest locally, rely on CI

2. **Make pitest optional locally**:
   ```bash
   # Fast local development
   ./gradlew check -x pitest

   # Full quality gate (before PR)
   ./gradlew check
   ```

3. **Documentation** (AC24):
   - Development workflow guide with "fast" vs "thorough" commands
   - Pre-commit hook guidance (optional pitest or skip rationale)

**Testing Focus**:
- AC23: Pre-commit hooks optionally run pitest (or document skip rationale)
- AC25: Document process for addressing future surviving mutants

**Residual Risk**: LOW - Optional execution pattern available
