## High Risks (Score 6) - Monitoring Required

### **TECH-004: Frontend-Backend Integration Failures**

**Score: 6 (High Risk - Orange)**

**Probability:** Medium (2)
- **Reasoning:** Common in micro-frontend first integration - authentication flows, CORS policies, data format mismatches
- **Likelihood:** 30-50% - Framework shell is tested, but first consumer app integration has typical risks

**Impact:** High (3)
- **Potential Consequences:**
  - Consumer app won't authenticate users (blocking)
  - API calls fail due to CORS issues (blocking)
  - Data format mismatches cause UI errors (degraded UX)
  - Tenant context not propagated correctly (security issue)
  - Widget CRUD operations don't work (blocks Epic 9 validation)
- **Business Impact:** High - Blocks Epic 9 (Licensing Server MVP validation)
- **User Impact:** High - Application unusable without working authentication and CRUD

**Affected Components:**
- `apps/admin/src/App.tsx` (Keycloak integration - lines 293-312)
- `apps/admin/vite.config.ts` (proxy configuration - lines 317-337)
- Backend `WidgetController` REST API
- Keycloak OIDC authentication flow
- JWT token propagation to backend
- Tenant context in API headers

**Detection Method:**
- Phase 4 testing: AC 16-19 (Keycloak Integration)
- Phase 5 testing: AC 20-27 (Widget CRUD Validation)
- Task 4: 7 subtasks validate OIDC flow
- Task 5: 11 subtasks validate CRUD operations

**Mitigation Strategy:**

**Preventive:**
- ✅ **Contract-first approach:** Use existing OpenAPI spec from WidgetController (Epic 2 deliverable)
- ✅ **Vite proxy configuration:** Forwards `/api` requests to `localhost:8080` (avoids CORS in development)
- ✅ **Framework shell provides auth logic:** Reuses Epic 3 Keycloak integration (tested infrastructure)
- ✅ **Troubleshooting guide:** 4 common issues with solutions (lines 558-574):
  - Issue 2: CORS errors → Vite proxy solution
  - Issue 3: Keycloak redirect loop → Configuration checklist
  - Issue 4: TypeScript build failures → Type export verification

**Detective:**
- ✅ **Comprehensive Phase 4-5 manual testing:** 16 acceptance criteria validate integration
- ✅ **3-terminal validation workflow:** Infrastructure + Backend + Frontend (lines 499-515)
- ✅ **Tenant isolation test:** Subtask 5.11 (create 2 tenants, verify separation)
- ✅ **Network inspection:** AC 18 (verify JWT in Authorization header)

**Corrective:**
- ✅ **Troubleshooting documentation:** Solutions for CORS, authentication, TypeScript issues
- ✅ **Framework shell fallback:** Known-working authentication logic if consumer app config fails

**Testing Requirements:**

**P0 Tests (Phases 4-5):**
- AC 16: Keycloak connection configuration
- AC 17: Complete OIDC flow (redirect → login → token → refresh → logout)
- AC 18: JWT headers in all API requests
- AC 19: Unauthenticated redirect
- AC 20-26: Full Widget CRUD operations
- AC 27: **Tenant isolation validation** (security-critical)

**P1 Tests (Phase 6):**
- AC 28-32: Error handling and user experience

**Validation Approach:**
```bash
# 3-terminal manual testing workflow
# Terminal 1: ./scripts/init-dev.sh (Keycloak)
# Terminal 2: ./gradlew :products:widget-demo:bootRun (Backend)
# Terminal 3: npm run dev (Frontend in apps/admin/)
# Browser: http://localhost:5173 → Test all CRUD operations
```

**Residual Risk:** Low - Comprehensive mitigation with troubleshooting support

**Owner:** Development team

**Timeline:** Phases 4-5 (Tasks 4-5) during implementation

**Recommended Mitigation Enhancement:**
- Consider contract testing (Pact) for API integration validation
- Consider adding automated E2E tests (Playwright) for regression prevention

  - Subtask 7.10: Focus indicator verification
  - Subtask 7.11: Form label association validation
- ✅ Validation tools explicitly specified (lines 484-487)
- ✅ WCAG AA standards documented (lines 459-483):

**Testing Requirements (Phase 7 - AC 33-37):**

- AC 34: Screen reader compatibility (ARIA labels, announcements, semantic HTML)
- AC 35: Color contrast WCAG AA (4.5:1 normal, 3:1 large)
