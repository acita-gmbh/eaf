## Test Strategy Overview

- **Total Test Scenarios:** 41 (one per Acceptance Criterion)
- **Manual Tests:** 21 (51% - setup, CLI, configuration validation)
- **Integration Tests:** 7 (17% - Gradle, error handling, proxy)
- **E2E Tests:** 13 (32% - authentication, CRUD, accessibility)
- **Priority Distribution:** P0: 23 scenarios (56%), P1: 13 scenarios (32%), P2: 5 scenarios (12%)

**Test Philosophy:** Manual-first with selective automation opportunities

**Why Manual-Heavy:** Frontend integration story requires manual validation for:
- User experience quality
- Visual correctness
- Accessibility compliance (WCAG AA)
- Authentication flow UX
- Form interactions and validation

## Mock/Stub Strategy

**No Mocking for Integration Story:** This story validates real integration between:
- React-Admin frontend
- Spring Boot backend
- Keycloak authentication
- PostgreSQL database (via backend)

**Why No Mocks:** Integration validation requires real dependencies to catch:
- CORS issues (Vite proxy integration)
- Authentication flow problems (Keycloak OIDC)
- Data format mismatches (React-Admin ↔ Spring Boot)
- Tenant context propagation

**Testing Philosophy:** Use real dependencies (3-terminal setup) to validate complete integration.

## Edge Cases and Error Scenarios

**Covered by Story:**

### Authentication Edge Cases (Phase 4):
1. **Unauthenticated Access** (AC 19):
   - User navigates to app without token → Redirect to Keycloak login
2. **Token Refresh** (AC 17):
   - Token expires during session → Automatic refresh → Session continues
3. **Logout** (AC 17):
   - User logs out → Token cleared → Redirect to login

### CRUD Edge Cases (Phase 5):
4. **Empty List** (AC 20):
   - No widgets exist → Empty state displayed with "Create Widget" prompt
5. **Pagination** (AC 20):
   - >10 widgets → Pagination controls appear
6. **Filter/Search No Results** (AC 21-22):
   - Filter/search returns no matches → Empty state with "No results" message
7. **Form Validation** (AC 24):
   - Invalid widget data → Validation errors highlighted → Submission prevented

### Error Handling Edge Cases (Phase 6):
8. **Backend API Error** (AC 28-29):
   - Backend returns 500 error → User-friendly message displayed (not stack trace)
   - RFC 7807 ProblemDetail parsed correctly
9. **Network Failure** (AC 30):
   - API call times out → Loading state → Error message
10. **Optimistic Update Failure** (AC 25):
    - Edit succeeds in UI → Backend fails → Rollback to previous state

### Accessibility Edge Cases (Phase 7):
11. **Keyboard-Only User** (AC 33):
    - All CRUD operations completable without mouse
    - No keyboard traps
12. **Screen Reader User** (AC 34):
    - All content announced logically
    - Form fields labeled correctly
    - Success/error states announced

### Tenant Isolation Edge Cases (AC 27):
13. **Cross-Tenant Access Attempt**:
    - Tenant-a user tries to access tenant-b widget → 403 Forbidden or filtered out
14. **Tenant Context Missing**:
    - JWT missing tenant claim → Authentication failure

**Additional Edge Cases to Consider:**

15. **Vite Proxy Failure** (Not explicitly covered):
    - Scenario: Vite proxy can't reach backend → API calls fail
    - Recommendation: Add validation that proxy error messages are clear

16. **Workspace Dependency Version Mismatch** (Not explicitly covered):
    - Scenario: Framework shell version incompatible with consumer app
    - Recommendation: Add version compatibility check in Phase 2

## Test Automation Recommendations

**Current Story:** 100% manual validation (appropriate for first implementation)

**Post-MVP Automation Opportunities:**

### High-Value E2E Automation (8 Critical Flows):

**Tool Recommendation:** Playwright or Cypress

1. **Login Flow** (Covers AC 17):
   ```typescript
   test('complete authentication flow', async ({ page }) => {
     await page.goto('http://localhost:5173');
     // Should redirect to Keycloak
     await expect(page).toHaveURL(/keycloak/);
     await page.fill('[name="username"]', 'test@tenant-a.com');
     await page.fill('[name="password"]', 'password');
     await page.click('[type="submit"]');
     // Should return to app with token
     await expect(page).toHaveURL('http://localhost:5173');
     // Verify JWT token exists
   });
   ```

2. **Create Widget Flow** (Covers AC 24):
   - Navigate to Widgets → Click Create → Fill form → Submit → Verify in list

3. **Edit Widget Flow** (Covers AC 25):
   - Navigate to widget → Click Edit → Modify fields → Save → Verify update

4. **Tenant Isolation Flow** (Covers AC 27):
   - Login as tenant-a → Create widget → Logout → Login as tenant-b → Verify widget not visible

5. **Error Handling Flow** (Covers AC 28-29):
   - Trigger backend error → Verify user-friendly message displayed

6. **Form Validation Flow** (Covers AC 24, 32):
   - Submit invalid form → Verify validation errors highlighted

7. **Search & Filter Flow** (Covers AC 21-22):
   - Apply filters → Verify results → Search → Verify search results

8. **Keyboard Navigation Flow** (Covers AC 33):
   - Complete CRUD operations using only keyboard (Tab, Enter, Space, Escape)

**Automation Priority:** P1 (Post-MVP Enhancement)

**Rationale:** Manual testing sufficient for initial implementation. Automation prevents regression as application evolves.

**Estimated Automation Effort:** 16-24 hours to create 8 Playwright E2E tests

## Test Reliability Considerations

**Flakiness Risk:** Low-Medium

**Potential Flaky Tests:**
- Authentication tests (Keycloak redirect timing)
- API integration tests (network timing)
- Optimistic update tests (race conditions)

**Stability Measures:**
- Use explicit waits for Keycloak redirects
- Verify backend is running before frontend tests
- Wait for loading states to disappear before assertions
- Use React-Admin's built-in loading indicators for timing

**Manual Testing Advantage:** Human testers adapt to timing variations naturally, reducing flakiness concerns
