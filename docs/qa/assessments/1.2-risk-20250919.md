## Detailed Risk Register

### OPS-001 – Testcontainers infrastructure unavailable or unstable
- **Category**: Operational
- **Description**: Integration tests depend on PostgreSQL, Redis, and Keycloak containers. Missing Docker resources, incompatible host environments, or CI runners without privilege can cause repeated failures.
- **Affected Components**: `shared/testing` container bootstrap, Gradle `integrationTest` task, CI pipeline.
- **Probability**: Medium (2)
- **Impact**: High (3) – Blocks Constitutional TDD enforcement and CI gates.
- **Score**: 6 (High)
- **Detection**: Monitoring build logs for container start failures, Gradle task outcomes.
- **Mitigation**:
  - Validate Docker availability in onboarding script (`scripts/init-dev.sh`) and CI runners.
  - Provide fallback documentation for running Testcontainers in rootless/remote environments.
  - Add health checks in Gradle TestKit suite ensuring bootstrap object initializes successfully.
- **Residual Risk**: Medium – relies on infrastructure discipline.

### TECH-001 – Catalog drift detection fails
- **Category**: Technical
- **Description**: Convention plugins rely on version catalog parsing. If guardrails regress, modules could hard-code versions, causing silent drift from mandated tooling.
- **Affected Components**: `build-logic` catalog parser, quality/testing plugins.
- **Probability**: Medium (2)
- **Impact**: Medium (2) – Inconsistent lint/static analysis results; potential policy violations.
- **Score**: 4 (Medium)
- **Detection**: Gradle TestKit coverage, manual review of plugin outputs, Konsist checks on build scripts.
- **Mitigation**:
  - Expand TestKit tests to mutate catalog values and assert mismatch detection.
  - Add CI task that greps for disallowed version strings outside catalog references.
- **Residual Risk**: Low-Medium – mitigated when tests are implemented.

### OPS-002 – Full `check` lifecycle slows builds
- **Category**: Operational
- **Description**: Running ktlint, Detekt, Konsist, Testcontainers integration tests, and Pitest every `check` can significantly extend feedback loops and consume CI minutes.
- **Affected Components**: Gradle `check` task, CI pipelines, developer workflows.
- **Probability**: Medium (2)
- **Impact**: Medium (2) – May cause pipeline timeouts or discourage frequent local runs.
- **Score**: 4 (Medium)
- **Detection**: Build scan analytics, CI duration reports, developer feedback.
- **Mitigation**:
  - Capture baseline runtime metrics (already documented) and monitor for regressions.
  - Allow targeted task execution for local fast loops while keeping CI gates intact (documented in Dev Notes/testing section).
  - Consider parallelizing quality tasks in CI runners.
- **Residual Risk**: Medium – ongoing monitoring required.

### TECH-002 – False positives from quality tools
- **Category**: Technical
- **Description**: Aggressive static analysis or mutation thresholds might fail legitimate builds, reducing developer trust in gates.
- **Affected Components**: ktlint, Detekt configs, Pitest thresholds.
- **Probability**: Low (1)
- **Impact**: Medium (2) – Productivity hits; potential bypass attempts.
- **Score**: 2 (Low)
- **Detection**: Track recurring suppressions, monitor QA gate feedback, review mutation reports.
- **Mitigation**:
  - Provide documented suppression workflow and review process.
  - Maintain shared Detekt configuration tuned for Kotlin 2.0.10 to minimize noise.
  - Regularly inspect mutation reports for flaky mutants and adjust excludes cautiously.
- **Residual Risk**: Low.

## Risk-Based Testing Strategy

1. **Priority 1 – OPS-001 (High)**
   - Automated smoke test to ensure Testcontainers bootstrap class starts each container and surfaces failures fast.
   - Add CI pre-flight check verifying Docker daemon availability and minimum resources.
   - Run `./gradlew integrationTest` in an isolated environment twice to confirm reuse behaviour and stability.

2. **Priority 2 – TECH-001 & OPS-002 (Medium)**
   - Gradle TestKit suite covering catalog drift detection; include negative tests where plugin versions are hard-coded.
   - Monitor build duration trends via CI metrics; set warning thresholds and alerting.
   - Execute `./gradlew check --scan` to verify tasks are parallelizable and identify bottlenecks.

3. **Priority 3 – TECH-002 (Low)**
   - Collect baseline mutation/Detekt reports post-implementation and adjust configs only after review.
   - Incorporate developer feedback loop to capture false positives quickly.

## Recommended Mitigations & Actions

- **Infrastructure Readiness (OPS-001)**
  - Update onboarding docs with Docker prerequisites and troubleshooting for Testcontainers.
  - Add CI job to cache Docker images for PostgreSQL, Redis, Keycloak to reduce flake rate.

- **Catalog Integrity (TECH-001)**
  - Implement automated lint rule (Konsist or custom Gradle task) that forbids explicit version literals in module builds.

- **Build Performance (OPS-002)**
  - Maintain build performance log in story Dev Notes or follow-up doc; flag any regression >20% vs documented baseline.
  - Encourage developers to use task-specific runs (`test`, `integrationTest`) locally while CI enforces full `check`.

- **Quality Tool Tuning (TECH-002)**
  - Establish process for triaging Detekt/Pitest noise via QA office hours.

## Monitoring & Alerting

- Hook Gradle Enterprise build scans or CI telemetry to alert when `check` exceeds baseline duration by >25%.
- Configure Testcontainers reuse metrics/logs to detect repeated cold starts.
- Track frequency of catalog drift detections; zero detections over time should trigger periodic spot audits.
