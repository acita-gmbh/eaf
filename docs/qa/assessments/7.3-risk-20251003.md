### High Risks (Score 6)

#### SEC-003: Template Injection via Aggregate Name

**Category**: Security
**Probability**: Medium (2) - Developer tools may be targeted by malicious developers
**Impact**: High (3) - Could generate unsafe code, inject malicious logic, or corrupt codebase
**Score**: 6 (High Risk - Orange)

**Description**:
Malicious aggregate names could exploit template rendering to inject code or commands into 16 generated files. Example attack vectors:
- `<script>alert('xss')</script>` → Template injection
- `Widget; rm -rf /` → Command injection
- `../../etc/passwd` → Path traversal

**Affected Components**:
- All 16 Mustache templates
- AggregateCommand input validation
- TemplateEngine rendering
- File system operations

**Mitigation Strategy** (3-Layer Defense-in-Depth):

**Layer 1: Input Validation**
```kotlin
val aggregatePattern = Regex("^[A-Z][A-Za-z0-9]*$")
require(aggregateName.matches(aggregatePattern)) {
    "Invalid aggregate name: '$aggregateName'. Must be PascalCase alphanumeric."
}

require(!aggregateName.contains("..") && !moduleName.contains("..")) {
    "Security violation: path traversal detected"
}

require(!aggregateName.contains("<") && !aggregateName.contains(">")) {
    "Security violation: potential injection detected"
}
```

**Layer 2: Template Escaping**
- Mustache HTML-escaping validated in Story 7.1
- Auto-escapes special characters in template variables

**Layer 3: File System Validation**
- Pre-generation checks for existing files
- Path sanitization before file creation
- Defensive error handling

**Test Coverage**: 7.3-UNIT-007 (P0 Security Critical)
- Attack Scenario 1: Path traversal `../../etc/passwd`
- Attack Scenario 2: Template injection `<script>alert('xss')</script>`
- Attack Scenario 3: Command injection `Widget; rm -rf /`
- Attack Scenario 4: Invalid PascalCase `widgetAggregate`, `Widget_Name`
- Attack Scenario 5: Special characters `Widget$Name`, `Widget@Entity`

**Residual Risk**: **LOW** - Triple-layer validation significantly reduces attack surface

**Owner**: Developer Agent (James)
**Timeline**: Implement security test FIRST per Story 7.2 security-first TDD pattern

[Source: Story 7.2 SEC-002 pattern, architecture/security.md]

#### TECH-008: Field Parser Type Safety

**Category**: Technical
**Probability**: Medium (2) - CLI string parsing is error-prone without strict validation
**Impact**: Medium (2) - Invalid type mappings generate code that fails compilation
**Score**: 4 (Medium Risk - Yellow)

**Description**:
Field parser converts CLI strings (`name:String, value:BigDecimal`) to Kotlin types. Risk of invalid type mappings:
- Unsupported types: `name:InvalidType` → runtime error
- Typos: `value:BigDecima` → generation fails
- Complex types: `items:List<String>` → unsupported

**Affected Components**:
- AggregateGenerator field parser (Task 3.4)
- Type mapping logic (Task 10.3)
- All generated files using field types
- FieldSpec data class

**Mitigation Strategy**:

**Type Whitelist**:
```kotlin
val SUPPORTED_TYPES = setOf(
    "String", "Int", "Long",
    "BigDecimal", "Boolean", "Instant"
)

fun parseFieldType(typeStr: String): Either<GeneratorError, String> =
    if (SUPPORTED_TYPES.contains(typeStr)) {
        typeStr.right()
    } else {
        GeneratorError.UnsupportedType(
            type = typeStr,
            supportedTypes = SUPPORTED_TYPES
        ).left()
    }
```

**Clear Error Messages**:
- List supported types in error message
- Show example field syntax
- Validate during parsing, not rendering

**Test Coverage**:
- 7.3-UNIT-004: Valid field syntax parsing
- 7.3-UNIT-005: Invalid type error handling
- 7.3-UNIT-009: Type mapping validation

**Residual Risk**: **LOW** - Whitelist prevents invalid types, clear errors guide users

**Owner**: Developer Agent (James)
**Timeline**: During generator implementation (Tasks 3, 10)

[Source: Field parsing requirement, type safety concerns]

#### TECH-010: Custom Validation Pattern Security (ReDoS)

**Category**: Technical
**Probability**: Low (1) - Enhancement feature with validation
**Impact**: High (3) - Malicious regex could cause ReDoS attacks in generated code
**Score**: 3 (Low Risk - Green)

**Description**:
Custom validation patterns from `--validation` option could include malicious regex causing catastrophic backtracking:
- Example: `(a+)+b` against `aaaaaaaaaaaaaaaaaac` → exponential time

**Mitigation Strategy**:

**Recommended Safeguards**:
```kotlin
fun validateRegexPattern(pattern: String): Either<GeneratorError, Regex> {
    // 1. Reject known dangerous patterns
    val dangerousPatterns = listOf("(.*)*", "(a+)+", "(a*)*")
    if (dangerousPatterns.any { pattern.contains(it) }) {
        return GeneratorError.UnsafeRegexPattern(pattern).left()
    }

    // 2. Test compilation
    return try {
        val regex = Regex(pattern)
        // 3. Optional: Test against sample input with timeout
        regex.right()
    } catch (ex: PatternSyntaxException) {
        GeneratorError.InvalidRegexPattern(pattern, ex.message).left()
    }
}
```

**Test Coverage**: 7.3-UNIT-012

**Residual Risk**: **LOW** - Validation prevents most attacks, timeout protection recommended

**Owner**: Developer Agent (James)
**Timeline**: During validation parser implementation (Task 3.5)

[Source: Custom validation enhancement, ReDoS security considerations]
