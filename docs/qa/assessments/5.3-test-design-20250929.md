#### 5.3-INT-001: OpenTelemetry SDK Initialization

**Level**: Integration
**Priority**: P0 (AC1 validation)
**File**: `framework/observability/src/integration-test/kotlin/.../TracingConfigurationIntegrationTest.kt`

**Test Description**:
Verify OpenTelemetry SDK initializes with correct configuration when Spring Boot application starts.

**Given**:
- Spring Boot application with opentelemetry-spring-boot-starter dependency
- TracingConfiguration auto-configuration active
- OTLP exporter endpoint configured to `http://localhost:4318`

**When**:
- Application context loads

**Then**:
- OpenTelemetry SDK is initialized
- Tracer bean is available in Spring context
- OTLP exporter is configured with correct endpoint
- Resource attributes include `service.name=eaf-service`

**Mitigates Risk**: TECH-002 (configuration validation)

#### 5.3-INT-005: Axon Command Trace Context Injection

**Level**: Integration
**Priority**: P0 (AC2 validation, TECH-001 mitigation)
**File**: `framework/cqrs/src/integration-test/kotlin/.../AxonTracePropagationIntegrationTest.kt`

**Test Description**:
Verify TracingCommandInterceptor injects active trace context into Axon command metadata before dispatch.

**Given**:
- Active OpenTelemetry span with known trace_id
- CreateWidgetCommand ready to dispatch

**When**:
- CommandGateway.sendAndWait() called within span context

**Then**:
- Command metadata contains keys: `trace_id`, `span_id`, `trace_flags`
- trace_id matches active span's trace ID
- Command handler receives command with trace metadata

**Mitigates Risk**: TECH-001 (trace propagation)

#### 5.3-INT-006: Axon Event Async Context Restoration

**Level**: Integration
**Priority**: P0 (AC2 validation, TECH-001 mitigation)
**File**: `framework/cqrs/src/integration-test/kotlin/.../AxonTracePropagationIntegrationTest.kt`

**Test Description**:
Verify TracingEventInterceptor restores trace context when async event handler processes event.

**Given**:

**Mitigates Risk**: TECH-001 (async propagation) - **HIGHEST RISK**

#### 5.3-INT-007: Parent-Child Span Relationships

**Level**: Integration
**Priority**: P0 (AC2 validation)
**File**: `framework/cqrs/src/integration-test/kotlin/.../AxonTracePropagationIntegrationTest.kt`

**Test Description**:
Verify spans created during command→event flow maintain correct parent-child hierarchy.

**Given**:
- Root span for HTTP request
- Command dispatch within request span
- Event publishing during command handling

**When**:
- Complete flow executes: HTTP → Command → Event → Projection

**Then**:
- Command span's parent is HTTP request span
- Event span's parent is command span
- Projection handler span's parent is event span
- All spans share same trace_id

**Mitigates Risk**: TECH-001 (trace continuity)

## Test Level Justification

### Why Integration-Heavy Strategy?

**Framework infrastructure stories benefit from integration tests because**:

1. **Real dependency interaction**: OpenTelemetry SDK behavior differs in Spring Boot auto-configuration vs isolated unit tests
2. **Context propagation validation**: ThreadLocal and async propagation require real thread pools (Testcontainers)
3. **Configuration accuracy**: Spring Boot properties binding needs full application context
4. **Constitutional TDD alignment**: Story explicitly requires Testcontainers integration-first approach

**Unit tests limited to**:
- Pure logic in interceptors (metadata injection parsing)
- Configuration bean creation logic
- MDC enrichment logic (can be tested with mock Span)

**E2E tests minimal because**:
- Integration tests with Testcontainers already use real PostgreSQL, Axon, OpenTelemetry SDK
- E2E adds value only for workflow validation (5.3-E2E-001) and visual trace inspection (5.3-E2E-002)

## Test Maintenance Considerations

### Testcontainers Startup Time Optimization

**Current baseline** (from Story 4.6 lessons):
- PostgreSQL Testcontainer: ~8 seconds first start, reused across tests
- Companion object pattern: Start containers once per test class

**Recommendation**: Reuse Testcontainers infrastructure from Stories 5.1, 5.2 - no additional container overhead.

### Test Data Strategy

**Nullable Pattern** for unit tests:
- NullableTracer returns no-op spans (fast, no SDK dependency)
- Story troubleshooting section suggests this approach

**Real OpenTelemetry SDK** for integration tests:
- Use in-memory span exporter for test inspection
- No external Jaeger dependency in CI (optional for local dev)
