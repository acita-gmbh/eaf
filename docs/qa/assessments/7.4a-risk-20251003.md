## High-Priority Risks (Score 6)

### SEC-001: JWT Storage in localStorage (XSS Vulnerability)

**Score: 6 (High)** — Probability: Medium (2) × Impact: High (3)

**Description**: Storing JWT access and refresh tokens in browser `localStorage` creates an XSS attack surface. If an attacker injects malicious JavaScript (e.g., via unescaped user content, compromised dependency), they can steal tokens and impersonate users.

**Affected Components**:
- `framework/admin-shell/src/providers/authProvider.ts` (stores/retrieves tokens)
- `framework/admin-shell/src/providers/dataProvider.ts` (injects tokens in headers)

**Probability: Medium (2)**:
- React-Admin's declarative model reduces direct DOM manipulation (lower XSS risk)
- Story mandates DOMPurify for user content sanitization
- Backend enforces CSP headers
- Modern browsers have built-in XSS protections
- However, zero-day XSS vulnerabilities in dependencies or subtle coding errors remain possible

**Impact: High (3)**:
- Stolen JWT = full user impersonation
- Access to all tenant data user has permissions for
- Potential privilege escalation if roles are elevated
- Session hijacking persists until token expiration (15 min access, 8 hr refresh)

**Mitigation Strategies** (from Story Security Considerations):

**MVP (Acceptable for Initial Release)**:
1. **Short-lived tokens**: Access token expires in 15 minutes (limits exposure window)
2. **Automatic token refresh**: authProvider refreshes access_token when <5 min remaining (reduces stale token usage)
3. **Backend CSP headers**: `Content-Security-Policy: script-src 'self'` to prevent inline script injection
4. **Input sanitization**: DOMPurify for all user-generated content before rendering (Task 2.5 code example)
5. **Security advisory**: README.md explicitly documents localStorage risk and recommends httpOnly cookies for production

**Post-MVP (Future Enhancement)**:
1. Use httpOnly cookies for token storage (browser prevents JavaScript access)
2. Implement BFF (Backend-for-Frontend) pattern for token management
3. Server-side session management

**Testing Requirements**:
- **7.4a-SEC-001**: Test XSS attack prevention (malicious input sanitization)
- **7.4a-SEC-002**: Verify tokens not vulnerable to CSRF
- **7.4a-SEC-003**: Test token refresh flow edge cases (expiration, network errors)

**Residual Risk**: **Low** — With MVP mitigations, the attack window is narrow (15 min tokens + sanitization + CSP). An attacker would need to both inject XSS AND execute within token lifetime.

**Owner**: Dev (Frontend Lead)
**Timeline**: Core mitigation (sanitization + refresh) in initial implementation (Task 2, Task 3). CSP headers in backend deployment config.

### SEC-002: Tenant Context Propagation Failure

**Score: 6 (High)** — Probability: Medium (2) × Impact: High (3)

**Description**: If `dataProvider` fails to extract `tenant_id` from JWT or inject `X-Tenant-ID` header in API requests, tenant isolation is compromised. Users could access data from wrong tenants or cause data corruption.

**Affected Components**:
- `framework/admin-shell/src/providers/dataProvider.ts` (tenant extraction + header injection)

**Probability: Medium (2)**:
- Depends on correct JWT parsing (jwt-decode library)
- Requires consistent header injection across all API methods (GET/POST/PUT/DELETE)
- Could be broken by JWT format changes, library updates, or coding errors in httpClient wrapper

**Impact: High (3)**:
- **Tenant isolation breach** (CWE-566: Authorization Bypass Through User-Controlled Key)
- Cross-tenant data exposure (confidentiality violation)
- Data corruption if wrong tenant context used for writes (integrity violation)
- Regulatory compliance violations (GDPR, HIPAA data segregation)

**Mitigation Strategies**:

1. **Fail-closed design** (Task 2.3): `dataProvider` checks tenant_id exists in JWT *before* every request. Missing tenant → throw error, reject request.
   ```typescript
   const tenantId = extractTenantFromJWT(token);
   if (!tenantId) throw new Error('Tenant context missing - access denied');
   options.headers.set('X-Tenant-ID', tenantId);
   ```

2. **Comprehensive testing** (7.4a-UNIT-003):
   - Test tenant extraction from valid JWT
   - Test rejection when tenant_id missing
   - Test header injection in all CRUD operations
   - Test multi-tenant scenarios (switch tenants, verify isolation)

3. **Backend defense-in-depth** (from `architecture/security.md`):
   - **Layer 1**: Request filter validates `X-Tenant-ID` header
   - **Layer 2**: Service boundary validates tenant context in commands
   - **Layer 3**: Database RLS enforces tenant filtering (PostgreSQL Row-Level Security)
   - Frontend failure caught by backend before data access

4. **Logging**: Log all API calls with tenant context (Task 2.5 observability)

**Testing Requirements**:
- **7.4a-UNIT-003**: Tenant ID extraction and X-Tenant-ID header injection test
- **7.4a-INTEGRATION-002**: Verify tenant isolation in full app context

**Residual Risk**: **Low** — Backend 3-layer enforcement prevents data breach even if frontend fails. Frontend validation provides UX benefit (fail-fast) and first defense layer.

**Owner**: Dev (Frontend Lead + Security Architect review)
**Timeline**: Task 2.3 implementation, Task 8.2 testing

### SEC-003: Keycloak OIDC Flow Implementation Errors

**Score: 6 (High)** — Probability: Medium (2) × Impact: High (3)

**Description**: Incorrect implementation of Keycloak OIDC login flow, token refresh, or logout could lead to security vulnerabilities (token leakage, session fixation, insufficient logout) or broken authentication (users unable to login).

**Affected Components**:
- `framework/admin-shell/src/providers/authProvider.ts` (login, checkAuth, getPermissions, logout methods)

**Probability: Medium (2)**:
- OIDC protocol is complex (token endpoints, refresh flow, token validation)
- Story does NOT use battle-tested OIDC client library (custom implementation)
- Subtle errors possible in token storage, expiration checks, refresh timing
- Integration with Keycloak requires precise configuration (realm, clientId, serverUrl)

**Impact: High (3)**:
- **Broken authentication**: Users unable to login (availability)
- **Session fixation**: Attacker hijacks session (CWE-384)
- **Token leakage**: Tokens exposed in logs, network traces, or error messages
- **Insufficient logout**: User believes logged out but session persists (CWE-613)
- **Privilege escalation**: Incorrect role extraction from JWT

**Mitigation Strategies**:

1. **Comprehensive unit testing** (Task 8.3, 7.4a-UNIT-005):
   - Test `login()` success with valid credentials
   - Test `login()` failure (invalid credentials, network error, invalid token response)
   - Test `checkAuth()` with valid/expired/missing tokens
   - Test `getPermissions()` role extraction from JWT `realm_access.roles`
   - Test `logout()` localStorage clearing + Keycloak logout redirect

2. **Integration testing** (7.4a-UNIT-006):
   - Test token expiration detection and automatic logout
   - Test automatic token refresh flow (access_token refresh before expiration)
   - Test refresh token expiration handling (force re-login)

3. **Token refresh implementation** (Story Security Considerations code example):
   - Check expiration in `checkAuth()` method
   - Auto-refresh if <5 minutes remaining
   - Handle refresh token expiration gracefully (redirect to login)

4. **Keycloak configuration validation**:
   - Verify `tenant_id` claim issued by Keycloak (SEC-002 dependency)
   - Verify `realm_access.roles` present for RBAC (Task 3.4)

5. **Security code review**: Expert review of authProvider implementation focusing on OIDC best practices

**Testing Requirements**:
- **7.4a-UNIT-005**: Keycloak login flow test (success, invalid credentials, network error)
- **7.4a-UNIT-006**: Token expiration detection and automatic logout test
- **7.4a-SEC-003**: Token refresh flow test (edge cases: expiration during refresh, network failure)

**Residual Risk**: **Medium** — Without OIDC client library, risk of subtle protocol errors. Comprehensive testing mitigates but doesn't eliminate risk. Consider adding OIDC library (e.g., `oidc-client-ts`) in future iteration.

**Owner**: Dev (Frontend Lead with Keycloak expertise) + Security Architect review
**Timeline**: Task 3 implementation, Task 8.3 testing, pre-release security audit

## Medium-Priority Risks (Score 4)

### BUS-001: Framework Publishability / Coupling

**Score: 4 (Medium)** — Probability: Medium (2) × Impact: Medium (2)

**Description**: Accidental inclusion of product-specific domain logic, types, or UI components in `framework/admin-shell` would violate Story 4.5's framework/product separation principle, preventing npm publication and causing tight coupling.

**Affected Components**: All framework components (`AdminShell.tsx`, providers, theme, shared components)

**Mitigation**:
- Strict code reviews enforcing "NO product imports" rule
- Architectural tests (Konsist-like, as in backend): `assertDoesNotDependOn("products/*")`
- Automated CI/CD check: `npm run build` on framework/admin-shell in isolation (Task 9.4)
- Regular review of `framework/admin-shell/package.json` dependencies

**Testing**: 7.4a-INTEGRATION-001 (build library in isolation, verify exports)

### TECH-001: React-Admin Architectural Coupling

**Score: 4 (Medium)** — Probability: Medium (2) × Impact: Medium (2)

**Description**: Tight coupling to React-Admin 5.4.0 core (`<Admin>`, `<Resource>` components) makes future migration to alternative admin frameworks or major React-Admin version upgrades costly and disruptive.

**Affected Components**: `AdminShell.tsx`, `ResourceConfig` interface

**Mitigation**:
- Accept coupling as pragmatic trade-off for MVP velocity
- Minimize direct usage of complex React-Admin internal APIs
- Current approach (accept React-Admin components as props in `ResourceConfig`) is reasonable abstraction
- Plan for regular React-Admin version updates with dedicated testing
- Consider abstraction layer in v2.0 if migration need emerges

**Residual Risk**: Medium — Migration cost is real but deferred. React-Admin is stable and widely used (acceptable dependency).

### TECH-002: RFC 7807 Error Mapping Incompleteness

**Score: 4 (Medium)** — Probability: Medium (2) × Impact: Medium (2)

**Description**: Incomplete or incorrect parsing of RFC 7807 Problem Details from API error responses could lead to poor UX (unclear error messages) or inadvertently expose sensitive backend error details.

**Affected Components**: `framework/admin-shell/src/providers/dataProvider.ts` (error mapper)

**Mitigation**:
- Extensive unit testing (7.4a-UNIT-004) with various RFC 7807 response structures (required fields, optional fields, malformed responses)
- Follow `security.md` principle: Use generic error messages for tenant isolation violations (CWE-209: Information Exposure Through Error Message)
- Test error mapper with real backend error responses during integration testing
- Implement fallback for malformed errors (log raw error, show generic message to user)

**Testing**: 7.4a-UNIT-004 (RFC 7807 error mapping test)

### SEC-004: XSS Prevention (Input Sanitization)

**Score: 4 (Medium)** — Probability: Medium (2) × Impact: Medium (2)

**Description**: Incomplete input sanitization or bypassed DOMPurify could allow XSS attacks when user-generated content is rendered in shared UI components.

**Affected Components**: `dataProvider.ts`, shared UI components (`EmptyState`, custom components)

**Mitigation**:
- **Mandatory DOMPurify** for all user-generated content (Task 2.5 code example)
- Apply sanitization in `dataProvider.create/update` before sending to API
- Sanitize on render if displaying user content in shared components
- Backend CSP headers as secondary defense
- Regular security audits of shared components for XSS vectors

**Testing**: 7.4a-SEC-001 (XSS attack prevention test)

### SEC-005: CSRF Protection Misimplementation
