# Test Design: Story 4.3

**Story**: 4.3 - Implement Layer 3 (Database Layer): PostgreSQL Row-Level Security
**Date**: 2025-09-26
**Designer**: Quinn (Test Architect)

## Test Strategy Overview

- **Total Test Scenarios**: 32
- **Integration Tests**: 28 (88%) - RLS requires real PostgreSQL validation
- **Unit Tests**: 4 (12%) - Limited to pure logic components
- **E2E Tests**: 0 (0%) - Not applicable at database layer
- **Priority Distribution**: P0: 14, P1: 12, P2: 6

**Rationale**: This story implements security-critical database-level tenant isolation. The vast majority of tests must be integration tests using PostgreSQL Testcontainers because:
1. RLS behavior cannot be tested without real PostgreSQL
2. Connection pooling and session variable behavior requires real database
3. Axon Framework transaction integration requires real event store
4. H2 is explicitly forbidden for tenant-critical flows

Unit tests are limited to pure validation logic (e.g., input validation, configuration parsing) that doesn't require database interaction.

## Test Implementation Guidance

### Test Framework and Tooling

**Use Kotest BehaviorSpec** per project standards:
```kotlin
class TenantSessionInterceptorIntegrationTest : BehaviorSpec({
    given("a database connection with tenant context set") {
        `when`("a transaction begins") {
            then("session variable app.current_tenant is set") {
                // Test implementation
            }
        }
    }
})
```

**Use PostgreSQL Testcontainers** (H2 forbidden):
```kotlin
companion object {
    private val postgresContainer = PostgreSQLContainer<Nothing>("postgres:16.1-alpine").apply {
        start()
    }

    @JvmStatic
    @BeforeAll
    fun setupDatabase() {
        // Run Flyway migrations including RLS policies
    }
}
```

### Test Data Strategy

**Multi-Tenant Test Dataset**:
- Tenant A: 10 events, 5 projections
- Tenant B: 15 events, 8 projections
- Tenant C: 5 events, 3 projections
- Use consistent UUIDs for reproducibility

**Isolation Principle**: Each test should verify:
1. Can access own tenant's data
2. Cannot access other tenant's data
3. Zero rows returned when no tenant context

### Performance Test Baselines

**Establish before RLS enablement**:
- Aggregate loading (100 events): p95 = ?ms
- Event store SELECT: p95 = ?ms
- Projection INSERT: p95 = ?ms

**Acceptance criteria after RLS**:
- Degradation <20% acceptable
- p95 must remain <200ms (SLA)

### Axon Integration Test Pattern

```kotlin
given("an aggregate with existing events") {
    `when`("loading the aggregate with RLS enabled") {
        then("events are loaded correctly") {
            // Verify aggregate rehydrates
        }
        then("only current tenant's events are visible") {
            // Verify tenant isolation
        }
    }
}
```

## Test Maintenance Considerations

### Test Data Management
- Use database snapshots for consistent test data
- Implement test data builders for maintainability
- Version control test data alongside migrations

### Test Performance
- Integration tests with Testcontainers are slower (expect 2-3s per test)
- Optimize with connection pooling and schema caching
- Consider parallel test execution for large suite

### Test Reliability
- Testcontainers can be flaky in CI/CD
- Implement retry logic for container startup
- Add healthchecks before running tests

## Integration with Quality Gates

This test design feeds directly into the quality gate decision:

**PASS Criteria**:
- All P0 tests passing
- ≥90% P1 tests passing
- Performance degradation <20%
- No security test failures

**CONCERNS Criteria**:
- <100% P0 tests passing OR
- <90% P1 tests passing OR
- Performance degradation 20-50%

**FAIL Criteria**:
- <90% P0 tests passing OR
- Any critical security test failure OR
- Performance degradation >50%

**Test Design Quality**: This test design provides comprehensive coverage with appropriate prioritization. The 88% integration test ratio is correct for database-layer infrastructure work. The test suite mitigates all identified risks with full or partial coverage.
