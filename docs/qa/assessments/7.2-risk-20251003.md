## High Risks

### SEC-002: Template Injection via Module Name

**Score**: 6 (High Risk)
**Category**: Security
**Probability**: Medium (2) - User input flows directly into template rendering
**Impact**: High (3) - Could generate malicious code or corrupt file system

**Description**:
Malicious module name parameter could inject harmful content into generated templates, potentially creating:
- Path traversal attacks (`../../etc/passwd`)
- Template injection (`<script>alert('xss')</script>`)
- Command injection (`module; rm -rf /`)
- Null byte attacks (`module\0.txt`)
- Unicode normalization exploits

**Affected Components**:
- `ModuleCommand.kt` - accepts user input
- `ModuleGenerator.kt` - processes input for file operations
- `TemplateEngine.kt` - renders templates with user data
- Generated Mustache templates (4 templates)

**Mitigation Strategy** (3-Layer Defense-in-Depth):

1. **Input Validation** (Preventive):
   - Regex validation: `^[a-z][a-z0-9-]*$` (alphanumeric + hyphen only)
   - Path traversal detection: Reject names containing `..`
   - Injection character check: Reject `<`, `>` characters
   - **Implementation**: Subtask 2.2 in ModuleCommand.kt

2. **Template Escaping** (Detective):
   - Mustache HTML-escaping by default (validated in Story 7.1)
   - No arbitrary code execution in logic-less templates
   - **Implementation**: Already validated in 7.1-UNIT-007

3. **File System Validation** (Corrective):
   - Directory existence checks before generation
   - Atomic file writes with error handling
   - **Implementation**: Task 3, Subtask 3.4

**Testing Requirements**:
- ✅ **7.2-UNIT-006** (P0 Security Test): 5 attack scenarios
  1. Path traversal: `../../etc/passwd`
  2. Template injection: `<script>alert('xss')</script>`
  3. Command injection: `module; rm -rf /`
  4. Null bytes: `module\0.txt`
  5. Unicode normalization: `\u202Emodule`

**Residual Risk**: **Low** - Triple-layer validation significantly reduces attack surface. Some zero-day template rendering vulnerabilities may remain, but Mustache's logic-less design minimizes exposure.

**Owner**: dev
**Timeline**: Security test implementation BEFORE any file generation code (Task 9, Subtask 9.7)

### TECH-006: Package Naming Edge Cases

**Score**: 4 (Medium Risk)
**Category**: Technical
**Probability**: Medium (2) - Package name transformation is complex
**Impact**: Medium (2) - Incorrect package structure breaks imports

**Description**:
Converting kebab-case module names to Java package names could fail for:
- Multiple consecutive hyphens: `my--module`
- Trailing hyphens: `my-module-`
- Reserved keywords: `package`, `class`, `interface`
- Numeric-only segments: `123-module`

**Affected Components**:
- `ModuleCommand.kt` - name sanitization logic (Subtask 2.4)
- Package structure in `ModuleMetadata.kt`
- Directory creation in `ModuleGenerator.kt`

**Mitigation Strategy**:

1. **Robust Sanitization** (Preventive):
   - Input validation regex: `^[a-z][a-z0-9-]*$` (ensures valid start, no special chars)
   - Transformation: Remove hyphens, lowercase (e.g., `my-module` → `mymodule`)
   - Reserved keyword check (future enhancement if needed)
   - **Implementation**: Subtask 2.4

2. **Validation Testing** (Detective):
   - Test invalid names produce errors (7.2-UNIT-002)
   - Test edge cases in name transformation
   - **Implementation**: Task 9

**Testing Requirements**:
- ✅ **7.2-UNIT-002** (P0): Invalid module name produces validation error
- ✅ **7.2-UNIT-004** (P0): Template rendering produces correct syntax

**Residual Risk**: **Low** - Regex validation prevents most edge cases.

**Owner**: dev
**Timeline**: Before Task 2 implementation
