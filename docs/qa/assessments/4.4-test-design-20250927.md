# Test Design: Story 4.4

Date: 2025-09-27
Designer: Quinn (Test Architect)

## Test Strategy Overview

- Total test scenarios: 15
- Unit tests: 5 (33%)
- Integration tests: 8 (54%)
- E2E tests: 2 (13%)
- Priority distribution: P0: 7, P1: 5, P2: 3

## Test Scenarios by Acceptance Criteria

### AC1: Interceptor Reads Tenant ID from Event Metadata

#### Scenarios

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 4.4-UNIT-001 | Unit | P0 | Interceptor extracts tenantId from metadata map | Pure metadata extraction logic without dependencies |
| 4.4-UNIT-002 | Unit | P0 | Interceptor throws SecurityException on missing tenantId | Fail-closed validation logic per CWE-209 |
| 4.4-INT-001 | Integration | P0 | Interceptor processes real EventMessage with valid metadata | Component interaction with Axon framework |
| 4.4-INT-002 | Integration | P0 | Interceptor rejects EventMessage with null metadata | Negative path validation with real Axon context |

**Given-When-Then Documentation**:

**4.4-UNIT-001**:
- Given: EventMessage with metadata `{"tenantId": "tenant-a"}`
- When: Interceptor's `handle()` method called
- Then: `TenantContext.set()` invoked with `"tenant-a"`

**4.4-UNIT-002**:
- Given: EventMessage with empty metadata map
- When: Interceptor's `handle()` method called
- Then: SecurityException thrown with message "Access denied: required context missing"

**4.4-INT-001**:
- Given: Real Axon EventMessage published from command handler
- When: Tracking processor invokes interceptor
- Then: TenantContext populated, event handler executes successfully

**4.4-INT-002**:
- Given: Manually constructed EventMessage without tenantId
- When: Tracking processor attempts to process event
- Then: Processing stops with SecurityException, no database writes

### AC2: TenantContext Populated Before @EventHandler Execution

#### Scenarios

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 4.4-INT-003 | Integration | P0 | TenantContext.getCurrentTenantId() succeeds in projection handler | Critical async boundary verification |
| 4.4-INT-004 | Integration | P0 | TenantContext cleared after handler (thread safety) | Prevents cross-tenant leakage (SEC-001) |
| 4.4-UNIT-003 | Unit | P1 | TenantContext.set() called with correct TenantInfo contract | Logic verification for documented API |

**Given-When-Then Documentation**:

**4.4-INT-003**:
- Given: Command dispatched for tenant-a creates WidgetCreatedEvent
- When: @EventHandler in projection handler executes
- Then: `TenantContext.getCurrentTenantId()` returns "tenant-a"

**4.4-INT-004**:
- Given: Two events (tenant-a, tenant-b) processed on same worker thread
- When: First event completes, second event starts
- Then: TenantContext reflects tenant-b, not tenant-a

**4.4-UNIT-003**:
- Given: TenantInfo contract with `id`, `name`, `status` fields
- When: Interceptor calls `TenantContext.set(tenantInfo)`
- Then: TenantInfo.id matches metadata tenantId value

---

### AC3: Database Interceptor Reads Propagated Context

#### Scenarios

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 4.4-INT-005 | Integration | P0 | PostgreSQL session variable set before projection write | RLS enforcement validation (critical security) |
| 4.4-INT-006 | Integration | P1 | Session variable reset after transaction completes | Cleanup verification prevents pollution |
| 4.4-UNIT-004 | Unit | P1 | Database interceptor queries TenantContext | Logic isolation without database |

**Given-When-Then Documentation**:

**4.4-INT-005**:
- Given: Projection handler writes to `widget_projection` table with RLS enabled
- When: Database interceptor executes before repository save
- Then: `SHOW app.current_tenant` returns correct tenant ID

**4.4-INT-006**:
- Given: Projection transaction completes successfully
- When: Transaction commits and cleanup aspect runs
- Then: `SHOW app.current_tenant` returns empty or reset value

**4.4-UNIT-004**:
- Given: NullableTenantContext with mocked `getCurrentTenantId()`
- When: Database interceptor's `beforeSave()` method called
- Then: Interceptor queries context (no database interaction)

### AC4: End-to-End Integration Tests with RLS

#### Scenarios

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 4.4-INT-007 | Integration | P0 | Command → Event → Projection succeeds with RLS | Critical business flow validation |
| 4.4-INT-008 | Integration | P1 | Multiple tenants processed sequentially without leakage | Thread safety validation (SEC-001 mitigation) |
| 4.4-E2E-001 | E2E | P1 | Full CQRS cycle with tenant isolation verification | User journey validation across all layers |
| 4.4-E2E-002 | E2E | P2 | Concurrent tenant operations isolated | Production scenario with parallelism |

**Given-When-Then Documentation**:

**4.4-INT-007**:
- Given: CreateWidgetCommand dispatched for tenant-a
- When: Command → WidgetCreatedEvent → Projection handler executes
- Then: `widget_projection` table contains new row with tenant_id = "tenant-a"

**4.4-INT-008**:
- Given: Two commands for different tenants dispatched sequentially
- When: Events processed on same Axon worker thread
- Then: Both projections written to correct tenant-scoped rows

**4.4-E2E-001**:
- Given: Full application context with Keycloak, PostgreSQL, Axon
- When: User creates widget via REST API with JWT
- Then: Widget appears in query API, isolated by tenant

**4.4-E2E-002**:
- Given: 10 concurrent commands from 5 different tenants
- When: Commands processed by Axon thread pool
- Then: All projections written to correct tenant rows, no cross-contamination

## Risk Coverage Mapping

### Critical Risks (Score 9)
- **SEC-001: Tenant Context Leakage**
  - Primary: 4.4-INT-004 (cleanup verification)
  - Secondary: 4.4-INT-008 (sequential multi-tenant)

### High Risks (Score 6)
- **SEC-002: Fail-Open on Missing Metadata**
  - Primary: 4.4-UNIT-002 (fail-closed logic)
  - Secondary: 4.4-INT-002 (integration negative path)

### Medium Risks (Score 4)
- **TECH-001: Async Context Propagation Complexity**
  - Primary: 4.4-INT-003 (context availability)
  - Secondary: 4.4-E2E-001 (full async flow)

### Low Risks (Score 2-3)
- **DATA-001: RLS Bypass Through Raw SQL**
  - Primary: 4.4-INT-009 (raw SQL negative test)
- **PERF-001: Interceptor Overhead**
  - Primary: 4.4-UNIT-005 (performance benchmark)

## Recommended Execution Order

### Phase 1: Fast Feedback (5-10 minutes)
1. **4.4-UNIT-001**: Metadata extraction
2. **4.4-UNIT-002**: Fail-closed validation
3. **4.4-UNIT-003**: TenantContext API usage
4. **4.4-UNIT-004**: Database interceptor logic

**Exit Criteria**: All unit tests pass, interceptor logic verified

---

### Phase 2: Critical Integration (15-20 minutes)
5. **4.4-INT-001**: Real EventMessage processing
6. **4.4-INT-002**: Negative path with null metadata
7. **4.4-INT-003**: TenantContext in @EventHandler
8. **4.4-INT-004**: Cleanup verification (SEC-001 mitigation)
9. **4.4-INT-005**: RLS session variable validation
10. **4.4-INT-007**: Full command → projection flow

**Exit Criteria**: Critical flows pass, no tenant leakage detected

---

### Phase 3: Thread Safety & Performance (10-15 minutes)
11. **4.4-INT-006**: Session variable cleanup
12. **4.4-INT-008**: Sequential multi-tenant processing
13. **4.4-UNIT-005**: Performance benchmark

**Exit Criteria**: Thread safety verified, performance acceptable

---

### Phase 4: End-to-End Validation (20-30 minutes)
14. **4.4-E2E-001**: Full CQRS cycle with JWT
15. **4.4-E2E-002**: Concurrent multi-tenant operations

**Exit Criteria**: Production-like scenarios pass

---

### Phase 5: Security Hardening (5 minutes)
16. **4.4-INT-009**: Raw SQL bypass attempt

**Exit Criteria**: Architectural constraints enforced

## Test Data Requirements

### Tenants
- **tenant-a**: Primary test tenant with full permissions
- **tenant-b**: Secondary tenant for isolation verification
- **tenant-c**: Optional third tenant for concurrency tests

### Widgets
- **widget-a1**: Belongs to tenant-a
- **widget-a2**: Second widget for tenant-a
- **widget-b1**: Belongs to tenant-b

### Event Metadata
```kotlin
val validMetadata = mapOf("tenantId" to "tenant-a")
val missingMetadata = emptyMap<String, Any>()
val nullMetadata = mapOf("tenantId" to null)
```

### JWT Tokens (E2E Tests)
- **JWT_TENANT_A**: Valid token with `tenant_id = "tenant-a"` claim
- **JWT_TENANT_B**: Valid token with `tenant_id = "tenant-b"` claim
