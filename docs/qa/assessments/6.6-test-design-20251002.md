# Test Design: Story 6.6

**Story**: 6.6 - Implement "Dockets Pattern" BPMN Template
**Date**: 2025-10-02
**Designer**: Quinn (Test Architect)
**Review Type**: Pre-Implementation

## Test Strategy Overview

- **Total Test Scenarios**: 5 (3 automated, 2 manual)
- **Unit Tests**: 0 (0%) - N/A for declarative XML
- **Integration Tests**: 3 (60%) - Deployment and structure validation
- **E2E Tests**: 0 (0%) - Deferred to Epic 8 (appropriate scope)
- **Manual Reviews**: 2 (40%) - Critical path validation
- **Priority Distribution**: P0: 3, P1: 1, P2: 1

**Test Strategy Justification**:
This story creates a declarative BPMN template for testing/reference purposes. Testing focuses on:
1. Schema compliance (BPMN 2.0 XSD)
2. Deployment validation (Flowable engine)
3. Structural completeness (7-step workflow)

E2E workflow execution is appropriately deferred to Epic 8 when real product use cases exist.

## Detailed Test Specifications

### 6.6-INT-001: BPMN Template Deployment

**Test ID**: 6.6-INT-001
**Priority**: P0 (Critical)
**Level**: Integration
**Type**: Positive Test
**Estimated Execution Time**: 2-5 seconds (Flowable auto-deployment)

**Requirements Coverage**: AC1 (indirect), AC2 (direct)
**Risk Mitigation**: TECH-001 (High), SEC-001 (Low)

**Given-When-Then**:
- **Given**: dockets-pattern-template.bpmn20.xml created in framework/workflow/src/axon-integration-test/resources/processes/
- **And**: Flowable engine initialized in @SpringBootTest context with @ActiveProfiles("test")
- **When**: RepositoryService attempts to deploy the process definition (automatic on context load)
- **Then**: Deployment succeeds without SAXParseException
- **And**: Process definition is registered in Flowable repository
- **And**: Query for process definition key "dockets-pattern-template" returns non-null result

**Test Implementation Reference**: Subtask 5.1-5.3

**Pass Criteria**:
- No SAXParseException thrown
- No BpmnError exceptions
- Process definition exists in repository

**Failure Scenarios**:
- SAXParseException: BPMN element ordering violation (TECH-001)
- BpmnError: Invalid delegate expression (SEC-001)
- Null process definition: Deployment failed silently

**Dependencies**:
- Flowable Spring Boot Starter configured (Story 6.1)
- Test PostgreSQL database available via Testcontainers
- BPMN file in correct classpath location

### 6.6-INT-003: BPMN Model Structure Validation

**Test ID**: 6.6-INT-003
**Priority**: P0 (Critical)
**Level**: Integration
**Type**: Structural Validation
**Estimated Execution Time**: <1 second

**Requirements Coverage**: AC1 (direct)
**Risk Mitigation**: TECH-002 (Low)

**Given-When-Then**:
- **Given**: BPMN template successfully deployed
- **And**: BPMN model accessible via RepositoryService.getBpmnModel()
- **When**: Querying deployed BPMN model for element types and counts
- **Then**: Exactly 1 Start Event exists
- **And**: At least 3 Service Tasks exist (PRESCRIPT, CORECOMMAND, POSTSCRIPT + compensation delegates)
- **And**: Exactly 1 Intermediate Catch Event exists (Wait for Axon Event)
- **And**: At least 1 End Event exists (Success + compensation ends)
- **And**: At least 3 Boundary Events exist (Error boundaries on main service tasks)

**Test Implementation Reference**: Subtask 5.5

**Element Count Validation**:
```kotlin
startEvents.size shouldBe 1
serviceTasks.size shouldBeGreaterThanOrEqual 3
intermediateCatchEvents.size shouldBe 1
endEvents.size shouldBeGreaterThanOrEqual 1
boundaryEvents.size shouldBeGreaterThanOrEqual 3
```

**Pass Criteria**:
- All 7 core workflow elements exist
- Element counts match or exceed expected values
- No critical elements missing

**Failure Scenarios**:
- Missing Start Event: Task 2 (Subtask 2.1) not completed
- Service Task count < 3: Task 2 (Subtasks 2.2-2.5) incomplete
- Missing Intermediate Catch Event: Task 2 (Subtask 2.4) not completed
- No Boundary Events: Task 3 (Subtask 3.1) not completed

**Note**: Test uses ≥ comparisons for service tasks, end events, and boundary events because compensation path adds additional elements beyond the core 7-step workflow.

### 6.6-MANUAL-001: XML Element Ordering Inspection

**Test ID**: 6.6-MANUAL-001
**Priority**: P1 (Important - High Risk Mitigation)
**Level**: Manual Code Review
**Type**: Preventive Validation
**Estimated Execution Time**: 5-10 minutes

**Requirements Coverage**: AC1 (schema compliance), AC2 (deployment success)
**Risk Mitigation**: TECH-001 (High - Primary risk mitigation)

**Given-When-Then**:
- **Given**: BPMN template XML file created
- **When**: Manually reviewing XML structure in IDE or text editor
- **Then**: Every `<serviceTask>` element has `<documentation>` before `<extensionElements>`
- **And**: Process-level `<documentation>` appears before process `<extensionElements>` (if present)
- **And**: No BPMN 2.0 XSD schema ordering violations exist

**Review Checklist**:

```markdown
## Element Ordering Validation Checklist

For EACH service task in the BPMN file:
- [ ] `<serviceTask id="prescriptTask">` - `<documentation>` before `<extensionElements>`?
- [ ] `<serviceTask id="dispatchCommandTask">` - `<documentation>` before `<extensionElements>`?
- [ ] `<serviceTask id="waitForEvent">` - `<documentation>` before `<extensionElements>`? (if present)
- [ ] `<serviceTask id="postscriptTask">` - `<documentation>` before `<extensionElements>`?
- [ ] `<serviceTask id="setCompensationCommand">` - `<documentation>` before `<extensionElements>`?

For process-level elements:
- [ ] Process `<documentation>` before process `<extensionElements>`? (if both exist)

Schema Compliance:
- [ ] No XML parsing warnings in IDE
- [ ] File validates against BPMN 2.0 XSD schema (if validator available)
```

**Correct Pattern Example**:
```xml
<serviceTask id="prescriptTask" flowable:delegateExpression="${runAnsiblePlaybookTask}">
  <!-- CORRECT: documentation FIRST -->
  <documentation>
    Executes Ansible playbook for pre-command automation...
  </documentation>
  <extensionElements>
    <flowable:field name="playbookPath">...</flowable:field>
  </extensionElements>
</serviceTask>
```

**Incorrect Pattern Example**:
```xml
<serviceTask id="prescriptTask" flowable:delegateExpression="${runAnsiblePlaybookTask}">
  <!-- WRONG: extensionElements before documentation causes SAXParseException -->
  <extensionElements>
    <flowable:field name="playbookPath">...</flowable:field>
  </extensionElements>
  <documentation>...</documentation>
</serviceTask>
```

**Pass Criteria**:
- All service tasks follow correct element ordering
- No schema validation warnings or errors
- Pattern matches existing BPMN files (hello-world-ansible.bpmn20.xml)

**Failure Scenarios**:
- Element ordering violation → FAIL gate (blocking issue)
- Inconsistent patterns across tasks → CONCERNS

**Remediation**:
- Reorder elements to match correct pattern
- Re-run 6.6-INT-001 to validate deployment succeeds

### 6.6-MANUAL-002: Process Variable Documentation Review

**Test ID**: 6.6-MANUAL-002
**Priority**: P2 (Should-have - Ensures Quality)
**Level**: Manual Code Review
**Type**: Documentation Validation
**Estimated Execution Time**: 10-15 minutes

**Requirements Coverage**: AC1 (comprehensive documentation), NFR6 (Extensibility)
**Risk Mitigation**: DOC-001 (Low)

**Given-When-Then**:
- **Given**: BPMN template completed with `<documentation>` blocks
- **When**: Reviewing documentation content against Dev Notes § Process Variable Documentation Requirements
- **Then**: All 10+ process variables are documented with types and purposes
- **And**: MANDATORY variables (e.g., tenantId) are clearly marked
- **And**: Documentation is self-contained (migration developers can use without external references)

**Documentation Review Checklist**:

```markdown
## Process Variable Documentation Checklist

### Input Variables (Set at process start)
- [ ] tenantId (String, MANDATORY) - Tenant identifier for isolation
- [ ] prescriptPlaybookPath (String) - Path to Ansible playbook for pre-command automation
- [ ] coreCommandClassName (String) - Fully-qualified Axon command class name
- [ ] coreCommandConstructorParameters (Map<String, Any>) - Constructor arguments for command
- [ ] eventBusinessKey (String) - Correlation key for AxonEventSignalHandler
- [ ] postscriptPlaybookPath (String) - Path to Ansible playbook for post-command automation
- [ ] compensationCommandClassName (String) - FQN of compensating command for error recovery

### Output Variables (Set by service tasks)
- [ ] ansibleExitCode (Int) - Playbook exit code (0 = success)
- [ ] ansibleStdout (String) - Standard output (truncated to 4KB)
- [ ] ansibleStderr (String) - Standard error (truncated to 4KB)

### Internal Variables (Set during error handling)
- [ ] commandClassName (String) - Current command class for DispatchAxonCommandTask

### Documentation Quality
- [ ] Each variable has type specified (String, Int, Map, etc.)
- [ ] Each variable has purpose/usage explained
- [ ] MANDATORY variables are explicitly marked
- [ ] Examples provided where helpful (e.g., FQN format for command classes)
- [ ] Legacy Dockets pattern mapping explained in comments (Subtask 6.4)
```

**Pass Criteria**:
- All 10+ variables documented in BPMN `<documentation>` block
- Types, purposes, and usage patterns clear
- Self-contained documentation (no external reference required)

**Concerns Criteria**:
- 1-2 variables missing → CONCERNS (should add)
- Types not specified → CONCERNS (should clarify)
- Usage patterns unclear → CONCERNS (should improve)

**Failure Scenarios**:
- 3+ variables completely missing → Document gap warrants story revision
- Zero documentation in BPMN → AC1 not met

**Remediation**:
- Add missing variable documentation to BPMN `<documentation>` block
- Clarify variable types and usage patterns
- Reference Dev Notes § Process Variable Documentation Requirements

**Reference Documentation**:
- Story Dev Notes § Process Variable Documentation Requirements
- Subtask 1.4, Subtask 6.3

## Test Execution Order

### Recommended Sequence

**Phase 1: Automated Integration Tests** (2-5 minutes total)
```bash
./gradlew :framework:workflow:axonIntegrationTest
```

1. **6.6-INT-001**: Deploy BPMN template
   - **Stop if fails**: Schema violation (TECH-001) or invalid delegates (SEC-001)
2. **6.6-INT-002**: Verify process definition properties
   - **Stop if fails**: Deployment metadata incorrect
3. **6.6-INT-003**: Validate 7-step workflow structure
   - **Stop if fails**: Missing workflow elements (TECH-002)

**Phase 2: Manual Code Review** (15-25 minutes total)

4. **6.6-MANUAL-001**: XML element ordering inspection (CRITICAL)
   - **Stop if fails**: Element ordering violation → Re-fix → Re-run Phase 1
5. **6.6-MANUAL-002**: Process variable documentation review
   - **Note gaps but don't block**: Documentation improvements can be iterative

**Phase 3: Quality Gate Decision**
- **All tests pass + no element ordering issues** → PASS
- **Element ordering issues found** → FAIL (blocking)
- **Documentation incomplete** → CONCERNS (should fix)

## Recommendations

### Immediate Actions

1. ✅ **Run Integration Tests Frequently**: During BPMN template creation, run deployment test after each major element addition
2. ✅ **Manual Review Before Commit**: Complete 6.6-MANUAL-001 (element ordering) before committing BPMN file
3. ✅ **Reference Existing Patterns**: Use hello-world-ansible.bpmn20.xml as structural template

### During Implementation

1. **Incremental Validation**: Test early and often to catch schema issues quickly
2. **Documentation First**: Write comprehensive `<documentation>` blocks before moving to next element
3. **Element Ordering Vigilance**: Double-check `<documentation>` placement when creating each service task

### Post-Implementation

1. **Epic 8 Feedback Loop**: Gather feedback from Epic 8 developers on template usability
2. **Pattern Documentation**: If helpful, update architecture docs with Dockets Pattern usage guide
3. **Test Refinement**: If Epic 8 reveals template issues, enhance test scenarios accordingly

## Conclusion

Test design for Story 6.6 is **comprehensive and appropriate for scope**. The strategy focuses on:

1. **Deployment Validation** (P0) - Ensures BPMN template is structurally valid
2. **Element Completeness** (P0) - Validates 7-step workflow exists
3. **Schema Compliance** (P1) - Mitigates high-risk element ordering issues
4. **Documentation Quality** (P2) - Ensures migration developer usability

**Test Efficiency**: 3 automated tests + 2 manual reviews provide full AC coverage with minimal overhead.

**Risk Mitigation**: All identified risks have corresponding test coverage, with dual coverage for high-risk areas.

**Deferred Testing**: E2E workflow execution is appropriately deferred to Epic 8 when real product context exists.
