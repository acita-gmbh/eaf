### AC 1: Convention Plugin Dependencies
**Requirement**: "The Spring Boot Actuator, Micrometer, and Prometheus registry dependencies are added and managed via the convention plugins."

#### AC1-Trace-001
```gherkin
Scenario: Observability convention plugin supplies Micrometer dependencies
  Given the ObservabilityConventionPlugin is applied to a module
  When the module resolves Gradle dependencies
  Then micrometer-core and micrometer-registry-prometheus come from gradle/libs.versions.toml
  And no inline version declarations exist in module build files
```

**Validation Checkpoints**
- Inspect dependency graphs for framework and product modules
- Ensure plugin copies configuration resources as required

#### AC1-Trace-002
```gherkin
Scenario: Registry bean auto-configures via convention
  Given a Spring context started in framework/observability
  When the application loads
  Then PrometheusMeterRegistry bean is available
  And CustomMetrics wiring registers counters/timers
```

### AC 2: Secured Prometheus Endpoint
**Requirement**: "The `/actuator/prometheus` endpoint is enabled, secured (requires AuthN from Epic 3), and exposed."

#### AC2-Trace-001
```gherkin
Scenario: Unauthorized scrape attempt is rejected
  Given the licensing-server is running without credentials
  When GET /actuator/prometheus is executed
  Then the response is 401 Unauthorized
  And no metrics payload is returned
```

#### AC2-Trace-002
```gherkin
Scenario: eaf-admin role can scrape metrics
  Given a JWT issued to an operator with eaf-admin realm role
  When GET /actuator/prometheus is executed with Authorization header
  Then the response is 200 OK
  And Prometheus exposition format is returned
```

### AC 3: Core Metrics Exposure

```gherkin

  When metrics are scraped
  Then JVM and http_server_requests_seconds metrics exist
```

#### AC3-Trace-002

### AC 4: Automatic Tagging
