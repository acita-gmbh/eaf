**Designer**: Quinn (Test Architect)
**Assessment Type**: Pre-Implementation Test Strategy

---

## Test Strategy Overview

- **Total Test Scenarios**: 8
- **Integration Tests**: 8 (100%)
- **Unit Tests**: 0 (0%) - Infrastructure component, no pure logic
- **E2E Tests**: 0 (0%) - No UI components
- **Priority Distribution**: P0: 4 (50%), P1: 2 (25%), P2: 2 (25%)

**Test Level Rationale**: All tests are integration level because:
- AxonEventSignalHandler integrates Axon (@EventHandler) with Flowable (RuntimeService)
- Requires real Flowable process engine and Axon event processing
- Business logic is minimal (correlation + signaling) - integration is the core concern
- Follows Constitutional TDD (integration-first with real dependencies)

## Test Scenarios by Acceptance Criteria

### AC1: Event Handler Listens for WidgetCreatedEvent

#### 6.3-INT-001: Event handler receives and processes WidgetCreatedEvent

- **Priority**: **P0** (Core functionality)
- **Level**: Integration
- **Description**: Validates @EventHandler method invoked when WidgetCreatedEvent published
- **Test Approach**:
  - Deploy AxonEventSignalHandler component
  - Publish WidgetCreatedEvent via Axon EventGateway
  - Verify handler's `on(WidgetCreatedEvent)` method executed
- **Success Criteria**: Event handler invoked without errors
- **Mitigates Risk**: N/A (basic wiring validation)
- **Subtask Reference**: 4.1, 4.5

#### 6.3-INT-002: Event handler exists as Spring bean

- **Priority**: P1 (Infrastructure validation)
- **Level**: Integration
- **Description**: Validates AxonEventSignalHandler bean initialized with dependencies
- **Test Approach**:
  - Start Spring Boot application context
  - Query for AxonEventSignalHandler bean
  - Verify RuntimeService and TenantContext injected
- **Success Criteria**: Bean exists with all dependencies
- **Mitigates Risk**: N/A (Spring context validation)
- **Subtask Reference**: 4.1

---

### AC2: Handler Correlates Event and Signals Process

#### 6.3-INT-003: RuntimeService correlation query finds waiting process

- **Priority**: **P0** (Core correlation logic)
- **Level**: Integration
- **Description**: Validates business key correlation finds waiting BPMN execution
- **Test Approach**:
  - Deploy BPMN process with Receive Event Task
  - Start process with business key = widgetId
  - Verify RuntimeService.createExecutionQuery() finds waiting execution
- **Success Criteria**: Execution found with matching business key and message subscription
- **Mitigates Risk**: TECH-001 (event ordering), TECH-003 (multiple matches)
- **Subtask Reference**: 4.3-4.4

#### 6.3-INT-004: RuntimeService delivers message signal to execution

- **Priority**: P1 (Message delivery validation)
- **Level**: Integration
- **Description**: Validates RuntimeService.messageEventReceived() signals process successfully
- **Test Approach**:
  - Find waiting execution via correlation query
  - Call RuntimeService.messageEventReceived()
  - Verify process resumes from Receive Event Task
- **Success Criteria**: Process completes successfully after signal
- **Mitigates Risk**: TECH-001 (async event processing)
- **Subtask Reference**: 4.6

#### 6.3-INT-005: Resilience when no waiting process found

- **Priority**: P2 (Resilience pattern validation)
- **Level**: Integration
- **Description**: Validates handler logs warning when no process waiting (not error)
- **Test Approach**:
  - Publish WidgetCreatedEvent without any waiting BPMN process
  - Verify handler logs WARN message
  - Verify event processing continues (does not throw exception)
- **Success Criteria**: WARN log output matches expected format (v1.1 example)
- **Mitigates Risk**: OPS-001 (no waiting process resilience)
- **Subtask Reference**: 4.8

---

### AC3: Integration Tests Validate Pause and Resume

#### 6.3-INT-006: Complete async workflow end-to-end

- **Priority**: **P0** (End-to-end validation)
- **Level**: Integration
- **Description**: Validates complete BPMN → Axon → BPMN async workflow
- **Test Approach**:
  - Deploy BPMN: Start → DispatchCommand (Story 6.2) → ReceiveEvent → End
  - Start process (pauses at Receive Event Task)
  - Publish WidgetCreatedEvent (triggers handler)
  - Verify process resumes and completes
- **Success Criteria**: historicInstance.endTime != null (process completed)
- **Mitigates Risk**: TECH-001 (async event processing), validates Stories 6.2 + 6.3 integration
- **Subtask Reference**: 4.1-4.6 (complete test example lines 216-266)

#### 6.3-INT-007: Tenant isolation validation (SECURITY CRITICAL)

- **Priority**: **P0** (Security validation)
- **Level**: Integration
- **Description**: Validates cross-tenant signaling is blocked by tenant validation
- **Test Approach**:
  - Set TenantContext to "tenant-a"
  - Start BPMN process for tenant-a (business key = widgetId-a)
  - Publish WidgetCreatedEvent for tenant-b
  - Verify handler throws exception (tenant mismatch)
  - Verify tenant-a process does NOT receive signal
- **Success Criteria**: Tenant validation exception thrown, process not signaled
- **Mitigates Risk**: **SEC-001** (tenant isolation violation) - **PRIMARY RISK**
- **Subtask Reference**: 4.7 (marked SECURITY CRITICAL)

#### 6.3-INT-008: Process wait state verification

- **Priority**: P2 (State verification)
- **Level**: Integration
- **Description**: Validates process correctly enters waiting state before event published
- **Test Approach**:
  - Start BPMN process with Receive Event Task
  - Query for executions with messageEventSubscriptionName
  - Verify execution found with correct subscription
- **Success Criteria**: Execution query returns waiting process
- **Mitigates Risk**: N/A (baseline functionality)
- **Subtask Reference**: 4.4
