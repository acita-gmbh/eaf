# Test Design: Story 8.3

**Date**: 2025-10-05
**Designer**: Quinn (Test Architect)
**Assessment Type**: Pre-Implementation Test Strategy

## Test Strategy Overview

This story requires a multi-layered testing approach to validate the architectural migration from JPA to jOOQ. The strategy prioritizes data integrity, performance validation, and regression prevention.

- **Unit Tests**: Focus on the logic within the new `JooqWidgetProjectionRepository` using the Nullable Object pattern to isolate it from the database.
- **Integration Tests**: Verify the jOOQ repository against a real PostgreSQL database using Testcontainers. This is the most critical layer.
- **Validation Tests**: A dedicated suite to compare the outputs of the legacy JPA implementation and the new jOOQ implementation to ensure data consistency.
- **Performance Tests**: Specific benchmarks to validate the non-functional requirement of a ≥20% performance improvement.

## Test Scenarios

### 1. Unit Tests (JooqWidgetProjectionRepository)

**Framework**: Kotest with a `NullableJooqWidgetProjectionRepository` mock.

| Test ID | Priority | Scenario Description |
|---|---|---|
| 8.3-UNIT-001 | P0 | `findByWidgetIdAndTenantId` returns the correct projection when it exists. |
| 8.3-UNIT-002 | P0 | `findByWidgetIdAndTenantId` returns null when the projection does not exist. |
| 8.3-UNIT-003 | P1 | `findByTenantIdOrderByCreatedAtDesc` returns a correctly sorted list of projections for a tenant. |
| 8.3-UNIT-004 | P1 | `getCategorySummaryByTenantId` correctly groups and counts projections by category. |
| 8.3-UNIT-005 | P2 | All other repository methods (10+) are tested for their specific logic against the in-memory map. |

### 2. Integration Tests (Testcontainers + PostgreSQL)

**Framework**: Kotest with Testcontainers running a PostgreSQL 16.1 instance.

| Test ID | Priority | Scenario Description |
|---|---|---|
| 8.3-INT-001 | P0 | **(Data Integrity)** `findByWidgetIdAndTenantId` retrieves a record from the database matching the exact data written by the event handler. |
| 8.3-INT-002 | P0 | **(Tenant Isolation)** A query for Tenant A (`findByTenantId...`) MUST NOT return any data belonging to Tenant B. |
| 8.3-INT-003 | P1 | The `getCategorySummaryByTenantId` jOOQ query produces the same result as an equivalent raw SQL `GROUP BY` query. |
| 8.3-INT-004 | P1 | The projection event handler correctly **inserts** a new record via jOOQ on `WidgetCreatedEvent`. |
| 8.3-INT-005 | P1 | The projection event handler correctly **updates** an existing record via jOOQ on `WidgetUpdatedEvent`. |
| 8.3-INT-006 | P2 | Pagination logic (`Pageable` parameter) works correctly with jOOQ queries. |

### 3. Validation Tests (JPA vs. jOOQ)

**Framework**: A dedicated test suite that uses the feature flag to switch between implementations.

| Test ID | Priority | Scenario Description |
|---|---|---|
| 8.3-VAL-001 | P0 | **(Equivalence)** For a given dataset, the result of `jpaRepo.findByTenantId(...)` is identical to `jooqRepo.findByTenantId(...)`. |
| 8.3-VAL-002 | P0 | **(Equivalence)** The `getCategorySummaryByTenantId` method returns identical summary data for both implementations. |
| 8.3-VAL-003 | P1 | All 10+ repository methods are tested for equivalence across a wide range of parameters (e.g., different tenants, categories, date ranges). This covers the "100+ scenarios" mentioned in the story. |
| 8.3-VAL-004 | P2 | Edge Case: Equivalence is maintained for projections with null or empty fields. |

### 4. Performance Tests

**Framework**: Dedicated performance test class, potentially using a library like JMH or simple timing assertions for this scope.

| Test ID | Priority | Scenario Description |
|---|---|---|
| 8.3-PERF-001 | P0 | **(Baseline)** Measure and record the p95 query latency for the top 3 queries using the **JPA implementation**. This MUST be the first step. |
| 8.3-PERF-002 | P0 | **(Verification)** After migration, measure the p95 latency for the same 3 queries using the **jOOQ implementation** and assert that the improvement is ≥20%. |
| 8.3-PERF-003 | P1 | Benchmark the performance of pagination queries with large offsets (e.g., page 100 of 10,000 records). |

## Test Data Strategy

- **Seeding**: Use a standard set of 1000 widget projections across 5 tenants with varied data (different categories, values, creation dates).
- **Consistency**: The same seed data must be used for both the JPA and jOOQ validation tests to ensure a fair comparison.
- **Performance Data**: A larger dataset (e.g., 100,000 projections) should be used for the performance benchmark tests to get meaningful latency measurements.

## Gaps Identified in Story Plan

1.  **Missing Baseline Task**: The story plan (AC 15, Task 8) requires validating a ≥20% performance improvement but lacks a task to measure the baseline performance *before* the migration. This is a critical omission. **Recommendation**: Add `Task 1.1: Measure and document baseline JPA performance`.
2.  **Undefined Validation Scenarios**: The story mentions "100+ scenarios" for validation testing but does not define them. **Recommendation**: Explicitly list the key query variations (e.g., by tenant, by category, by date range, with pagination, with sorting) that must be tested for equivalence.
