### AC3: The CLI is configured with a templating engine (Mustache or Velocity)

**Test Coverage:** Unit tests validate template engine integration and security

#### Test Scenarios

| ID | Level | Priority | Test Description | Justification |
|---|---|---|---|---|
| 7.1-UNIT-005 | Unit | P0 | Render test template with simple context | Core functionality - basic template rendering |
| 7.1-UNIT-006 | Unit | P0 | Verify missing template file error handling | Error handling - template not found scenario |
| 7.1-UNIT-007 | Unit | P0 | Verify empty/malicious context handling | **Security** - SEC-001 mitigation (template injection) |

**Coverage Assessment:** FULL (including security risk mitigation)

**Given-When-Then Mappings:**

**7.1-UNIT-005:** Basic Template Rendering
- **Given:** TemplateEngine initialized with DefaultMustacheFactory
- **When:** Rendering `test-template.mustache` with context `{name: "EAF Developer", version: "0.1.0"}`
- **Then:** Output contains "Hello, EAF Developer!" and "Version: 0.1.0" with placeholders replaced

**7.1-UNIT-006:** Missing Template Error Handling
- **Given:** TemplateEngine with valid configuration
- **When:** Attempting to render non-existent template "missing.mustache"
- **Then:** Descriptive error message indicates template file not found (not generic exception)

**7.1-UNIT-007:** Security - Template Injection Protection
- **Given:** TemplateEngine with test template
- **When:** Rendering with context containing:
  - Empty strings
  - Special characters (`<>`, `&`, quotes)
  - Path traversal attempts (`../`, `../../`)
  - Null values
- **Then:** Template renders safely without:
  - Code execution
  - File system access
  - Injection vulnerabilities
  - Crashes or exceptions
- **Validates:** SEC-001 (Template injection risk)

## Test Level Justification

### Why Unit Tests for CLI Commands?

**Decision:** Use Picocli's `CommandLine.execute()` in unit tests (not integration tests)

**Rationale:**
- CLI is a standalone tool with no external dependencies
- Picocli execution is pure JVM logic (no I/O, databases, networks)
- Fast execution enables rapid feedback
- No environmental differences to test (TECH-003 risk is minimal)

**Alternative Considered:** Integration tests with actual shell execution
**Rejected Because:** Adds complexity (shell differences, environment setup) without additional coverage value

### Why Integration Tests for Build?

**Decision:** Validate Gradle build and distribution generation

**Rationale:**
- Verifies complete dependency resolution (TECH-001 mitigation)
- Validates Application Plugin configuration (TECH-002 mitigation)
- Catches classpath, version, and configuration issues
- Cannot be validated in unit tests (requires full Gradle context)

### Why Manual Test for Project Structure?

**Decision:** Manual validation of directory layout (one-time)

**Rationale:**
- Low regression risk (directory structure unlikely to change)
- Visual inspection more efficient than automated file system verification
- Provides value in catching misalignment with Dev Notes
- Automation cost > benefit for one-time validation

## Test Maintainability Considerations

### Test Design Principles Applied

**1. Independent Tests**
- Each test is atomic and self-contained
- No test execution order dependencies
- Clean test data setup in each test

**2. Clear Test Names**
- Test IDs follow convention: `{story}-{LEVEL}-{SEQ}`
- Descriptive test case names (e.g., "should render test template with context")

**3. Fast Execution**
- 70% unit tests average 5ms each
- Total automated suite < 1 minute
- Enables frequent test execution during development

**4. Maintainable Assertions**
- Specific assertions (e.g., `shouldContain "EAF CLI 0.1.0"`)
- No brittle matchers (e.g., no exact string matching of multi-line help text)
- Focus on behavior, not implementation details
