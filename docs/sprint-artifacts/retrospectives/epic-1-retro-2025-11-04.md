## Challenges Encountered

### 1. Test Flakiness (Story 1.10)

**Issue:** `PreCommitHooksConventionPluginFunctionalTest` failed first run with `IllegalStateException`, passed on retry.

**Root Cause:** Timing/initialization issue in test setup (non-lazy property access).

**Resolution:**
- Added lazy initialization for `repositoryRoot` and `buildLogicRoot` in `ProjectFixtures.kt`
- Post-review fix applied same day
- Tests now stable

**Severity:** Medium (affected test reliability, but quickly fixed)

**Lesson:** Watch for initialization timing in Gradle functional tests.

### 2. CI Performance Tuning (Story 1.9)

**Issue:** pitest (mutation testing) included in `./gradlew check` task, causing build time >1 minute.

**Impact:** Violated <15min CI target.

**Resolution:**
- Removed pitest from check task dependency
- Moved to nightly pipeline only
- Build time optimized to 12 seconds

**Severity:** Medium (performance issue, quickly resolved)

**Lesson:** Mutation testing too expensive for fast feedback CI - belongs in nightly.

### 3. Post-Merge CI Failure (Story 1.11)

**Issue:** Enhanced .editorconfig (max line length 120) exposed pre-existing violations in `framework/workflow/build.gradle.kts`.

**Impact:** CI build failed on main branch after PR #11 merge.

**Resolution:**
- Identified 3 lines with comments exceeding 120 characters
- Created PR #12 with fixes (moved inline comments to separate lines)
- Merged within 15 minutes
- Added path filters to security-review.yml to optimize CI

**Severity:** Medium (main branch temporarily red, but rapid fix)

**Lesson:** When tightening quality standards, scan codebase for pre-existing violations before merge.

### 4. AC Deferral Management (Story 1.8)

**Issue:** AC5 (Module Canvas Generation) requires Application class not available until Epic 10.

**Approach:** Deferred with infrastructure prepared.

**Risk Identified:** Deferred ACs can be forgotten if not tracked.

**Mitigation:** Infrastructure in place (ModuleCanvasGeneratorTest), clear documentation in story.

**Severity:** Low (pragmatic decision, well-documented)

**Lesson:** Deferred ACs need explicit tracking and re-verification in target epic.

## Key Insights & Lessons Learned

### 💡 Insight #1: Comprehensive Preparation Drives Velocity

**Observation (Wall-E):** "Die gute Vorbereitung war das A und O"

**Evidence:**
- PRD + Architecture (89 decisions) + Tech-Spec + 112 pre-drafted stories
- Story Context XMLs with docs, code artifacts, interfaces, constraints
- Version Catalog pre-verified (28+ dependencies)
- Zero "which version?" or "which approach?" discussions during implementation

**Impact:**
- Zero requirements changes mid-story
- Fast code reviews (clear AC validation)
- Minimal rework or refactoring
- Team could focus on implementation, not discovery

**Lesson for Epic 2:**
✅ Create Epic 2 Tech-Spec **before** Story 2.1
✅ Generate Story Contexts for **all** 13 stories upfront
✅ Define CQRS/ES test patterns **before** implementation

### 💡 Insight #2: Just-Enough Planning ≠ Over-Planning

**Discussion Point (Elena):** "Ist das nicht over-planning? Agile sagt 'working software over comprehensive documentation'?"

**Team Consensus (Charlie):** "Nein - es ist **just-enough planning**."

**Distinction:**
- Defined **'what'** and **'why'** (architecture, requirements, constraints)
- Left **'how exactly'** to developers (implementation freedom)
- Documentation enabled decisions, didn't prescribe code

**Evidence:** Each story had multiple valid implementation approaches within constraints.

**Lesson:** Planning architecture/requirements ≠ designing code. Former accelerates, latter constrains.

### 💡 Insight #3: Fail-Fast on Quality Issues

**Observation:** All problems resolved same-sprint, zero debt carried forward.

**Examples:**
- Test flakiness → Fixed same day
- Performance issue → Resolved in same story
- Post-merge CI failure → Fixed in 15 minutes

**Cultural Factor:** Team prioritized fixing over deferring.

**Lesson:** "Fix forward" mindset prevents debt accumulation.

### 💡 Insight #4: Systematic Reviews Find Real Issues

**Observation:** Evidence-based reviews (file:line validation) caught genuine problems.

**Findings Identified:**
- Test flakiness (Story 1.10)
- Documentation inconsistencies (Story 1.9)
- Performance anti-patterns (Story 1.9)
- Pre-existing violations (Story 1.11)

**False Positive Rate:** Zero (all flagged issues were real)

**Lesson:** Systematic validation > cursory approval. Invest time in thorough reviews.
