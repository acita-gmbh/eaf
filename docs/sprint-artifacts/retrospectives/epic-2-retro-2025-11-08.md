## Key Insights & Lessons Learned 💡

### Lesson 1: O(n) vs O(1) Complexity Matters at Scale

**Discovery:**
Without aggregate caching, each command loaded ALL events from database:
- Command 1: loads 1 event (~1ms)
- Command 2: loads 2 events (~2ms)
- Command 250: loads 250 events (~250ms)
- **Average overhead: ~125ms per command**

With WeakReferenceCache:
- Command 1: loads 1 event (~1ms)
- Commands 2-250: cached aggregate lookup (~4ms each)
- **70x improvement for hot aggregates**

**Application:** Always consider caching for hot aggregates in production. The WeakReferenceCache is production-safe (automatic GC) and provides massive performance gains.

### Lesson 2: Test Configuration Parity vs Performance Trade-offs

**Discovery:**
- Snapshots disabled in test profile (@Profile("!test")) for 70x performance
- PostgreSQL fsync=off + tmpfs in Testcontainers for speed
- mode=subscribing for synchronous test execution

**Trade-off:**
- Tests run 70x faster BUT don't test snapshot creation
- Tests use non-durable PostgreSQL BUT validate functional correctness
- Tests use synchronous processors BUT production uses async

**Resolution:**
- Document expected behavior: "0 snapshots in test, 2 in production"
- Functional correctness validated (projection updates work)
- Performance optimization is the priority for test suite

**Lesson:** Test configuration can diverge from production for performance, but MUST be documented explicitly.

### Lesson 3: Multi-Vector Optimization Strategy

**Approach:**
1. **Vector 1 (Database I/O):** fsync=off, tmpfs → ~140ms savings
2. **Vector 2 (Axon Framework):** mode=subscribing, snapshots disabled → ~70ms savings
3. **Vector 3 (Spring Boot):** Metrics disabled, jOOQ optimization → ~5-10ms savings
4. **Vector 4 (Aggregate Caching):** WeakReferenceCache → ~125ms savings ⭐ PRIMARY WIN

**Insight:**
- Vectors 1-3 are necessary but not sufficient (total: ~210ms savings)
- Vector 4 (Caching) delivered the breakthrough (~125ms additional savings)
- Combined effect: 280ms → 4ms per command

**Lesson:** Performance optimization requires systematic elimination of ALL bottlenecks, not just the obvious ones. The "Silver Bullet" (caching) only emerged after removing other bottlenecks.

### Lesson 4: Constitutional TDD Value

**Evidence:**
- All 13 stories followed test-first approach
- Real dependencies (Testcontainers) caught integration issues early
- Zero mocks policy prevented behavioral divergence
- @SpringBootTest pattern (field injection + init block) worked reliably

**Specific Win:**
- WidgetProjectionEventHandler integration test (Story 2.7) caught event ordering issue
- PropagatingErrorHandler ensured test failures propagated correctly
- Eventually pattern for async validation prevented flaky tests

**Lesson:** Constitutional TDD with real dependencies is slower initially but prevents rework and ensures production parity.

### Lesson 5: AI Research Effectiveness

**Process:**
1. Local investigation (identified mode=subscribing need)
2. AI research (5 experts consulted)
3. Implementation (all 4 vectors applied)
4. Validation (70x improvement achieved)

**AI Research Accuracy:**
- ✅ Aggregate Caching recommendation: SPOT-ON (70x improvement)
- ✅ Multi-vector strategy: ALL 4 vectors validated
- ✅ Performance expectations: "50-100ms achievable" → achieved 4-6ms
- ⚠️ Diminishing returns warning: Correct (Vector 3 only saved 5-10ms)

**Lesson:** AI research is highly valuable for complex technical investigations, especially when consulting multiple specialized models.

## Significant Discoveries

### Discovery 1: Aggregate Caching is Production-Critical

**What We Thought:**
- Aggregate caching is optional performance optimization
- Tests don't need caching (small event counts)

**What We Discovered:**
- Without caching: O(n) complexity causes quadratic performance degradation
- Command N loading N events creates ~125ms average overhead at 250 commands
- Hot aggregates in production will suffer same issue without caching

**Impact on Epic 3:**
- ✅ No changes needed - caching already implemented in Epic 2
- ✅ WeakReferenceCache is production-safe (GC-friendly)
- ⚠️ Epic 4 must ensure cache keys include tenant context

**Action:** Document aggregate caching as MANDATORY pattern in architecture.md

### Discovery 2: Testcontainers Performance Optimization Patterns

**What We Learned:**
- fsync=off + synchronous_commit=off eliminates 150ms+ per transaction
- tmpfs mount (/var/lib/postgresql/data) provides in-memory performance
- mode=subscribing for tests is 21x faster than mode=tracking

**Pattern Established:**
```kotlin
PostgreSQLContainer(DockerImageName.parse("postgres:16.10-alpine"))
  .withCommand("postgres", "-c", "fsync=off", "-c", "synchronous_commit=off")
  .withTmpFs(mapOf("/var/lib/postgresql/data" to "rw"))
```

**Impact:** All future integration tests should use this pattern. Documented in AxonTestConfiguration.kt.

### Discovery 3: @Profile("!test") Pattern for Performance

**Pattern:**
```kotlin
@Bean
@Profile("!test")
fun snapshotTriggerDefinition(snapshotter: Snapshotter): SnapshotTriggerDefinition
```

**Use Cases:**
- Disable expensive operations in tests (snapshots, metrics, background jobs)
- Maintain production behavior while optimizing test execution
- Document test vs production differences explicitly

**Impact:** Established reusable pattern for Epic 3+ test optimization.
