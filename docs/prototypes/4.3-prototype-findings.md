## Technical Analysis

### PostgreSQL Session Variable Behavior Investigation

**Question**: Why does `current_setting('app.current_tenant', true)` return empty string instead of null?

**PostgreSQL Documentation** (paraphrased):
- `current_setting(setting_name, missing_ok)` returns the current value of a setting
- When `missing_ok` is `true`, returns `NULL` if setting doesn't exist
- When `missing_ok` is `false`, raises error if setting doesn't exist

**Expected Behavior**:
```sql
SET LOCAL app.current_tenant = 'tenant-a';
SELECT current_setting('app.current_tenant', true); -- Returns 'tenant-a'
COMMIT;
SELECT current_setting('app.current_tenant', true); -- Should return NULL
```

**Actual Behavior** (in tests):
```sql
SET LOCAL app.current_tenant = 'tenant-a';
SELECT current_setting('app.current_tenant', true); -- Returns 'tenant-a'
COMMIT;
SELECT current_setting('app.current_tenant', true); -- Returns '' (empty string)
```

**Possible Explanations**:

1. **GUC (Grand Unified Configuration) Default Values**
   - PostgreSQL may have set empty string as default for custom variables
   - Need to verify with: `SHOW app.current_tenant;` vs `current_setting()`

2. **Connection Pooling Interference**
   - HikariCP may be executing connection reset logic
   - Need to test with `autoCommit = false` explicitly set

3. **Testcontainers JDBC URL Parameters**
   - JDBC driver may be modifying connection state
   - Need to verify JDBC URL doesn't have conflicting parameters

4. **PostgreSQL Version Behavior**
   - PostgreSQL 16.1 may have different semantics than earlier versions
   - Need to test with production-equivalent PostgreSQL configuration

### Recommended Investigation Steps

1. **Direct PostgreSQL Console Test**
   ```bash
   docker exec -it <postgres-container> psql -U test -d test
   SET LOCAL app.current_tenant = 'test';
   SELECT current_setting('app.current_tenant', true);
   COMMIT;
   SELECT current_setting('app.current_tenant', true);
   ```

2. **Check for Default Value**
   ```sql
   SHOW app.current_tenant;  -- Should error if not set
   SELECT pg_settings.* FROM pg_settings WHERE name = 'app.current_tenant';
   ```

3. **Test Alternative Cleanup Approach**
   ```kotlin
   // Instead of relying on SET LOCAL auto-clear
   connection.createStatement().execute("RESET app.current_tenant")
   ```

4. **Validate Transaction Boundaries**
   ```kotlin
   // Ensure transaction is actually committed
   connection.autoCommit = false
   // ... set variable, do work ...
   connection.commit()
   assert(!connection.autoCommit) { "AutoCommit should still be false" }
   ```

## Lessons Learned

### What Worked Well ✅

1. **PostgreSQL Testcontainers Integration**
   - Container startup reliable and fast (~30 seconds)
   - RLS policies created successfully
   - Schema setup automated via SQL file

2. **Kotest BehaviorSpec**
   - Given-When-Then structure clear and readable
   - Test organization logical and maintainable

3. **Prototype Approach**
   - Identified critical session variable behavior issue early
   - Saved days/weeks of implementation effort
   - Clear validation criteria prevented false confidence

### What Didn't Work ⚠️

1. **Session Variable Lifecycle Assumptions**
   - Assumed `SET LOCAL` would clear variable to null on commit/rollback
   - Actual behavior returns empty string, which has different semantics
   - Need explicit `RESET` or better verification logic

2. **Test Infrastructure**
   - Spring Boot context not needed for pure database tests
   - `@SpringBootTest` annotation caused unnecessary overhead
   - Direct JDBC connections more appropriate for this prototype

3. **Initial Directory Structure**
   - Tests initially placed in `src/test` instead of `src/integration-test`
   - Required directory restructuring mid-execution
