# SEC-002 Deep Research Brief: PostgreSQL Session Variable Clearing Behavior

**Date**: 2025-09-26
**Priority**: CRITICAL - Blocking Story 4.3 Implementation

**Required Expertise**: PostgreSQL internals, JDBC, connection pooling, multi-tenant security

## Research Objective

Investigate why `current_setting('app.current_tenant', true)` returns an **empty string** instead of **null** after transaction commit/rollback when using `SET LOCAL`, and determine the correct implementation pattern for tenant-scoped session variables in a connection-pooled environment.

## Background Context

### The Security Requirement

We are implementing Row-Level Security (RLS) in PostgreSQL for multi-tenant isolation. The security model requires:

1. **Session variable sets tenant context**: `SET LOCAL app.current_tenant = '<tenant-id>'`
2. **RLS policies enforce isolation**: `USING (tenant_id = current_setting('app.current_tenant')::UUID)`
3. **Variable clears on transaction boundary**: Prevents tenant context leakage across pooled connections
4. **Fail-closed semantics**: Missing session variable blocks all queries (returns 0 rows)

### The Problem

**Expected Behavior**:
```sql
BEGIN;
SET LOCAL app.current_tenant = 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa';
SELECT current_setting('app.current_tenant', true);  -- Returns: 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa'
COMMIT;
SELECT current_setting('app.current_setting', true);  -- Expected: NULL
```

**Actual Behavior** (in Testcontainers + JDBC):
```sql
BEGIN;
SET LOCAL app.current_tenant = 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa';
SELECT current_setting('app.current_tenant', true);  -- Returns: 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa'
COMMIT;
SELECT current_setting('app.current_tenant', true);  -- Returns: '' (empty string)
```

**Security Impact**: Empty string vs. null has different semantics:
- `NULL` → Fails RLS policy check → 0 rows returned (fail-closed) ✅
- `''` (empty string) → May be treated as valid UUID or cause different failure mode → Unpredictable behavior ❌

**Helper Methods**:
```kotlin
private fun setSessionVariableLocal(connection: Connection, tenantId: UUID) {
    val sql = "SET LOCAL app.current_tenant = '$tenantId'"
    connection.createStatement().execute(sql)
}

private fun verifySessionVariable(connection: Connection): String? {
    val sql = "SELECT current_setting('app.current_tenant', true)"
    connection.createStatement().use { statement ->
        val resultSet = statement.executeQuery(sql)
        if (resultSet.next()) {
            return resultSet.getString(1)
        }
        return null
    }
}
```

### Experiment 5: Alternative Cleanup Approaches

**Hypothesis**: Explicit `RESET` is more reliable than `SET LOCAL` auto-clear

**Procedure**:
```kotlin
// Approach 1: Explicit RESET
connection.createStatement().execute("RESET app.current_tenant")

// Approach 2: Transaction callback
connection.commit()
connection.createStatement().execute("RESET app.current_tenant")

// Approach 3: SET to NULL explicitly
connection.createStatement().execute("SET app.current_tenant TO DEFAULT")

// Approach 4: Connection wrapper
class CleanupConnection(private val delegate: Connection) : Connection by delegate {
    override fun commit() {
        delegate.commit()
        delegate.createStatement().execute("RESET app.current_tenant")
    }
    override fun rollback() {
        delegate.rollback()
        delegate.createStatement().execute("RESET app.current_tenant")
    }
}
```
