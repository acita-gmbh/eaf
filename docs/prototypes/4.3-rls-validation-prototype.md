# RLS + Axon + Connection Pooling Validation Prototype

**Purpose**: Validate the three critical unvalidated risks before full Story 4.3 implementation

**Date Created**: 2025-09-26
**Status**: In Progress
**Owner**: Dev Team
**Estimated Effort**: 2-3 days

## Objectives

This prototype validates three critical risks identified in the QA assessment:

1. **SEC-001**: RLS Policy Bypass - Verify interceptor executes before Axon queries
2. **SEC-002**: Session Variable Leakage - Validate connection pooling with `SET LOCAL`
3. **TECH-001**: Axon Compatibility - Test event replay/projection with RLS enabled

## Validation Scenarios

### 1. Interceptor Execution Order (SEC-001)

**Scenario**: Verify database interceptor sets session variable BEFORE Axon queries execute

**Test Cases**:
- ✅ Interceptor runs on transaction start
- ✅ Session variable is set before SELECT queries
- ✅ Session variable is set before INSERT queries
- ✅ Multiple transactions maintain correct session variable

**Expected Results**:
- Interceptor executes in correct Spring/Axon transaction lifecycle
- All queries see correct `app.current_tenant` value
- No queries execute without session variable set

### 2. Connection Pool Session Variable Management (SEC-002)

**Scenario**: Validate `SET LOCAL` prevents session variable leakage across pooled connections

**Test Cases**:
- ✅ `SET LOCAL` scopes variable to transaction only
- ✅ Connection reuse doesn't leak previous tenant context
- ✅ Rapid tenant switching on same connection is safe
- ✅ Connection validation detects incorrect session variable

**Expected Results**:
- `SET LOCAL` automatically clears variable on transaction commit/rollback
- HikariCP connection reuse is safe
- No cross-tenant data contamination possible

### 3. Axon Framework Compatibility (TECH-001)

**Scenario**: Verify Axon event sourcing and projections work with RLS enabled

**Test Cases**:
- ✅ Command handler writes events with RLS
- ✅ Aggregate loading reads events with RLS
- ✅ Projection handler updates read model with RLS
- ❓ Event replay without tenant context (document required pattern)

**Expected Results**:
- Normal Axon operations work with RLS
- Event sourcing integrity maintained
- Projection updates work correctly
- Clear guidance for system operations

## Implementation Plan

### Phase 1: Minimal Interceptor (4 hours)

1. Create `TenantSessionInterceptor.kt`:
   - Implement `ConnectionCallback` or use Hibernate interceptor
   - Execute `SET LOCAL app.current_tenant = ?` on transaction start
   - Use `TenantContext.getCurrentTenantId()` (fail-closed)
   - Log all session variable operations

2. Create test RLS policy SQL:
   - Minimal test table with `tenant_id` column
   - RLS policy checking `current_setting('app.current_tenant')::UUID`
   - Enable RLS on test table

3. Write basic interceptor test:
   - Verify session variable is set
   - Verify RLS filters queries correctly

### Phase 2: Connection Pool Validation (4 hours)

4. Configure minimal HikariCP pool (2 connections)

5. Test rapid tenant switching:
   - Transaction for Tenant A
   - Transaction for Tenant B (on same connection)
   - Verify no leakage

6. Validate `SET LOCAL` behavior:
   - Verify variable clears after commit
   - Verify variable clears after rollback

### Phase 3: Axon Integration (8 hours)

7. Create minimal test aggregate:
   - Single command handler
   - Single event handler
   - Uses tenant-aware event store table

8. Test Axon command processing with RLS:
   - Write event
   - Load aggregate
   - Verify RLS doesn't block operations

9. Test projection with RLS:
   - Event handler updates projection table
   - Query projection with RLS
   - Verify correct tenant isolation

10. Document event replay pattern:
    - Identify how to handle system operations
    - Document required admin user configuration

### Phase 4: Documentation (2 hours)

11. Document findings:
    - What works out of the box
    - What requires configuration
    - Identified pitfalls/gotchas

12. Update QA gate:
    - Mark critical risks as validated/mitigated
    - Upgrade gate status to CONCERNS
